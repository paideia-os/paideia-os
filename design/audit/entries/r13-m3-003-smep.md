# Audit: r13-m3-003 SMEP enable (#447)

## Justification

**Meltdown/kernel-exec prevention**: Supervisor Mode Execution Prevention (SMEP) is a hardware security feature that prevents the kernel from accidentally (or maliciously, via speculative execution) fetching and executing instructions from user-space pages. When SMEP is enabled (CR4 bit 20 set), a ring-0 fetch from a page marked User/Supervisor=1 (U/S=1) causes a page fault (#PF with error code indicating SMEP violation).

At the point of enablement (R13-m3-003), **no user-space pages are yet mapped**, so setting SMEP is safe: the kernel's own .text and .data segments are all mapped as Supervisor (U/S=0). The instruction sequence is atomic with respect to kernel execution flow: read CR4, OR bit 20, write CR4.

## Intended Sequence

1. **Read CR4 into RAX** (`mov rax, cr4`)
2. **Set SMEP bit (bit 20)** (`or rax, 0x100000` where 0x100000 = 1048576 decimal)
3. **Write RAX back to CR4** (`mov cr4, rax`)
4. **Return** (`ret`)

Encoding verification (paideia-as CR-move):
- `mov rax, cr4`: `0F 20 E0` (MODRM R=0, opcode 20, register 0)
- `or rax, imm32`: `48 81 C8 00 00 10 00` (REX.W, opcode 81, /1, imm32 = 0x00100000)
- `mov cr4, rax`: `0F 22 E0` (MODRM R=4, opcode 22, register 0)

## Invariants

**I1**: After `smep_enable()` returns, CR4.SMEP (bit 20) is set.

**I2**: Kernel .text and .data remain U/S=0 (Supervisor, unaffected by SMEP enable).

**I3**: No page fault is generated at the time of SMEP enable, since no user pages are mapped yet.

## Non-Invariants

**NI1**: No user-space pages exist yet at the point of enablement (N/A after Task A / Task B are running).

**NI2**: SMAP (Supervisor Memory Access Prevention, CR4 bit 21) is a separate mechanism; this entry covers SMEP only.

**NI3**: CPUID must advertise SMEP support (MSR 0x3A6 bit 7, or CPUID.07H:EBX[7]). If the CPU does not support SMEP, the `mov cr4, rax` instruction may cause #UD (Undefined Opcode) or silently ignore the bit. QEMU with `-cpu max` and `-cpu host` expose SMEP. Single-threaded testing (boot_r8_only) will not exercise SMEP violations; future tests (when user pages are mapped) will.

## Verification

Build: `./tools/build.sh` must complete without errors.

Symbol table: `nm build/kernel.elf | grep smep_enable` confirms symbol export from Smep module.

Disassembly: `objdump -d build/kernel.elf --disassemble=smep_enable` must show:
- `0f 20 e0` (mov rax, cr4)
- `48 81 c8 00 00 10 00` (or rax, 0x100000)
- `0f 22 e0` (mov cr4, rax)

## Implementation Notes

- SMEP is enabled after IDT install (R9-m1-002) but before APIC initialization (R9-m2-001), ensuring the security boundary is established before any device I/O or interrupt handling.
- The call is unconditional: no CPUID check is performed at boot time. If the CPU does not support SMEP, the behavior is undefined (fault or silent ignore). Future work (R13-m3-004+) will add CPUID validation and graceful fallback.
- The encoding relies on paideia-as CR-move support (0F 20/22 opcodes for mov cr4). Hex literals (0x100000) are preferred; decimal 1048576 is used as fallback if hex encoding fails.
