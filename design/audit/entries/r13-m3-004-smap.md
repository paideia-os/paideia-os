# Audit: r13-m3-004 SMAP enable (#448)

## Justification

**Meltdown/kernel-data prevention**: Supervisor Mode Access Prevention (SMAP) is a hardware security feature that prevents the kernel from accidentally (or maliciously, via speculative execution) reading or writing user-space pages. When SMAP is enabled (CR4 bit 21 set), a ring-0 memory access (read or write) to a page marked User/Supervisor=1 (U/S=1) causes a page fault (#PF with error code indicating SMAP violation), unless the access is explicitly permitted via the stac instruction (Set AC flag, allowing temporary user-memory access within stac/clac windows).

At the point of enablement (R13-m3-004), **no user-space pages are yet mapped**, so setting SMAP is safe: the kernel's own .data and .bss segments are all mapped as Supervisor (U/S=0). The instruction sequence is atomic with respect to kernel execution flow: read CR4, OR bit 21, write CR4.

## Intended Sequence

1. **Read CR4 into RAX** (`mov rax, cr4`)
2. **Set SMAP bit (bit 21)** (`or rax, 0x200000` where 0x200000 = 2097152 decimal)
3. **Write RAX back to CR4** (`mov cr4, rax`)
4. **Return** (`ret`)

Encoding verification (paideia-as CR-move):
- `mov rax, cr4`: `0F 20 E0` (MODRM R=0, opcode 20, register 0)
- `or rax, imm32`: `48 81 C8 00 00 20 00` (REX.W, opcode 81, /1, imm32 = 0x00200000)
- `mov cr4, rax`: `0F 22 E0` (MODRM R=4, opcode 22, register 0)

## Invariants

**I1**: After `smap_enable()` returns, CR4.SMAP (bit 21) is set.

**I2**: Kernel .data and .bss remain U/S=0 (Supervisor, unaffected by SMAP enable).

**I3**: No page fault is generated at the time of SMAP enable, since no user pages are mapped yet.

## Non-Invariants

**NI1**: No user-space pages exist yet at the point of enablement (N/A after Task A / Task B are running).

**NI2**: SMEP (Supervisor Mode Execution Prevention, CR4 bit 20) is a separate mechanism; this entry covers SMAP only.

**NI3**: CPUID must advertise SMAP support (CPUID.07H:EBX[20]). If the CPU does not support SMAP, the `mov cr4, rax` instruction may cause #UD (Undefined Opcode) or silently ignore the bit. QEMU with `-cpu max` and `-cpu host` expose SMAP. Single-threaded testing (boot_r8_only) will not exercise SMAP violations; future tests (when user pages are mapped) will.

**NI4**: Kernel accessors of user memory must employ stac/clac windows (Supervisor Access Control, future work PA-R13-006 m5-002) to temporarily permit user-page access within a critical section. Without such windows, any kernel attempt to dereference a user-space pointer after SMAP is enabled will fault.

## Implementation Notes

- SMAP is enabled after SMEP (R13-m3-003) but before APIC initialization (R9-m2-001), establishing both execution and memory access security boundaries before any device I/O or interrupt handling.
- The call is unconditional: no CPUID check is performed at boot time. If the CPU does not support SMAP, the behavior is undefined (fault or silent ignore). This implementation mirrors SMEP's conservative guard: CPUID validation happens inside `smap_enable()` to avoid #UD faults on systems without SMAP support (e.g., QEMU TCG).
- The encoding relies on paideia-as CR-move support (0F 20/22 opcodes for mov cr4). Hex literals (0x200000) are preferred; decimal 2097152 is used as fallback if hex encoding fails.
- Future kernel accessors of user memory will employ stac/clac windows (PA-R13-006 m5-002) to temporarily permit user-page access within a critical section. Without such windows, any kernel attempt to dereference a user-space pointer after SMAP is enabled will cause a #PF (page fault) with SMAP violation error code.
