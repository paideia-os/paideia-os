# Audit: r13-m3-005 NX enable (#449)

## Justification

**Data-execution prevention (W^X policy)**: No-Execute (NX) is a hardware security feature that prevents data pages from being executed as code. When NX is enabled via IA32_EFER.NXE (bit 11 set), a ring-0 fetch from a page marked with the XD bit (bit 63 in the page-table entry) causes a page fault (#PF with error code indicating NX violation). This enforces the Write-XOR-Execute (W^X) principle: pages may be written or executed, but not both.

At the point of enablement (R13-m3-005), **no user-space pages are yet mapped**, so setting NX is safe: the kernel's own .text and .rodata are mapped as executable (U/S=0, XD=0), while .data and .bss will be marked non-executable (XD=1). The instruction sequence is atomic with respect to kernel execution flow: read IA32_EFER (MSR), OR bit 11, write back.

## Intended Sequence

1. **Check CPUID.80000001H:EDX[20]** (mov rax, 0x80000001; cpuid; test EDX[20])
2. **If NX unsupported, skip to end** (cmp/je)
3. **Read IA32_EFER into RAX** (mov rcx, 0xC0000080; rdmsr — reads EDX:EAX)
4. **Set NXE bit (bit 11)** (or rax, 0x800 where 0x800 = 2048 decimal)
5. **Write RAX back to IA32_EFER** (wrmsr)
6. **Return** (ret)

Encoding verification (paideia-as MSR-move):
- `mov rax, 0x80000001`: `B8 01 00 00 80` (REX not needed for imm32, but MOV reg, imm32)
- `cpuid`: `0F A2`
- `and rdx, 0x100000`: `48 81 E2 00 00 10 00` (REX.W, opcode 81, /4, imm32 = 0x00100000)
- `cmp rdx, 0`: `48 83 FA 00` (REX.W, cmp reg64, imm8)
- `je skip_nx`: `74 xx` (short jz)
- `mov rcx, 0xC0000080`: `48 B9 80 00 00 C0 00 00 00 00` (REX.W, MOV reg64, imm64)
- `rdmsr`: `0F 32` (reads EDX:EAX from MSR in RCX)
- `or rax, 0x800`: `48 81 C8 00 08 00 00` (REX.W, opcode 81, /1, imm32 = 0x00000800)
- `wrmsr`: `0F 30` (writes EDX:EAX to MSR in RCX)

## Invariants

**I1**: After `nx_enable()` returns, IA32_EFER.NXE (bit 11) is set (if CPU supports NX).

**I2**: Kernel .text and .rodata remain executable (XD=0, U/S=0).

**I3**: Kernel .data and .bss can be marked non-executable (XD=1) by aspace_map callers (future PTE setup).

**I4**: No page fault is generated at the time of NX enable, since no data pages yet have XD set.

## Non-Invariants

**NI1**: No user-space pages exist yet at the point of enablement (N/A after Task A / Task B are running).

**NI2**: SMEP (Supervisor Mode Execution Prevention, CR4 bit 20) and SMAP (Supervisor Mode Access Prevention, CR4 bit 21) are separate mechanisms; this entry covers NX only.

**NI3**: CPUID must advertise NX support (CPUID.80000001H:EDX[20]). If the CPU does not support NX, the `wrmsr` instruction may cause #GP (General Protection Fault) or silently ignore the bit. QEMU with `-cpu max` and `-cpu host` expose NX. NX is nearly universal on modern x86-64 CPUs (1TB+ memory, Intel since Pentium 4, AMD since Athlon64), but defensive CPUID guard is good practice.

**NI4**: PTE.XD flag (bit 63) propagation is the responsibility of aspace_map callers; this module only enables the hardware feature at the IA32_EFER level.

**NI5**: Future kernel accessors of executable data (e.g., code patches, JIT) must temporarily relax XD bits via PTE remapping or use self-modifying code sections; this is out of scope for R13-m3-005.

## Implementation Notes

- NX is enabled after SMAP (R13-m3-004) but before APIC initialization (R9-m2-001), establishing the data-execution prevention boundary before any device I/O or interrupt handling.
- The call includes a CPUID guard: if the CPU does not support NX (CPUID.80000001H:EDX[20] = 0), the function skips the MSR write to avoid #GP fault on older systems (defensive; NX is nearly universal on x86-64).
- The encoding relies on paideia-as MSR-move support (rdmsr/wrmsr opcodes 0F 32/30). Hex literals (0xC0000080, 0x800) are preferred; decimal 2048 is provided as fallback for IA32_EFER_NXE constant.
- Future PTE setup in aspace_map will set bit 63 (XD) for data pages, preventing execution. This module only enables the hardware feature; PTE management is caller's responsibility.
