# R16-M1-003 — vops function-pointer table + dispatch helpers (#572)

## 1. Scope

Give every backend type (tmpfs, devfs, procfs, tty, ...) a **shared per-type
function-pointer table** — a virtual method table — and one **thin
dispatch helper per op** that:

- loads `ops_ptr` from the vnode (fixed offset +24, frozen by #570 §3),
- null-checks it,
- loads the op slot at a fixed offset within the table,
- null-checks it,
- forwards the call with the callee's argument registers untouched.

This is the dispatch substrate the whole VFS surface rides on. Every
subsequent VFS syscall (`sys_read`, `sys_write`, `sys_open`, `sys_close`,
`sys_unlink`, path resolution's `lookup` step, directory `create`)
becomes a call to one of these dispatchers. The dispatcher is the
single point where "which backend is this vnode?" turns into "run this
backend's op." No caller ever branches on vnode type; the branch is
folded into the fixed offset in the vops table.

Out of scope (deliberately deferred):

- **Any real backend table.** `tmpfs_vops`, `devfs_vops`, `procfs_vops`
  land in subsequent issues (R16.M2, R17). This issue only ships the
  dispatch layer + a witness-only stub table.
- **`mmap`, `ioctl`, `fsync`, `truncate`, `seek`.** The 7 ops in this
  freeze are the R16.M4 minimum for a working shell + textual I/O.
  Growing the table is a **layout re-freeze issue** (§7).
- **Reference-count discipline on vnode lookup/create returns.** The
  returned vnode idx is a bare u16 at this layer; refcount hold/put
  is the caller's job (R16.M1-006 `vfs_open` / #575).

## 2. vops table layout — frozen (7 slots × 8 B = 56 B)

The table is a plain `[u64; 7]` in `.rodata` (or `.data` if the backend
initializes at boot). Every slot is a function pointer — either a
valid code address, or `0` meaning "this backend does not support
this op."

| Offset | Slot | Op       | Signature (in registers)                                                        | Return semantics                                                                 |
|--------|------|----------|---------------------------------------------------------------------------------|----------------------------------------------------------------------------------|
| +0     | 0    | `read`   | `rdi=vn`, `rsi=buf`, `rdx=len`, `rcx=off`                                       | `rax` = bytes read on success (may be < len at EOF); `-1` on error.              |
| +8     | 1    | `write`  | `rdi=vn`, `rsi=buf`, `rdx=len`, `rcx=off`                                       | `rax` = bytes written on success; `-1` on error.                                 |
| +16    | 2    | `open`   | `rdi=vn`, `rsi=flags`                                                           | `rax` = 0 on success; `-1` on error.                                             |
| +24    | 3    | `close`  | `rdi=vn`                                                                        | `rax` = 0 on success; `-1` on error.                                             |
| +32    | 4    | `lookup` | `rdi=dir_vn`, `rsi=name_ptr`                                                    | `rax` = child vnode idx (u16 in low 16 bits) on success; 0 = ENOENT.             |
| +40    | 5    | `create` | `rdi=dir_vn`, `rsi=name_ptr`, `rdx=mode`                                        | `rax` = new vnode idx (u16 in low 16 bits) on success; 0 = failure.              |
| +48    | 6    | `unlink` | `rdi=dir_vn`, `rsi=name_ptr`                                                    | `rax` = 0 on success; `-1` on error.                                             |

**Total table extent: [+0, +56) — 7 slots, 56 bytes.**

**Alignment**: `@align(8)`. The table is not required to be
cache-line-aligned because backend dispatch is a single indirect load
that touches at most 8 bytes; the whole table fits in a single L1D
sub-line spanning at most one cache-line boundary.

### 2.1 Null-slot semantics — E_NOT_SUPPORTED

A slot value of `0` in the table means **"this backend does not
implement this op."** Every dispatch helper checks the slot before
calling:

```
mov rax, [rax + N]   ; op_ptr from table
cmp rax, 0
je  vops_XXX_nosup   ; slot is null → return E_NOT_SUPPORTED
```

The E_NOT_SUPPORTED sentinel is:

```
VOPS_ERR_NOT_SUPPORTED : u64 = 0xFFFFFFFFFFFFFFFF   ; -1 as u64
```

Matches the pre-existing FD_TABLE_ERR_EMFILE / ELF_LOAD_FAIL
convention (u64 -1 = generic failure). Callers distinguish "op not
supported" from other errors by convention downstream (R16.M4 grows
a shared error-code set once user-facing errno is threaded through).

### 2.2 Null-ops_ptr semantics — same error

`ops_ptr == 0` on the vnode itself (fresh alloc that hasn't been
assigned a backend, or explicitly detached vnode) also returns
`VOPS_ERR_NOT_SUPPORTED`. The dispatcher gate is:

```
mov rax, [rdi + 24]   ; VNODE_OPS_PTR_OFFSET (frozen by #570)
cmp rax, 0
je  vops_XXX_nosup
```

Rationale: a live vnode with a real backend has a non-null ops_ptr
(vnode-layout.md §7.5 invariant). A caller that reaches
`vops_read` on a vnode with null ops_ptr has hit a semantic bug —
returning E_NOT_SUPPORTED surfaces it instead of segfaulting on
`mov rax, [null + 0]`.

### 2.3 Why the argument registers pass through untouched

SysV AMD64 passes the first six args in
`rdi, rsi, rdx, rcx, r8, r9` — all of which are caller-saved. The
dispatch helper receives `rdi/rsi/rdx/rcx` from its caller and never
touches them before invoking the op. It only uses `rax` as scratch
(load ops_ptr; load op_ptr; call op_ptr). Consequence: the op sees
`rdi/rsi/rdx/rcx` exactly as the caller of the dispatcher passed
them. No spill, no reshuffle — the dispatcher is effectively a
compile-time-fixed indirection with one add (fixed offset into the
table) added to the cost of a direct call.

### 2.4 Why `call rax` and not `jmp rax` (tail-call form rejected)

A tail-call `jmp rax` would save one stack frame per dispatch. The
paideia-as encoder supports `call reg64` (PA-R13-003) but does not
yet support `jmp reg64` (crates/paideia-as-encoder/src/encode_instruction.rs
`encode_jmp` — no `Operand::Reg` arm). We use `call rax; ret` and
absorb the one extra stack frame. When the encoder grows `jmp reg64`
support, the dispatchers can be re-encoded to tail-call without any
caller-visible change (same signature, same return value).

## 3. Dispatch helper set — 7 helpers, uniform 9-instruction body

Every helper is 9 instructions with an identical shape modulo the
table offset:

```asm
; vops_<op>(rdi=vn, rsi=arg1, rdx=arg2, rcx=arg3) -> rax
mov rax, [rdi + 24]            ; load ops_ptr
cmp rax, 0
je  vops_<op>_nosup
mov rax, [rax + <OP_OFFSET>]   ; load op_ptr
cmp rax, 0
je  vops_<op>_nosup
call rax                        ; indirect call — args pass through
ret                             ; return op's rax
vops_<op>_nosup:
mov rax, 0xFFFFFFFFFFFFFFFF     ; E_NOT_SUPPORTED
ret
```

The 7 helpers share this exact shape; the only per-helper delta is
the byte constant in `mov rax, [rax + <OP_OFFSET>]` (0/8/16/24/32/40/48).
Total code size: `7 × 9 instructions ≈ 130-150 bytes` — sub-line
per helper, single cache-line total.

### 3.1 Register discipline

- **Scratch**: `rax` only.
- **Callee-saves touched**: none. Every dispatch helper is a
  leaf-shaped function under the classical definition (no prologue,
  no callee-save spill), even though it makes an indirect call.
  The `call rax` step relies on the op to preserve its own
  callee-saves; the dispatcher doesn't hold state across the call.
- **Flags**: clobbered by `cmp`. No caller relies on flags across
  the dispatch call (all callers land here from a Pdx-lambda call
  boundary).

### 3.2 Not a leaf — stack alignment

Every dispatch helper makes a nested `call`, so it is **not** a leaf
per SysV. However it has no prologue that pushes anything, so the
stack alignment at the point of `call rax` is:

```
caller's rsp at entry to vops_XXX  : 16-byte aligned (SysV guarantee)
rsp after push return-addr         : rsp mod 16 == 8
rsp at `call rax`                  : rsp mod 16 == 8 ← WRONG for SysV
```

For SysV, the call target expects `rsp mod 16 == 0` at the moment
`call` executes (so that the target's own frame is aligned). Our
dispatch helper enters with `rsp mod 16 == 8` (post-return-addr
push), does not adjust rsp, and calls with `rsp mod 16 == 8`. This
looks wrong on paper but is correct: the op is entered with
`rsp mod 16 == 0` (the `call rax` push produces alignment 0 at op
entry, which is the SysV requirement at function entry).

Verification: match the vnode_pool helper's stack story —
`vnode_alloc` also has no prologue and makes no nested call, so
alignment discipline is inherited from its (SysV-compliant)
callers. Our dispatch helper matches that discipline modulo the
nested indirect call.

## 4. Constants exported by `src/kernel/core/fs/vops.pdx`

Single source of truth for offset arithmetic. No `.pdx` file outside
this module may embed a numeric vops offset.

```
// Size + slot count
VOPS_SIZE                 : u64 = 56       // MUST equal 7 × 8
VOPS_NUM_OPS              : u64 = 7

// Slot offsets — fixed for the R16 series
VOPS_READ_OFFSET          : u64 = 0
VOPS_WRITE_OFFSET         : u64 = 8
VOPS_OPEN_OFFSET          : u64 = 16
VOPS_CLOSE_OFFSET         : u64 = 24
VOPS_LOOKUP_OFFSET        : u64 = 32
VOPS_CREATE_OFFSET        : u64 = 40
VOPS_UNLINK_OFFSET        : u64 = 48

// Vnode field mirror — frozen by design/kernel/vfs-layout.md §3
VNODE_OPS_PTR_OFFSET      : u64 = 24

// Error sentinel
VOPS_ERR_NOT_SUPPORTED    : u64 = 0xFFFFFFFFFFFFFFFF   // -1 as u64
```

## 5. Test canary — R16 VOPS OK

The witness runs in `kernel_main` immediately after the vnode_pool
witness (§5 in vfs-layout.md) so it inherits an initialized VFS
lattice. Three composed sub-tests, one marker:

### 5.1 Sub-test A — dispatch through non-null op returns stub value

Storage: a witness vnode (`_vops_witness_vnode : [u64; 8]`, one L1
line) and a witness vops table (`_vops_witness_table : [u64; 7]`),
both `.bss`. A stub function `_vops_witness_read_stub` returns a
magic constant `0xDEADBEEFCAFEBABE`.

The witness:
1. Zeros the vnode's cold-half and hot-half fields.
2. Zeros the vops table (all 7 slots = null).
3. Writes `&_vops_witness_read_stub` into the `read` slot (+0)
   of the table.
4. Writes `&_vops_witness_table` into the vnode's ops_ptr slot (+24).
5. Calls `vops_read(vn, 0, 0, 0)` and asserts
   `rax == 0xDEADBEEFCAFEBABE`.

Passing this sub-test proves:
- The dispatcher loads ops_ptr from the frozen vnode +24 offset.
- The dispatcher loads read_ptr from the frozen table +0 offset.
- The dispatcher's indirect `call rax` reaches the stub and returns
  the stub's rax to the witness.

### 5.2 Sub-test B — dispatch through null slot returns E_NOT_SUPPORTED

Same witness vnode with the same ops_ptr installed. The `write`
slot at +8 is still null (from step 2). Call
`vops_write(vn, 0, 0, 0)` and assert
`rax == 0xFFFFFFFFFFFFFFFF`.

Passing this sub-test proves the inner null-guard (op slot) works.

### 5.3 Sub-test C — dispatch on vnode with null ops_ptr returns E_NOT_SUPPORTED

Zero the vnode's ops_ptr again (clear +24). Call
`vops_read(vn, 0, 0, 0)` and assert
`rax == 0xFFFFFFFFFFFFFFFF`.

Passing this sub-test proves the outer null-guard (ops_ptr on the
vnode) works — a fresh vnode never crashes even if a caller
prematurely dispatches on it.

## 6. Alternatives considered (rejected)

### 6.1 Type tag + cmp/je chain (no vops table)

**Proposal.** Skip the vtable — the vnode has `type` (VNODE_TYPE_REG etc.)
at +0, and every VFS op does `switch (type) { case REG: tmpfs_read(...); ... }`.

**Rejected.** Every op-of-every-backend has to know every type
discriminant — O(backend_count × op_count) source-level coupling.
Adding a new backend touches every VFS op's dispatch. The vtable
folds this to O(backend_count) — one table per backend, populated
once at boot.

The cmp/je chain also has a per-call latency roughly proportional to
the number of backends; the vtable is a single indirect load
independent of backend count.

### 6.2 8-slot table (round to power of 2)

**Proposal.** Round to 8 slots × 8 B = 64 B for cache-line alignment.

**Rejected.** The 8th slot is dead weight until we add another op.
The 7-op layout ships now; the 8th (`mmap`, `ioctl`, `fsync`,
`truncate`, or `seek` — first-come) becomes a layout re-freeze
event. No hot path traverses the whole table; single-line
alignment (via `@align(8)`) is unnecessary. Rounding costs 8 B of
padding per backend type; at ~4 backends by R17, that's 32 B saved
by refusing to round.

### 6.3 Inline vops in vnode (no ops_ptr indirection)

**Proposal.** Every vnode carries its own 56-byte vops table inline.

**Rejected.** VNODE_SIZE is frozen at 64 B (#570); we cannot afford
56 B for a per-vnode table when the vnode is 4 vnodes × 16 KiB pool
memory-worth of overhead per backend type. The shared-table pattern
is exactly why the vtable exists: 256 vnodes × 4 backend types =
still just 4 × 56 B = 224 B of ops-table memory, versus 256 × 56 B
= 14 KiB inline per vnode.

### 6.4 `mmap`, `ioctl`, `fsync` in the initial freeze

**Proposal.** Include all 10-12 POSIX-style ops in the initial vtable.

**Rejected.** The R16.M4 minimum for a working shell + textual I/O
is `read`, `write`, `open`, `close`, `lookup`, `create`, `unlink`.
`mmap` is a syscall-family that lands in R17 alongside real
memory-mapping semantics. `ioctl` waits for the tty/devfs
substrate. `fsync` waits for a durable backend. Growing the table
now for ops with no immediate consumer would waste 32 B per
backend and force a re-freeze if any signature changes before
first use.

### 6.5 Return op via memory-indirect call (`call [ops_ptr + N]`)

**Proposal.** Skip the intermediate `mov rax, [rax + N]` load —
use `call [rax + N]` (FF /2 memory form; supported by the encoder
per `encode_call` arm at line 3232).

**Rejected for R16.M1.** Two-instruction load-then-call is easier
to null-guard: with memory-indirect `call [rax + N]`, we'd need
`cmp qword [rax + N], 0` before the call to enforce the null-slot
semantics, which is a 6-byte cmp encoding. Two loads (op_ptr into
rax, then cmp rax, 0) is the same instruction count with a
cleaner mental model. Revisit if profiling shows dispatch cost
matters (§8).

## 7. Invariants

### 7.1 Fixed slot offsets

`VOPS_{READ,WRITE,OPEN,CLOSE,LOOKUP,CREATE,UNLINK}_OFFSET` are frozen
for the entire R16 series. Any change is a **layout re-freeze
issue** that bumps a `VOPS_LAYOUT_VERSION` constant, updates §2's
table, and re-runs the witness against the new offsets.

### 7.2 Signature-per-slot freeze

The argument-register set for each op is pinned by §2. Backends
that implement an op MUST accept exactly the argument set for that
slot — extra bytes ignored, missing bytes not permitted.

### 7.3 Null-slot semantics are contract

If a backend leaves a slot null (unsupported op), every consumer
gets `VOPS_ERR_NOT_SUPPORTED` back. Backends must NOT return other
error codes for "not implemented" — the null-slot convention is
the single source of truth for "this backend doesn't support this
op." Backends that partially implement an op (e.g., `read` that
only handles offset==0) return their own error codes.

### 7.4 Ops_ptr set-before-type-published (vnode side)

Per vfs-layout.md §7.5 — the vnode's `ops_ptr` slot is written
before its `type` slot is set to a non-FREE value. Our dispatcher
tolerates a lifecycle bug (null ops_ptr on a live vnode) by
returning E_NOT_SUPPORTED, but well-behaved backends never hit
this path.

## 8. Performance sketch

- **Cost per dispatch (hot cache)**: 2 loads + 2 cmp/je + 1 indirect
  call = ~5-6 cycles + call/ret pair overhead. On modern µarch this
  is a branch-predicted indirect call once the ops_ptr is warm.
- **Cost per dispatch (cold vops table)**: +1 cache-line fill.
  Vops tables per backend fit in ~64 B, so once warmed they stay
  in L1D across all dispatches on that backend.
- **Table-scaling**: `1 backend = 1 vops table = 56 B`. Even at
  10 backend types (theoretical R19 scope) we're at 560 B of
  static rodata — negligible.

## 9. Growth plan (R16.M4 and beyond)

The 7-op minimum ships this issue. Growth events are all
**layout re-freezes**:

- **R17.M1 — `stat`**: add slot +56 (bringing table to 64 B).
- **R17.M2 — `mmap`**: add slot +64.
- **R18.M1 — `ioctl`**: add slot +72.
- **R18.M2 — `fsync`**: add slot +80.
- **R19.M1 — `truncate`, `seek`**: add slots +88, +96.

Each re-freeze increments `VOPS_LAYOUT_VERSION`, updates the table,
and re-runs every backend's slot-fill verification. No dispatch
helper changes shape — only the OP_OFFSET immediate changes for
new ops.
