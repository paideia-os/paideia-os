# PaideiaOS — paideia-as Calling Convention Specification

**Status:** Draft v0.1
**Date:** 2026-06-17
**Scope:** Concrete specification of the PaideiaOS-native calling convention (the "paideia-cc"), the System V AMD64 ABI bridge thunks, exception/interrupt entry conventions, cross-AS IPC entry sequences, varargs, position-independent code rules, and the stack frame layout. Addresses AS3 from `custom-assembler.md`.

**Hard inputs:**
- `custom-assembler.md` §8 — the binding calling-convention design (Q-A6 custom PaideiaOS-native convention with System V interop bridge).
- `ipc/wait-free-dataflow.md` — the IPC primitive's session-typed channel boundary uses this convention internally; the IPC entry sequence is normative.
- `kernel/scheduler.md` — context switch saves/restores per this convention.
- `capabilities/linearity-and-tags.md` — R12/R13 hold LAM-tagged capability handles; the convention preserves linearity discipline.
- `security/pq-trust-root.md` — the operational signer's call sites use this convention.

## 1. Register file partition

Per `custom-assembler.md` §8.2.

### 1.1 General-purpose registers (GPRs)

| Band | Registers | Discipline | Caller/Callee | Usage |
|---|---|---|---|---|
| **General** | RAX, RBX, R8, R9, R10, R11 | unrestricted | RAX/RCX/RDX/R8/R9/R10/R11 caller-saved; RBX callee-saved | Scratch, return values, computation |
| **Capability** | R12, R13 | linear | both callee-saved | Capability handles (LAM-tagged) |
| **Effect** | R14, R15 | affine | both callee-saved | Effect environment pointer, handler stack |
| **Argument** | RDI, RSI, RDX, RCX | unrestricted | caller-saved | First 4 integer args |
| **Reserved** | RSP, RBP, RIP | reserved | RSP/RBP callee-saved | Stack, frame, instruction |

### 1.2 Vector registers (XMM/YMM/ZMM)

| Band | Registers | Caller/Callee | Usage |
|---|---|---|---|
| **Scratch SIMD** | ZMM0–ZMM15 | ZMM0–ZMM7 caller-saved; ZMM8–ZMM15 caller-saved | Floating-point / SIMD args (ZMM0–ZMM3), computation |
| **Reserved SIMD** | ZMM16–ZMM31 | callee-saved | Vectorized PQ-crypto state; non-PQ callees preserve |

K-mask registers (K0–K7) are all caller-saved; treated as scratch.

### 1.3 Argument-passing slots

The first arguments use registers in declaration order:

| Position | Integer / pointer | Floating / SIMD | Capability |
|---|---|---|---|
| 1 | RDI | ZMM0 | R12 |
| 2 | RSI | ZMM1 | R13 |
| 3 | RDX | ZMM2 | (stack, capability-tagged slot) |
| 4 | RCX | ZMM3 | (stack) |
| 5+ | stack | stack | stack |

Mixed-class arguments: a function `(a: Cap, b: u64, c: Cap, d: f64)` places `a` in R12, `b` in RDI, `c` in R13, `d` in ZMM0.

### 1.4 Return values

| Type | Register |
|---|---|
| Integer / pointer | RAX |
| Capability | R12 |
| Floating / SIMD | ZMM0 |
| Tuple (small, fits in registers) | RAX + RDX for integers; RAX + R12 for mixed; ZMM0 + ZMM1 for floats |
| Large struct (> 16 bytes) | Caller provides RDI as return-pointer; callee writes through it; returns the pointer in RAX |

## 2. Stack frame layout

### 2.1 Stack alignment

At every `call` boundary, RSP is **64-byte aligned** (in contrast to System V's 16-byte alignment). The 64-byte alignment accommodates ZMM saves without unaligned-access penalties.

### 2.2 Frame structure

```
   high address
   ┌─────────────────────────────────┐
   │ Caller's arguments on stack     │  (positions 5+, varargs)
   ├─────────────────────────────────┤
   │ Return address                  │  pushed by CALL
   ├─────────────────────────────────┤  ← RSP at entry (before prologue)
   │ Saved RBP (if frame-pointer set) │
   ├─────────────────────────────────┤  ← RBP after prologue
   │ Saved callee-saved registers    │
   │   RBX, R12, R13, R14, R15       │
   │   ZMM16..ZMM31 (if used)        │
   ├─────────────────────────────────┤
   │ Local variables                 │
   ├─────────────────────────────────┤
   │ Outgoing argument area          │  for any function calls this function makes
   ├─────────────────────────────────┤  ← RSP, 64-byte aligned at call sites
   │ (no red zone — RSP+0 is below   │
   │  any storage)                   │
   ▼  low address
```

### 2.3 Outgoing argument area

For variadic or many-argument calls, the outgoing argument area is pre-allocated at the top of the frame (just below the saved registers + locals). The size is computed at compile time from the maximum call's arguments.

### 2.4 No red zone

System V allows 128 bytes below RSP to be used by leaf functions without adjusting RSP. PaideiaOS does *not* allow this because exception/interrupt entry (kernel-mode) would corrupt the red zone. RSP must always point to or above the top of allocated storage.

### 2.5 Stack-overflow detection

A guard page (4 KiB, non-readable, non-writable) sits at the bottom of each thread's stack. Stack overflow triggers a page fault; the kernel's pager (per `kernel/memory-model.md` §6) reports stack overflow to the supervisor.

For large frames (> 4 KiB), the prologue must "probe" the stack by touching each page in descending order (writing a known value) to ensure the guard page would be hit by sequential growth, not skipped. The toolchain emits probes automatically for frames > 4 KiB.

## 3. Function prologue and epilogue

### 3.1 Standard prologue

```asm
function:
   push rbp                  ; save caller's frame pointer
   mov rbp, rsp              ; establish this frame's pointer
   sub rsp, <local_size>     ; allocate locals + outgoing args
   ; save callee-saved registers used by this function:
   mov [rbp - 8], rbx
   mov [rbp - 16], r12
   mov [rbp - 24], r13
   mov [rbp - 32], r14
   mov [rbp - 40], r15
   ; ZMM16..ZMM31 saves (if used):
   vmovdqa64 [rbp - 64 - <align>], zmm16
   ...
   ; stack-probe for large frames:
   mov rax, rsp
   mov rcx, <pages>
.probe:
   mov [rax], rax            ; touch the page
   sub rax, 0x1000
   loop .probe
```

The toolchain omits saves for callee-saved registers not actually used (per-function liveness analysis).

### 3.2 Standard epilogue

```asm
   ; restore callee-saved registers:
   mov rbx, [rbp - 8]
   mov r12, [rbp - 16]
   mov r13, [rbp - 24]
   mov r14, [rbp - 32]
   mov r15, [rbp - 40]
   vmovdqa64 zmm16, [rbp - 64 - <align>]
   ...
   ; tear down frame:
   mov rsp, rbp
   pop rbp
   ret
```

### 3.3 Leaf-function prologue

A leaf function (one that makes no calls) with frame size ≤ 64 bytes may skip the frame pointer:

```asm
leaf_function:
   sub rsp, <local_size>     ; align to 64 bytes
   ; ... body ...
   add rsp, <local_size>
   ret
```

Leaf functions cannot save callee-saved registers (except by spill); they typically use only caller-saved.

### 3.4 Tail-call epilogue

The `#[tailcall]` optimization pass (per `custom-assembler.md` §6.4) replaces final `call + ret` with `jmp` when the ABI allows (same calling convention, no frame remnants needed by callee).

## 4. Effect environment (R15)

### 4.1 Layout

R15 holds the address of the active effect-handler table. The table is per-AS (one for each address space) and per-thread (effect handlers are installed by user code, not by the AS).

```
R15 → ┌─────────────────────────────────────┐
      │ effect_table_id : u32                │  type-tag for verification
      │ size            : u32                │  number of effects
      │ effect_0_handler: u64                │  pointer to handler routine
      │ effect_0_state  : u64                │  handler-specific state
      │ effect_1_handler: u64                │
      │ effect_1_state  : u64                │
      │ ...                                  │
      └─────────────────────────────────────┘
```

### 4.2 Effect dispatch

To invoke effect `Ipc.enqueue`:

```asm
   mov rax, [r15 + Ipc_enqueue_handler_offset]
   call rax
```

Two instructions to dispatch (mov + call). The handler offset is a compile-time constant per effect.

### 4.3 Handler installation (`with H handle E`)

```asm
   ; push new handler:
   sub rsp, 16                          ; allocate save slot
   mov [rsp], r15                       ; save old effect env
   mov [rsp + 8], <handler_id>          ; record which we replaced
   mov rax, <new_handler_for_E>         ; new handler routine
   mov [r15 + E_handler_offset], rax    ; install
   ; run the with-block:
   call <with_block>
   ; restore:
   mov r15, [rsp]
   add rsp, 16
```

This pattern is generated by the elaborator; user code does not write it directly.

### 4.4 R15 is callee-saved

A called function may temporarily change R15 (e.g., to install a sub-handler) but must restore the original value on return. If the function panics or aborts before restoration, the kernel's exception-handler restores from the per-CPU shadow.

### 4.5 Per-CPU shadow

The kernel maintains a per-CPU shadow of the active R15 value. On exception/interrupt entry, the shadow is consulted to recover the effect environment for the interrupted thread. The shadow is updated on every context switch.

## 5. Capability registers (R12, R13)

### 5.1 LAM-tagged handles

R12 and R13 hold LAM-tagged 64-bit capability handles (per `capabilities/linearity-and-tags.md` §2.1). The high 15 bits are tag bits; the low 49 bits are pointer.

### 5.2 Linearity discipline

R12 and R13 are *linear* — the convention requires that on entry to a function:
- R12 contains a linear capability (if the function's signature declares one).
- R13 may be unused or contain a second linear cap.

On return:
- R12 must be either consumed (released back via the substructural type system) or returned as the call's capability result.
- R13 must be similarly accounted.

The toolchain emits checks that detect missing consumption at compile time.

### 5.3 Software-LAM fallback

On hardware without LAM (per CAP-Q6), the toolchain emits inline masking before every dereference of R12/R13 (per `capabilities/linearity-and-tags.md` §6).

## 6. Argument passing — detailed

### 6.1 Integer / pointer arguments

The first 4 integer/pointer arguments go in RDI, RSI, RDX, RCX. Further arguments go on the stack, 8-byte aligned, in declaration order.

### 6.2 Capability arguments

The first 2 capability arguments go in R12 and R13. Further capability arguments go on the stack, with their stack slots marked in a side-table (the toolchain emits a sidecar metadata record per call).

### 6.3 Floating / SIMD arguments

The first 4 FP/vector arguments go in ZMM0–ZMM3. Further go on the stack, 64-byte aligned per ZMM.

### 6.4 Struct arguments

| Struct size | Passing |
|---|---|
| ≤ 16 bytes (fits in 2 GPRs) | Passed in 2 successive GPR slots |
| 16 < size ≤ 32 bytes | Passed in 2 successive ZMM slots |
| > 32 bytes | Caller provides a pointer (in the next argument register); callee reads through it; the storage is on the caller's stack |

### 6.5 Varargs

A variadic function takes a fixed prefix of arguments per the standard rules, then a `VarArgs` capability as the next argument:

```paideia-as
let printf : (fmt : String, args : VarArgs) -> i32 !{io} = ...
```

The `VarArgs` cap is itself passed in R12 (or the next available capability slot); it holds a count and a pointer to a heap-allocated array of typed values. The caller constructs this array; the callee reads it.

This is more structured than System V's hidden register/stack layout for varargs; the cost is one allocation per varargs call (mitigatable by stack-allocation when the count is known at compile time).

## 7. Callee-saved vs caller-saved discipline

### 7.1 Callee-saved registers

The callee must preserve:
- RBX, RBP, R12, R13, R14, R15
- RSP (always preserved by ret)
- ZMM16–ZMM31

The callee saves these on entry (if used) and restores on exit.

### 7.2 Caller-saved registers

The caller must save (if it needs the value across the call):
- RAX, RCX, RDX, RSI, RDI, R8, R9, R10, R11
- ZMM0–ZMM15
- K0–K7
- Flags register (RFLAGS), x87 state

### 7.3 Why the split

The PaideiaOS-native split makes the capability and effect registers (R12, R13, R14, R15) callee-saved. This reflects the project's pillars:
- Capabilities are heavyweight (linear discipline, LAM tags); callers benefit from not having to save them on every call.
- The effect environment is similarly heavyweight.

The cost: callees that *do* modify these registers (which are uncommon) pay slightly more for the save/restore. The benefit dominates because most callees don't touch them.

## 8. System V AMD64 ABI bridge

The bridge thunks (per `custom-assembler.md` §8.6) wrap calls across the PaideiaOS-native and System V boundaries. Used for:

- The ACPICA bubble (per `acpi/acpica-bubble.md` §10.3 — every OSL function entry).
- The WASM/VM jail (when calling into the wasmtime C runtime per `runtime/wasm-vm-jail.md`).
- Any future C-language component.

### 8.1 Native-to-SysV (calling C from paideia-as)

```asm
; Save PaideiaOS-native R15 (effect env):
push r15
xor r15, r15                          ; clear; C has no effect env

; Convert capability arguments to native form (if applicable):
; (the bridge knows from the signature; this is generated code)
mov rdi, [r12]                        ; deref the cap to its target

; Realign stack to 16 bytes (System V):
push rdi                              ; placeholder
mov rdi, ... (actual arg from R12)

; Call C function:
call c_function

; Restore PaideiaOS-native R15:
add rsp, 8                            ; remove placeholder
pop r15

; Translate result back (if applicable).
ret
```

The actual generated bridge is more elaborate; this is a sketch.

### 8.2 SysV-to-native (calling paideia-as from C)

The reverse: the entry point converts System V args to PaideiaOS-native and dispatches.

### 8.3 Cost

Each bridge thunk adds ~30 cycles (per `acpi/acpica-bubble.md` §10.3) — acceptable for the low-call-rate use cases (ACPICA, the wasmtime host functions).

## 9. Exception and interrupt entry

### 9.1 Kernel exception entry

When the CPU takes an exception (page fault, division by zero, etc.) in kernel mode:

```
1. Hardware: pushes RIP, CS, RFLAGS, RSP, SS onto the per-CPU IST stack.
2. Hardware: jumps to the IDT-specified handler entry point.
3. Software (entry stub, generated by toolchain):
   a. Save all caller-saved GPRs.
   b. Save R12, R13 (capability) and R14, R15 (effect) to the per-CPU shadow.
   c. Save XSAVE-area state (ZMM, K-regs, x87, MXCSR).
   d. Switch to the exception handler's stack (per-CPU IST).
   e. Call the C-language (paideia-as language, really) exception handler.
4. On return from the handler:
   a. Restore the saved state.
   b. iretq.
```

### 9.2 User exception entry (via syscall instruction)

PaideiaOS does not use `syscall`/`syscallq` (the legacy Linux mechanism). Inter-AS communication is the IPC primitive (per `ipc/wait-free-dataflow.md`), which uses a different entry path. See §10.

### 9.3 Interrupt entry

Similar to exception entry, but RIP-pointing-into-userspace is the common case:

```
1. Hardware: pushes interrupted context onto IST stack.
2. Software:
   a. CR3 swap (KPTI per Q15 / MEM-Q8) — switch from user to kernel CR3.
   b. Save caller-saved state.
   c. Save R12, R13, R14, R15 to the per-CPU shadow.
   d. Mark the interrupted thread as "preempted".
   e. Run the IRQ handler.
3. On return:
   a. Restore caller-saved state.
   b. Restore R12, R13, R14, R15 from per-CPU shadow (or the next-scheduled thread's, if context switched).
   c. CR3 swap back to user.
   d. iretq.
```

### 9.4 Mitigation insertion

Per SCH-Q10: the kernel-to-user transition emits IBRS write, RSB stuff, BHB clear (full mitigations under Q15 default). When the transition is intra-AS (rare but possible), the optimization skips these.

## 10. Cross-AS IPC entry

The IPC primitive's wait-free dataflow (per `ipc/wait-free-dataflow.md`) does not use exception entry. Instead, it uses dedicated entry trampolines.

### 10.1 Enqueue path (intra-AS)

```asm
; R12 = SendCap (verified at compile time via type system)
; R13 = Message (or message pointer)
; Other registers per the function call convention

; Dispatch via R15 effect environment (the Ipc effect handler):
mov rax, [r15 + ipc_enqueue_handler_offset]
call rax
```

### 10.2 Cross-AS dispatch (when producer and consumer are in different ASes)

When the channel's consumer is in another AS, the IPC primitive's slot is shared memory (per IPC §10.4); the producer writes to the shared slot; the consumer reads from its mapping. The cross-AS *notification* (wake) is per SCH-Q9 (UMWAIT-monitored or IPI).

If a notification fires while the consumer's AS is not active on a CPU, the scheduler context-switches to the consumer; the standard context-switch sequence applies:

```
1. Scheduler decides to switch to the consumer's TCB.
2. Save outgoing thread's GPRs, R12, R13, R14, R15, XSAVE-area.
3. Switch CR3 to consumer's AS (with PCID).
4. Restore consumer's GPRs, R12, R13, R14, R15, XSAVE-area.
5. iretq into consumer.
```

For same-AS context switches (per SCH-Q10), skip the CR3 swap and the cross-AS mitigations.

## 11. Position-independent code

### 11.1 PIC by default

All code is position-independent. Data references use RIP-relative addressing:

```asm
   ; Load a global:
   mov rax, [rip + global_data]
   ; Take its address:
   lea rax, [rip + global_data]
```

The toolchain enforces no absolute-address references in code segments (the `.text` section is strictly read-only and PIC).

### 11.2 Why PIC

- KASLR (kernel address-space layout randomization, future enhancement) requires PIC code.
- The kernel may be loaded at different addresses across boots.
- User-mode shared libraries (if any future userland uses them) require PIC.

### 11.3 GOT and PLT

PaideiaOS does *not* use the System V GOT (Global Offset Table) or PLT (Procedure Linkage Table) idiom. Static linkage is the only model in phase 1–2. Dynamic linkage is deferred to phase 3+ (and may use a different mechanism — capability-based dynamic loading via the functor module system).

## 12. Examples

### 12.1 A leaf function

```paideia-as
let add_one : (x : u64) -> u64 = fn x -> x + 1
```

Generated assembly:

```asm
add_one:
   lea rax, [rdi + 1]
   ret
```

(No prologue/epilogue; leaf with no frame.)

### 12.2 A function consuming a capability

```paideia-as
let close_channel : (ch : ChannelCap ↓) -> unit = fn ch ->
  perform Ipc.close ch
```

Generated assembly (sketch):

```asm
close_channel:
   ; R12 = ch (linear)
   ; Dispatch Ipc.close via R15:
   mov rax, [r15 + ipc_close_handler_offset]
   call rax
   ; ch is consumed; nothing to return; result = unit
   xor rax, rax
   ret
```

### 12.3 A function calling System V

```paideia-as
extern "system-v" let strlen : (s : *u8) -> usize !{io_aux}

let log_message : (msg : *u8) -> unit !{audit_log} = fn msg ->
  let n = strlen(msg)
  audit_write(msg, n)
```

Generated assembly (the bridge):

```asm
log_message:
   push rbp
   mov rbp, rsp
   sub rsp, 16
   ; Save R15:
   mov [rbp - 8], r15
   ; Bridge to System V:
   mov rdi, rsi          ; (msg → System V arg 1)
   xor r15, r15           ; clear effect env for C
   sub rsp, 8             ; align stack to 16 bytes
   call strlen            ; System V call
   add rsp, 8             ; restore alignment
   ; rax = n
   mov r15, [rbp - 8]     ; restore R15
   ; Continue with native code:
   mov rdi, rsi           ; msg
   mov rsi, rax           ; n
   call audit_write       ; PaideiaOS-native call
   ; Cleanup:
   leave
   ret
```
