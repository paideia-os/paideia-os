# 1. Scope

Land the **second stable public entry point** on the `paideia-as-runtime` /
`paideia-as-emit` pair: an operand-level resolver that rewrites every
reloc-producing operand in a slice of `Instruction`s into a form that
`emit_instruction` accepts (no `SymbolRef`, `LabelRef`, `MemRipRelSym`,
`MemSymIndexed` remain), given a caller-supplied name-to-address table and a
label-to-index map.

```rust
pub fn resolve_symbols(
    instructions: &mut [Instruction],
    symbols:      &SymbolTable,
    labels:       &LabelMap,
    policy:       ResolvePolicy,
) -> Result<(), ResolveError>;
```

Downstream consumer #1 is the paideia-os Phase 10 WASM jail: it will bump-load
guest bytecode into a JIT page, seed a `SymbolTable` with the addresses of
`wasm_trap_handler`, `wasm_grow_memory`, etc., then call `resolve_symbols`
followed by `emit_instruction` in a loop. Consumer #2 is the paideia-as
self-hosting driver (v0.21+): compiled `.pdx` modules that reach the runtime
carry a `SymbolTable` populated by the loader's fixup pass.

**In scope:**

- One new module tree in the runtime crate:
  - `crates/paideia-as-runtime/src/symbol_table.rs`
  - `crates/paideia-as-runtime/src/label_map.rs`
  - `crates/paideia-as-runtime/src/resolve.rs`
- One new integration-test file `crates/paideia-as-runtime/tests/resolve_api.rs`
  with **8 canary tests + 4 discipline tests** (matching AC + belt-and-braces).
- Public re-export chain: `paideia-as-runtime` (owner) → `paideia-as-emit`
  (re-export) so JIT callers link one path (parallel to how `CodeBuffer`
  re-exports today via `paideia-as-emit/src/lib.rs:16`).
- Documentation update: append a `## Symbol resolution` section to
  `design/toolchain/dynamic-emit.md` (created by #1005; extended here).

**Out of scope (owned by later issues or backtrack candidates):**

- **Byte-level backpatching.** The tactical shorthand
  (`design/roadmap/paideia-as-tactical-issues.md:2291-2292`) writes
  `resolve_symbols(&mut Buf, &SymbolTable)`. That shape is achievable but requires
  a companion `emit_instruction_with_fixups` producer that landed neither at
  d2803b3 nor in the roadmap. §7.1 documents the byte-level shape as a
  named-and-owned backtrack; the operand-level shape shipped here composes
  cleanly with the strict pre-flight in `emit_instruction` (see §2.4).
- **rel8 tightening for LabelRef.** Layout uses `Mnemonic::estimated_size`
  (already conservative). rel8 tightening is a follow-up (§7.2).
- **Iterative layout convergence.** Upper-bound layout leaves ≤8 bytes of slack
  per Jmp. Convergence to tight offsets is §7.3.
- **Symbol resolution over already-emitted bytes.** No back-patching path.
  A caller who has bytes but no `Instruction`s must decode first — outside
  runtime's remit.

# 2. Design premises drawn from the tree at HEAD d2803b3

## 2.1 The two operand kinds we must resolve

`paideia-as-runtime/src/instruction.rs:722-808` at d2803b3 defines four
reloc-producing operand variants, and `paideia-as-emit/src/lib.rs:142-155`
rejects every one of them at pre-flight:

```rust
for op in &ins.operands {
    if matches!(op, Operand::Var { .. })                           { return Err(InvalidOperand); }
    if matches!(op, Operand::SymbolRef  { .. }
                  | Operand::LabelRef   { .. }
                  | Operand::MemRipRelSym  { .. }
                  | Operand::MemSymIndexed { .. }) { return Err(UnresolvedRelocation); }
}
```

The v0.20 resolver targets exactly those four variants. `Operand::Var` is a
different lineage (unresolved local binding — belongs to the elaborator's
`resolve_var_operands` pass at `crates/paideia-as/src/resolve_var_operands.rs`)
and stays out of scope.

## 2.2 Symbols are external, labels are internal

- **Symbol** (`Operand::SymbolRef`, `Operand::MemRipRelSym`,
  `Operand::MemSymIndexed`): the address lives outside the slice being resolved
  — kernel-exported (WASM trap handler), PLT stub, loader-fixed data. Caller
  provides address in the `SymbolTable`.
- **Label** (`Operand::LabelRef`): the target is another instruction in the
  same slice. The resolver computes its byte-offset from the slice layout.

The AC signature in the issue ("SymbolTable" only) is not enough. LabelRef is
in the pre-flight rejection set at `paideia-as-emit/src/lib.rs:149`, so without
a companion `LabelMap` the resolver couldn't clear the pre-flight for any
control-flow-carrying slice. **§4.2 introduces `LabelMap` as a peer to
`SymbolTable`**, keyed by label name → instruction index in the slice; layout
converts index to byte-offset via `Mnemonic::estimated_size`.

## 2.3 The runtime crate is `no_std`; SymbolTable / LabelMap must be too

`crates/paideia-as-runtime/Cargo.toml` has no encoder dependency in `[dependencies]`
(only `[dev-dependencies]`), and `src/lib.rs:8-9` declares
`#![no_std] extern crate alloc;`. #1019 v2's verification recipe asserts
`cargo check --target x86_64-unknown-none -p paideia-as-runtime` passes, and
that check is load-bearing for the crate's identity as a linkable runtime
library.

Consequences for #1024:

- `SymbolTable` cannot depend on `std::collections::HashMap`. Use
  `alloc::collections::BTreeMap<alloc::string::String, u64>` (deterministic
  iteration order — bonus for byte-identical emit reproducibility across
  hosts, which is a v0.20 exit criterion).
- `LabelMap` uses the same `BTreeMap<String, usize>` shape.
- `ResolveError` variants use `alloc::string::String` for name payloads.
- The resolver code path is `alloc`-only. No `HashMap`, no `format!` in hot
  path (build error strings once at the failure site).

## 2.4 Interaction contract with `emit_instruction`

`emit_instruction`'s pre-flight is intentionally strict:
`Operand::SymbolRef | LabelRef | MemRipRelSym | MemSymIndexed` → immediate
`EmitError::UnresolvedRelocation`. That means the following two-line call
pattern is the invariant we're building:

```rust
resolve_symbols(&mut insts, &syms, &labs, ResolvePolicy::AbsoluteImm64)?;
for ins in &insts {
    emit_instruction(&mut buf, ins.clone())?;
}
```

If `resolve_symbols` returns `Ok(())`, no instruction in `insts` still carries
a reloc-producing operand. This is the property the discipline canary in §5
tests directly: post-resolve, feed the slice through emit and assert
`Ok(())` on every call.

## 2.5 Estimated-size infrastructure already exists

`paideia-as-runtime/src/instruction.rs:1066-1165` (and beyond) implements
`Mnemonic::estimated_size(&self, operands: &[Operand]) -> u32` — a
**conservative upper bound** on the encoded byte length per mnemonic. It's
already operand-aware (e.g., `Jmp` returns 7 or 8 bytes for `MemSymIndexed`
depending on whether the index register is R8+). The resolver's layout pass
reuses this verbatim — no new estimation infrastructure needs to land in
#1024. The upper-bound imprecision is acceptable: rel32 (±2GiB range) is
saturated on real workloads; the ≤8-byte-per-instruction overestimate is not
observable in the emitted bytes when we use `MemRipRel { disp: i32 }` because
the encoder always emits the full 4-byte disp32 form.

# 3. API signature and shape

## 3.1 The one public function

```rust
/// Resolve every `SymbolRef`, `LabelRef`, `MemRipRelSym`, and `MemSymIndexed`
/// operand in `instructions` into a form `emit_instruction` accepts.
///
/// After a successful return, no instruction in `instructions` carries a
/// reloc-producing operand. Feeding the slice through `emit_instruction`
/// will not return `EmitError::UnresolvedRelocation`.
///
/// # Errors
///
/// See [`ResolveError`] for the taxonomy. On error, `instructions` may have
/// been **partially mutated** (some entries rewritten, some not) — callers
/// that need transactional semantics should snapshot the slice with `clone`
/// before the call. This is documented and tested at §5 canary R-9.
///
/// # Stability
///
/// This function is part of the paideia-as v0.20 stable public API. Signature
/// changes require a major-version bump; adding variants to [`ResolveError`]
/// or [`ResolvePolicy`] is compatible because both enums are
/// `#[non_exhaustive]`.
pub fn resolve_symbols(
    instructions: &mut [Instruction],
    symbols:      &SymbolTable,
    labels:       &LabelMap,
    policy:       ResolvePolicy,
) -> Result<(), ResolveError>;
```

Owned in `crates/paideia-as-runtime/src/resolve.rs`. Re-exported from
`crates/paideia-as-runtime/src/lib.rs`. Re-exported from
`crates/paideia-as-emit/src/lib.rs` (top of the file, near the existing
`pub use paideia_as_runtime::Instruction`) so JIT callers link one path.

## 3.2 `SymbolTable`

```rust
/// Caller-owned map from external symbol name → resolved address.
///
/// Populated by the caller (WASM jail, loader, dynamic linker) with addresses
/// known before `resolve_symbols` runs. Addresses are `u64` absolute values
/// in the process's virtual address space at the point where the emitted code
/// will execute.
///
/// Uses `BTreeMap` (not `HashMap`) for two reasons:
/// 1. `no_std + alloc`-only compatibility (`HashMap` requires `std`).
/// 2. Deterministic iteration order — supports byte-identical emit across
///    hosts (v0.20 exit criterion).
#[derive(Default, Debug, Clone)]
pub struct SymbolTable {
    entries: BTreeMap<String, u64>,
}

impl SymbolTable {
    pub fn new() -> Self;
    pub fn insert(&mut self, name: impl Into<String>, address: u64);
    pub fn get(&self, name: &str) -> Option<u64>;
    pub fn contains(&self, name: &str) -> bool;
    pub fn len(&self) -> usize;
    pub fn is_empty(&self) -> bool;
    pub fn iter(&self) -> impl Iterator<Item = (&str, u64)>;
}
```

Chosen `insert(impl Into<String>)` (not `insert(&str)`) so callers writing
tests or lowering paths with `&'static str` don't need `.to_string()`
boilerplate.

## 3.3 `LabelMap`

```rust
/// Map from label name → index in the caller's `instructions` slice.
///
/// The instruction at the mapped index is the label target — its byte
/// offset in the emitted stream (computed by the resolver's layout pass) is
/// the address to substitute for `LabelRef { name }`.
///
/// Constructed by the caller before `resolve_symbols` runs. Typical use:
/// walk the instruction slice once looking for label markers (or receive an
/// externally-populated map from a lowering pass).
#[derive(Default, Debug, Clone)]
pub struct LabelMap {
    entries: BTreeMap<String, usize>,
}

impl LabelMap {
    pub fn new() -> Self;
    pub fn insert(&mut self, name: impl Into<String>, index: usize)
        -> Result<(), DuplicateLabel>;
    pub fn get(&self, name: &str) -> Option<usize>;
    pub fn contains(&self, name: &str) -> bool;
    pub fn len(&self) -> usize;
    pub fn is_empty(&self) -> bool;
}

/// Returned from `LabelMap::insert` when a name was already registered.
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct DuplicateLabel { pub name: String }
```

`insert` returns `Result<(), DuplicateLabel>` (not silent replace) because
duplicate labels are a caller bug and should be caught at map-construction
time — the encoder pipeline emits U1609 for the same condition
(`crates/paideia-as-runtime/src/instruction.rs:783`).

## 3.4 `ResolvePolicy`

```rust
/// How to materialize a resolved symbol address in the operand.
///
/// This is `#[non_exhaustive]` so v0.21+ may add richer policies (e.g., a
/// hybrid `RipRel32IfNear`) without breaking downstream matches.
#[derive(Clone, Copy, Debug)]
#[non_exhaustive]
pub enum ResolvePolicy {
    /// Rewrite `SymbolRef { name, addend }` to
    /// `Imm64((symbols[name] as i64) + addend as i64)`.
    ///
    /// Suitable for the WASM-JIT `mov reg, imm64; call reg` idiom and for any
    /// caller who doesn't know the buffer's emit-time base address. Every
    /// symbol reference costs 10 bytes (`MOV r64, imm64`) — no rel32
    /// tightening.
    AbsoluteImm64,
}
```

**Deliberately shipping only one policy variant in v0.20** to keep #1024 at
~350 LOC. `RipRel32IfNear { emit_base: u64 }` is §7.4's backtrack: it needs
per-instruction byte-offset arithmetic against a known emit base, which the
WASM jail doesn't have at resolve time (the JIT page's virtual address is
learned from `mmap` after resolution). AbsoluteImm64 works for every v0.20
consumer.

## 3.5 `ResolveError`

```rust
/// Errors returned by [`resolve_symbols`].
///
/// `#[non_exhaustive]` — v0.21+ may add variants without a major bump.
#[derive(Clone, Debug, PartialEq, Eq)]
#[non_exhaustive]
pub enum ResolveError {
    /// A `SymbolRef` / `MemRipRelSym` / `MemSymIndexed` operand referenced a
    /// name that is not present in the `SymbolTable`.
    UnknownSymbol {
        /// Instruction slice index where the reference lives.
        instr_index: usize,
        /// Operand index within that instruction (0 = first operand).
        operand_index: usize,
        /// The unresolved symbol name.
        name: String,
    },

    /// A `LabelRef` operand referenced a label not present in the `LabelMap`.
    UnknownLabel {
        /// Instruction slice index where the reference lives.
        instr_index: usize,
        /// Operand index within that instruction.
        operand_index: usize,
        /// The unresolved label name.
        name: String,
    },

    /// A resolved symbol/label address fell outside the encodable range for
    /// the operand form the mnemonic requires (e.g., `MemSymIndexed` requires
    /// the resolved address to fit in i32 for its Abs32 relocation slot).
    OutOfRange {
        /// Instruction slice index where the out-of-range value materialised.
        instr_index: usize,
        /// The resolved address that overflowed.
        resolved: u64,
        /// The encoding form that the mnemonic requires.
        required_form: &'static str,
    },

    /// A `Mnemonic::Vpxor`-family (or other) instruction cannot legally accept
    /// the operand form the resolver would have produced (e.g., a `Jcc` with
    /// an absolute `Imm64` — Jcc encodes rel32 only). Signals a
    /// caller-supplied policy / mnemonic mismatch.
    NotEncodableForMnemonic {
        /// Instruction slice index.
        instr_index: usize,
        /// The mnemonic that rejected the resolved form.
        mnemonic: Mnemonic,
    },
}

impl core::fmt::Display for ResolveError { … }   // no std::error::Error impl
                                                  // in the no_std crate; the
                                                  // emit crate re-exports and
                                                  // adds the impl.
```

`instr_index` + `operand_index` are load-bearing for diagnostics: the WASM
jail wants to point at the exact WASM opcode that lowered to the bad
instruction, and the elaborator wants source spans via the `Instruction.byte_offset_in_text`
side-table.

# 4. Algorithm

Three passes over the instruction slice. Each pass is `O(n)` in the number
of instructions; total work is `O(n + m)` where `m` is total operand count.

## 4.1 Pass 1 — Layout

Compute a monotonically increasing `Vec<u32>` of length `instructions.len()`
giving each instruction's byte-offset in the eventual emitted stream, using
`Mnemonic::estimated_size(operands)` as an upper bound.

```rust
let mut offsets = Vec::with_capacity(instructions.len());
let mut cursor: u32 = 0;
for ins in instructions.iter() {
    offsets.push(cursor);
    cursor = cursor.saturating_add(ins.mnemonic.estimated_size(&ins.operands));
}
```

Upper-bound layout is safe because:
- `SymbolRef → Imm64` always encodes as the 10-byte `MOV r64, imm64` form
  under `AbsoluteImm64` policy — no shrinkage possible after resolution.
- `LabelRef` targets that land within an instruction whose upper-bound
  offset overshoots the actual byte-offset produce **larger** rel32 values
  than needed, but rel32 signed range (±2GiB) trivially accommodates the
  overshoot for any realistic slice.
- `MemRipRel` operands hold a full i32, so upper-bound arithmetic doesn't
  push them out of range.

## 4.2 Pass 2 — Rewrite operands

Walk `instructions` in index order. For each operand, dispatch on variant:

| Input operand                                           | Output operand                                         | Data source           |
|---------------------------------------------------------|--------------------------------------------------------|-----------------------|
| `Reg`, `SegReg`, `Imm64`, `MemSib`, `MemDisp`, `MemRipRel`, `MemSeg` | (unchanged)                                           | —                     |
| `Var { name }`                                          | (unchanged — belongs to elaborator, not the resolver) | —                     |
| `SymbolRef { name, addend }`                            | `Imm64((symbols[name] as i64).wrapping_add(addend as i64))` | `SymbolTable`      |
| `LabelRef { name, addend }`                             | `Imm64((offsets[labels[name]] as i64).wrapping_add(addend as i64))` — encoder's Jmp/Jcc will accept `Imm64` and emit as rel32 to that absolute-offset-in-buffer target under AbsoluteImm64 (a relative computation deferred to the encoder's Jmp Imm64 arm at `crates/paideia-as-encoder/src/encode_instruction.rs`). | `LabelMap` + `offsets` |
| `MemRipRelSym { name, addend }`                         | `MemRipRel { disp: (i32::try_from((symbols[name] as i64).wrapping_add(addend as i64)))? }` — `OutOfRange` if not fits | `SymbolTable`        |
| `MemSymIndexed { name, addend, index, scale }`          | `MemSib { base: <sentinel — no base>, index: Some(index), scale, disp: (…as i32)? }` — layout requires SIB with `base=0b101` (no-base form). See §4.2.1 | `SymbolTable`        |

Every rewrite is in-place via `mem::replace` on the operand.

### 4.2.1 `MemSymIndexed` → `MemSib` caveat

`Operand::MemSib` uses `base: RegId` — no `Option<RegId>`. The encoder's
existing MemSymIndexed emission path (encoder side, `RelocKind::Abs32`)
already handles the no-base SIB form with `base=0b101`. To thread through
that path from a resolved operand, we introduce a **sentinel** convention:
`base = RegId(0xFF)` — a value already outside the valid 0..15 range so
encoder rejection paths at `crates/paideia-as-encoder/src/encode_instruction.rs:158-175`
would fire — combined with a new `Operand::MemDispIndexed` variant:

```rust
/// Absolute-address indexed memory: [addend + index*scale], no base, no RIP.
/// Produced by `resolve_symbols` from `MemSymIndexed`. Encoder emits FF/4
/// ModRM with SIB, base=0b101 (no base), no relocation.
MemDispIndexed {
    disp: i32,
    index: RegId,
    scale: Scale,
}
```

**Backtrack decision (§7.5)**: adding this variant to `Operand` is an
additive change and safe under the "additive-only Operand through v0.20"
discipline in the roadmap. LOC impact: +30 lines (variant + encoder arm).
The alternative — reusing the existing `MemSymIndexed` variant with a
sentinel empty `name` — is fragile and worse than the additive variant.

## 4.3 Pass 3 — Post-condition assert (debug builds only)

```rust
#[cfg(debug_assertions)]
for (i, ins) in instructions.iter().enumerate() {
    for (j, op) in ins.operands.iter().enumerate() {
        debug_assert!(
            !matches!(op, Operand::SymbolRef { .. }
                       | Operand::LabelRef { .. }
                       | Operand::MemRipRelSym { .. }
                       | Operand::MemSymIndexed { .. }),
            "resolve_symbols left reloc operand at [{}][{}]", i, j
        );
    }
}
```

Not observable in release; catches resolver bugs during development.

## 4.4 Partial-mutation semantics on error

The rewrite loop returns `Err(ResolveError)` at the first offending operand.
Prior instructions in the slice have been mutated. This is documented and
tested (§5, canary R-9). Callers who require transactional semantics call
`instructions.to_vec()` before invocation and drop the copy on error. The
alternative — buffering rewrites and applying atomically — would double
peak memory and buy little (the diagnostic-first workflow is: report first
error, caller reconstructs from source).

# 7. Backtrack candidates

## 7.1 Byte-level `resolve_symbols(&mut CodeBuffer, &SymbolTable)` (aspirational AC signature)

**Deferred to v0.21.** Would require:

- A companion `emit_instruction_with_fixups(&mut CodeBuffer, Instruction) -> Result<Vec<RelocSite>, EmitError>` producer that writes placeholder bytes and returns fixup records.
- A byte-patch resolver that walks fixup records and rewrites disp32/imm64 slots in the buffer.

Rejected for #1024 because:

- The producer function is new API (+150 LOC) not accounted for in the r20 tactical LOC budgets.
- The v0.20 consumer (WASM jail) plans to lower one WASM opcode at a time and doesn't need byte-level patching — it composes `resolve_symbols` + `emit_instruction` in a tight loop.
- The byte-level shape becomes attractive when a batch producer emerges (v0.21 self-hosting linker retrofit) — track then.

Reversal cost if adopted later: additive. Both shapes can coexist; no
retrofit to callers of the operand-level shape.

## 7.2 rel8 tightening for LabelRef

Under `AbsoluteImm64` policy plus rel8 tightening, backward-branching Jcc
targets within 128 bytes would encode as 2-byte `Jcc rel8` instead of 6-byte
`Jcc rel32`. Requires a second layout pass with iterative shrinking. Deferred
to v0.21 (pa-r21-XXX); v0.20 consumers don't emit inner loops.

## 7.3 Iterative layout convergence

Upper-bound layout overshoots. A three-round iterative layout tightens to
exact byte-offsets, at the cost of ~40 LOC in the resolver and an
observability requirement that `estimated_size` be monotonic in operand size.
Not needed for v0.20 correctness; deferred.

## 7.4 `RipRel32IfNear { emit_base: u64 }` policy

Rewrite `SymbolRef` → `MemRipRel { disp }` when `symbols[name] - (emit_base + callsite + 5)` fits in i32. Saves 5 bytes per call/jump compared to `AbsoluteImm64`'s `mov r64, imm64; call r64`. Requires the caller to know `emit_base` before resolution — feasible for a JIT that pre-allocates the mmap page, feasible for a loader with fixed load addresses; not feasible for the WASM jail's current design. Land in v0.21 when a consumer requests it.

## 7.5 Reuse `MemSymIndexed` with sentinel empty name instead of adding `MemDispIndexed`

Rejected. Introduces a stringly-typed post-resolve invariant ("empty `name`
means resolved") that would fail silently if a caller populates a
`MemSymIndexed` with `name: ""` for other reasons. The additive `MemDispIndexed`
variant is +30 LOC well spent.

## 7.6 Ship `SymbolTable` and `LabelMap` in `paideia-as-emit`, not `paideia-as-runtime`

The AC ("Touching files: paideia-as-runtime/") points to runtime. The
alternative — shipping in emit — is defensible because `SymbolTable` will
almost always be used adjacent to `emit_instruction`. Rejected because:

- Runtime is `no_std`; emit is not (transitively pulls in `paideia-as-encoder`
  which has `std::error::Error`). Keeping the resolver in runtime preserves
  the no_std verification bar and lets embedded consumers (bootloader,
  microcontroller shims) resolve symbols without linking the encoder.
- The AC is explicit. Deviations from the AC increase future-reader friction.

## 7.7 Delete `Operand::Var` handling from the resolver

Currently the resolver leaves `Operand::Var` unchanged (§4.2 table). Alternative:
return `NotEncodableForMnemonic` on `Operand::Var`. Rejected because the
elaborator's `resolve_var_operands` runs upstream; if `Var` reaches the resolver
it's a compiler bug, not a caller bug, and `emit_instruction` already catches
it with `EmitError::InvalidOperand`. Keeping the resolver silent on `Var`
means both entry points (resolver and emit) have one owner for the diagnostic.

# 8. Cross-repo impact

- **paideia-os Phase 10 (WASM jail).** The consumer that motivates this API.
  Once #1024 lands, the WASM lowering pass can seed a `SymbolTable` at the
  boundary of the WASM VM (trap handlers, memory helpers) and call
  `resolve_symbols` + `emit_instruction` in the per-opcode lowering loop.
  Unblocks paideia-os Phase 10.

- **paideia-os Phase 14 (self-hosting).** The runtime loader for `.pdx`
  modules will maintain a `SymbolTable` populated from the module's PAX
  header at load time; `resolve_symbols` is the fixup pass.

- **paideia-as v0.21 linker retrofit.** The batch linker's
  `crates/paideia-as-linker/src/resolve.rs` runs BLAKE3-hash-based PAX
  resolution. It's the wrong shape for runtime callers. Post-#1024, we
  can plan a shared vocabulary between the batch and runtime resolvers
  in v0.21 — the split is deliberate through v0.20.
