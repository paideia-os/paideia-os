# v0.18 issue #1241 — `emit_enum_match` cmp/jne cascade stamps stale offsets

**Repo**: paideia-os/paideia-as | **Milestone**: v0.18 (#105) | **HEAD**: 2a2b60b (4951 tests green)
**Blocks**: #996 hashmap stdlib, #1003 hash-dispatch, #1004 AVX2 baseline
**Prior art**: #1120 (jump-table variant already fixed via same primitive), #1199 (label placement clarification)

## 1. TL;DR

`visit_match`'s cmp/jne cascade in `crates/paideia-as-elaborator/src/emit_enum_match.rs` records match-arm labels via `register_label`, which snapshots `EmitPassState::estimated_offset` at walker time. That counter never advances by the byte size of `IrKind::Unsafe` bodies — they are deferred (`emit_walker.rs:650-671`) and lowered later by `UnsafeWalker`, whose instructions are inserted with `emission_order = 0` and therefore sort **before** every walker instruction in the `(emission_order, node_id)` order that `emit_text_from_instructions` uses (`crates/paideia-as-emitter-pe/src/text_emitter.rs:87-93`). Cascade-arm labels therefore point `sum(unsafe_body_bytes)` bytes short of the arm's real position. Consequence: `jne match_arm_M_N` lands mid-instruction → SIGILL.

The sibling `visit_match_jump_table` path was fixed under #1120 by anchoring each label to a virtual NOP and calling `insert_label(name, nop_id)` — offsets are then resolved after encoding via `offset_map`. This fix mirrors that pattern for the three surviving cascade-path sites.

## 2. Root cause — evidence

`register_label` stamps walker-time offsets (`emit_pass_state.rs:589-591`):

```rust
pub fn register_label(&mut self, name: String) {
    self.labels.insert(name, self.estimated_offset);
}
```

`insert_label` records an IR node id whose actual byte offset is resolved after encoding (`emit_pass_state.rs:467-469`, consumed at `cmd_build/elf.rs:110-125`):

```rust
for (label_name, instr_id) in label_to_instr {
    if let Some(&byte_offset_u64) = offset_map.get(instr_id) {
        resolved_labels.insert(label_name.clone(), byte_offset_u64 as u32);
    }
}
```

The three offending call sites in `emit_enum_match.rs` (`visit_match`, cmp/jne cascade):
- **line 1485** — `register_label(default_label)` when a default arm appears in the list.
- **line 1535** — `register_label(arm_label)` immediately before each non-default arm's `cmp` (jne cascade target).
- **line 1648** — `register_label(body_label)` for multi-alt or-patterns (`p1 | p2 | ...`).

Why estimated_offset drifts:
1. `emit_walker.rs:650-671` records `IrKind::Unsafe` as pending and returns without `emit_inst` — `estimated_offset` never advances by the Unsafe body's byte size.
2. `UnsafeWalker::run` (`unsafe_walker.rs:1141,1158,1175`) inserts its instructions directly into `arena.instructions_mut()` with `emission_order: 0` — the sentinel value.
3. `emit_walker.rs:162-165` assigns walker instructions monotonic orders starting at 1.
4. `text_emitter.rs:91-93` sorts by `(emission_order, node_id)` — Unsafe body instructions therefore sort **before** every walker instruction in .text.

Net effect: real arm position = `estimated_offset + Σ unsafe_body_bytes`, but the label carries only `estimated_offset`. Drift is exactly the total Unsafe-body byte count.

## 3. Fix approach — mirror #1120

For each of the three sites, emit a virtual NOP anchor via `emit_inst` (which assigns the correct monotonic `emission_order`) and call `insert_label(name, nop_id)`. The label's real offset is then looked up as `offset_map[nop_id]` after encoding — authoritative regardless of where UnsafeWalker's instructions land.

Because `emit_inst` sets `emission_order = next_emission_order++` *before* any subsequent arm-dispatch instruction is emitted, the NOP is guaranteed to sort just before the arm's `cmp`, keeping the label semantically pinned to the arm's dispatch start.

Virtual IrNodeId slots (chosen to fit alongside the existing scheme so no collisions occur — cf. `emit_enum_match.rs:1861-1891` for the pattern reused here):
- **arm_nop_id** = `M*100 + idx*10 + 6` — slot 6 in the per-arm block (0,1,3,4,5 are taken; 2,6,7,8,9 free).
- **default_nop_id** (in-loop default) = `M*100 + 996` — parallel to the post-loop `M*100 + 997` (line 1862).
- **body_nop_id** (multi-alt) = `M*1000 + idx*100 + 98` — safe because `debug_assert!(alts.len() * 2 < 100)` bounds `a_idx*2+1 ≤ 97`.

## 4. Concrete change

**File**: `crates/paideia-as-elaborator/src/emit_enum_match.rs` (`fn visit_match`)

Each of the three replacements has the same shape (~10 LOC each):

```rust
// #1241: Route <label> through label_to_instr; register_label captured a
// pre-Unsafe-body estimated_offset that drifts once UnsafeWalker prepends
// deferred bodies at encode time. Mirrors #1120's fix for end_label.
let nop_id = IrNodeId::new(<slot>).expect("<name> anchor NOP virtual id");
self.emit_inst(nop_id, Instruction {
    mnemonic: Mnemonic::Nop,
    operands: SmallVec::new(),
    encoding_hint: None,
    byte_offset_in_text: None,
    mode: self.current_mode(),
    emission_order: 0,
});
self.state.insert_label(<label>, nop_id);
```

Sites:
- **line 1485** → default_label (slot `M*100 + 996`).
- **line 1535** → arm_label (slot `M*100 + idx*10 + 6`).
- **line 1648** → body_label (slot `M*1000 + idx*100 + 98`).

Estimated net LOC: **+27 in production** (3 sites × ~10 lines − 3 old lines).

## 6. Backtrack candidates

1. **`emit_int_match.rs` has the same shape** at lines 246, 256, 369, 503, 520. Integer-scrutinee matches inside functions with Unsafe bodies will drift identically. Not exercised by #996 (which uses enum matches), but this is the natural next issue if any int-match canary regresses. Recommend filing a companion issue and applying the same pattern in one sweep.
2. **Sentinel `emission_order = 0` in `UnsafeWalker::run`** (unsafe_walker.rs:1141,1158,1175). Alternative fix: assign proper monotonic orders to Unsafe-body instructions, thereby making `register_label`'s stamp semantically correct. Rejected here — broader blast radius, would change .text layout ordering across the entire codebase and re-invalidate every existing golden file. Narrow fix per-site is safer.
3. **Off-by-one in `emit_int_match.rs:503,520`**: `register_label` is called *after* `emit_inst(NOP)`, so `estimated_offset` already reflects the NOP's byte size — label points 1 byte past the NOP anchor. This is a latent adjacent defect worth capturing but not gating #1241.
4. **`patch_label_fixups` merge order** (`cmd_build/elf.rs:113-125`): `label_to_instr` currently overrides `direct_labels`. If a label is inadvertently registered via both paths, the `label_to_instr` value wins — safe today, but a stricter "one canonical route per label" invariant would surface future regressions faster. Consider adding a debug_assert.

## 7. Risk & sequencing

- **Risk**: low. Pattern is byte-for-byte parallel to #1120's landed fix; the sort-order guarantee `(emission_order, node_id)` is already the invariant #1120 relies on. Non-Unsafe matches remain unchanged in observable behaviour because the NOP anchor's `offset_map` entry equals the walker's estimated offset when no Unsafe drift is present.
- **Sequencing**: Land #1241 → re-apply reverted #996 patch → confirm 5 hashmap canaries pass → close #1003 hash-dispatch prerequisite.
- **New symbols**: three additional `match_arm_*` / `match_default_*` NOPs per match expression won't inflate .text meaningfully (each is 1 byte).
