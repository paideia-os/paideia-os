# v0.18 issue #1240 — unsafe-block emit gap (mis-labelled as "labels + control flow")

**Repo**: paideia-os/paideia-as | **Milestone**: v0.18 (#105) | **HEAD**: 19a536f (4942 tests green)
**Blocks**: #996 hashmap stdlib, #1003 hash-dispatch, #1004 AVX2
**Filed by**: workerbee; **repro provided in issue does NOT reproduce**. See §2.

## 1. TL;DR

The reported gap — "unsafe block emit path fires B1705 when block contains local labels + jmp/jz/jnz control flow" — **is a mis-diagnosis**. Labels and control flow inside `unsafe { block: {...} }` work end-to-end today, produce correct rel32 fixups, and disassemble cleanly. The issue's minimal repro (the `clear_loop` / `clear_done` pattern) compiles to a valid ELF object on HEAD `19a536f`.

The **actual** cause of the B1705 that killed the workerbee's 400-line hashmap is a missing encoder arm for `Mnemonic::Mov` in 64-bit mode:

```
mov [reg], imm            ; MemSib{index:None, disp:0}, Imm64
mov [reg + disp], imm     ; MemSib{index:None, disp:!=0}, Imm64
mov [reg + reg*n], imm    ; MemSib{index:Some, ...},     Imm64
mov [reg + reg*n + d], imm; MemSib{index:Some, disp:!=0}, Imm64
```

The mis-attribution was amplified by a secondary defect: `find_failing_instruction` (in `crates/paideia-as/src/cmd_build/diagnostics.rs:88`) returns the smallest `IrNodeId` in the whole table, not the instruction whose encode call actually returned `EncodeError`. Every B1705 today points at "first instruction of the first function". The workerbee saw `mov rax, rdi` at line 28 of `fibhash` (the file's first function) and rationalised backwards to a "labels + control flow" hypothesis.

## 2. Evidence — reported repro does NOT reproduce

The exact block body from the issue, copy-pasted into a naming-compliant module:

```pdx
// scratchpad/label_repro.pdx
module LabelRepro = structure {
  let mut hm_state: [u64; 32] = uninit
  pub let init: () -> () !{} @{} = fn() -> unsafe {
    effects: {}, capabilities: {},
    justification: "init",
    block: {
      lea rax, hm_state
      mov rcx, 32
      clear_loop:
      cmp rcx, 0
      jz clear_done
      dec rcx
      jmp clear_loop
      clear_done:
      mov r9, 0
      mov [rax], r9
    }
  }
}
```

Result on HEAD `19a536f`:

```
$ paideia-as build --emit elf64 label_repro.pdx -o out.o
$ echo $?
0
$ objdump -drwCS out.o
0000000000000000 <init>:
   0: lea    0x0(%rip),%rax  # R_X86_64_PC32 hm_state-0x4
   7: mov    $0x20,%rcx
   e: cmp    $0x0,%rcx
  12: je     20 <init+0x20>       ; jz → clear_done, correctly patched
  18: dec    %rcx
  1b: jmp    e  <init+0xe>        ; jmp → clear_loop, correctly patched
  20: mov    $0x0,%r9
  27: mov    %r9,(%rax)
```

Both forward (`jz clear_done`) and backward (`jmp clear_loop`) references resolve. The extended repro I bisected from `crates/paideia-stdlib/pdx/hashmap_u64_u64.pdx` — 5 labels, mixed forward/backward, indexed SIB memory operands, param references, RIP-relative symbol loads — also produces correct code. The label pipeline is not the problem.

**The real minimal failing case** (found by bisecting the actual stdlib file):

```pdx
pub let f : () -> () !{} @{} = fn() -> unsafe {
  effects: {}, capabilities: {}, justification: "t",
  block: {
    mov rax, 0
    mov [rax], 1   // <-- B1705 fires here; blame lands on line above
  }
}
```

No labels, no control flow. `mov [reg], imm` alone fires B1705.

## 3. Root cause — encoder gap in `encode_mov` (Mode64)

`crates/paideia-as-encoder/src/encode_instruction.rs::encode_mov` (line 1800-2168) exhaustively covers `Mnemonic::Mov` operand shapes in 64-bit mode:

| Shape                                    | Handled? | Line   |
|------------------------------------------|----------|--------|
| `Reg, Reg`                               | yes      | 1915   |
| `Reg, Imm64`                             | yes      | 1920   |
| `Reg, MemSib(idx, no-disp)`              | yes      | 1934   |
| `Reg, MemSib(no-idx, disp)`              | yes      | 1961   |
| `MemSib(no-idx, disp), Reg`              | yes      | 1974   |
| `Reg, MemSib(idx, disp)`                 | yes      | 1987   |
| `MemSib(idx, disp), Reg`                 | yes      | 2013   |
| `Reg, MemDisp`                           | yes      | 2040   |
| `MemDisp, Reg`                           | yes      | 2053   |
| `MemDisp, Imm64`                         | yes      | 2066   |
| `Reg, SymbolRef` / `SymbolRef, Reg`      | yes      | 2078   |
| `Reg, MemRipRelSym` / `MemRipRelSym, Reg`| yes      | 2123   |
| **`MemSib(no-idx), Imm64`**              | **NO**   | —      |
| **`MemSib(idx, disp), Imm64`**           | **NO**   | —      |
| **`MemRipRelSym, Imm64`**                | **NO**   | —      |

Fallthrough goes to line 2165 `Err(EncodeError::Unsupported("mov form not in phase-3-m2-002 minimum"))`, which bubbles as `TextEmitterError::EncodeError` and then B1705.

Cross-reference: the **typed** store variant `Mnemonic::MovSized { width }` (line 467, arms at 537 and 568) already implements the identical shape via `mov_mem_base_disp_imm32_sxt` / `mov_mem_sib_disp_imm32_sxt`. The plain-`Mov` path just never got these arms wired up. `MovSized` is what typed lowering emits; raw asm in `unsafe { block: {...} }` resolves to `Mnemonic::Mov` and hits the untyped path.

## 4. Secondary defect — diagnostic mis-attribution

`crates/paideia-as/src/cmd_build/diagnostics.rs:86` (`find_failing_instruction`):

```rust
pub(super) fn find_failing_instruction(instruction_table: &InstructionSideTable) -> Option<IrNodeId> {
    let mut entries: Vec<_> = instruction_table.entries().iter().collect();
    entries.sort_by_key(|&(&node_id, _)| node_id);
    entries.first().map(|&(&node_id, _)| node_id)
}
```

This returns the lowest-`IrNodeId` entry — deterministically the first instruction of the first function in the file — regardless of which instruction actually returned `EncodeError`. The failing encode call happens deep inside `emit_text_from_instructions` (`crates/paideia-as-emitter-pe/src/text_emitter.rs:96`), which knows the `node_id` it was processing at the point of failure but drops it: the `?`-propagated `TextEmitterError::EncodeError(format!("{:?}", e))` carries the error string, not the id.

Consequence: every B1705 mis-labels its source location. The workerbee's issue report faithfully documented what the compiler said, then hypothesised backwards from a wrong-line span.

## 5. Scope — deliver vs defer

**Deliver in this issue** (~60 LOC in one file):

1. Three new arms in `encode_mov` (Mode64) matching the shapes at `encode_mov_sized` lines 537 and 568. Delegate to the same encoding helpers (`mov_mem_base_disp_imm32_sxt`, `mov_mem_sib_disp_imm32_sxt`); default to W64 semantics with the i32 sign-ext gate already present in `encode_mov_sized`.
2. `MemRipRelSym, Imm64` shape (fourth arm) — needed by patterns like `mov [rip + hm_len], 0`. Emits `48 C7 05 <disp32_placeholder> <imm32>` with a PC32 reloc + placeholder.
3. Thread the failing `node_id` through `TextEmitterError` so B1705 spans the actually-failing instruction. Add a new variant `TextEmitterError::EncodeErrorAt { node: IrNodeId, msg: String }` and prefer it in `find_failing_instruction`.
4. Rename the fallthrough error string from "mov form not in phase-3-m2-002 minimum" to something diagnostic-oriented — e.g. include operand shape via `Debug`.
5. Fixtures under `crates/paideia-as/tests/build-emit/mov_mem_imm/` (see §7).

**Defer** (do not attempt here):
- `[symbol + reg*scale]` memory addressing (parser gap; U1606, not B1705; blocks a different set of stdlib recipes but is orthogonal).
- `imul r64, imm64` short-form (already supported for `imul r64, r64` — the workerbee's fibhash `mov rcx, imm64; imul rax, rcx` sequence works).
- `[symbol]` load without RIP anchor (already covered by `MemRipRelSym`).
- Broader diagnostic overhaul for B1704/B1706 attribution (same root-cause pattern in the same file; do separately).

**Do NOT do**: any change to the parser, AST, unsafe-walker label handling, `Operand::LabelRef`, `label_to_instr`, or `patch_label_fixups`. All are correct.

## 6. Parser / AST / IR changes — NONE

The issue's suggested fix scope ("Local labels in unsafe blocks (label definitions + forward/backward jmp/jz/jnz references)") is entirely mis-directed. For the record, this is what already exists and works:

- Parser: `crates/paideia-as-parser/src/parse_unsafe.rs:354-384` — recognises `ident:` pattern in block body, emits `StmtData::Label { name }`.
- AST: `crates/paideia-as-ast/src/stmts.rs:63` — `StmtData::Label` variant.
- IR side-table: `crates/paideia-as-elaborator/src/unsafe_walker.rs:600-720` — two-pass label collection (pass 1: names → offsets placeholder; pass 2: register per-instruction, alias back-to-back labels via `pending_labels` per #924).
- Operand routing: `crates/paideia-as-elaborator/src/unsafe_walker/symbol_ref.rs:37` — `supports_label_ref(Jcc | Jmp | Call)`; `parse_operand_from_ast:395-398` prefers `LabelRef` over `SymbolRef` when name is in local labels map (#900).
- Encoder: `crates/paideia-as-encoder/src/encode_instruction.rs:3006` (jcc), `:3058` (jmp) — emit placeholder rel32 + `LabelFixup{ instruction_size, byte_offset }`.
- Fixup: `crates/paideia-as/src/cmd_build/fixup.rs:36` — after encoding, compute `label_offset - (fixup.byte_offset + 4)`, patch i32 LE.
- Diagnostics: U1609 (duplicate label), U1610 (unknown reference).

Nothing here needs revisiting. Ship the encoder arms.
