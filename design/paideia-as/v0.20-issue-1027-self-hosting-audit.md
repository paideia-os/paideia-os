# 0. What this audit does

Enumerate what still stands between paideia-as v0.20 (Rust-hosted) and a
paideia-as that can compile itself on paideia-os. Not code; a written audit.
Two axes:

1. **Compiler crate by crate.** For each of the 23 workspace crates, assess
   how far a `.pdx` re-implementation would need the language to reach.
2. **Language feature by feature.** For each Rust construct the compiler
   depends on, mark whether `.pdx` already supports it, whether it parses but
   does not elaborate/emit, or whether it is a real gap.

Output: a ranked gap list, a proposed v0.21 milestone shape, and an explicit
blocker log.

**Scope discipline.** This is the v0.20 audit — it covers self-hosting the
*Rust compiler crates*. It does **not** cover self-hosting the toolchain's
peripheral binaries (LSP, PQ-sign, doc, fmt). Those are Phase 6+ per
`design/toolchain/self-hosting-phase5-plan.md §5`. This audit reaffirms that
deferral and does not reopen it.

**Relation to prior planning.** Two prior documents cover overlapping ground:
`design/toolchain/self-hosting-phase5-plan.md` (crate tier catalogue) and
`design/toolchain/rust-dep-gap-analysis.md` (Rust-dep bucketing). Both were
written at Phase 4 m13, before v0.15–v0.20 landed. This document is the
v0.20-close refresh: same shape, current numbers, current language surface.
Where those documents contradict this one, this one wins.

# 1. Executive summary (read first)

## 1.1 The v0.20 SELF-HOST milestone is *not* self-hosting

v0.20 delivers the **runtime library** (`paideia-as-runtime` + `paideia-as-emit`)
that a `.pdx` self-hosted driver would eventually link against. It does **not**
deliver a self-hosted compiler. Actual self-hosting is a v0.21+ multi-milestone
undertaking that the runtime crate now unblocks.

The v0.20 milestone description spells this out: *"paideia-as-runtime crate:
linkable library with stable public API for JIT/WASM (P10) + eventual
self-hosting (P14)."* Eventual, not now.

## 1.2 Three tiers of gap, ranked by day-count to close

The audit surfaces three categorically different gap classes, in ascending
close cost:

| Class | Count | Weight | Character |
|---|---|---|---|
| **A. Language surface** | 8 concrete features | ~6–12 months of language work | Structural — cannot be worked around by cleverness at the .pdx source layer. |
| **B. Stdlib primitives** | 5 major libraries | ~3–6 months at current velocity | Additive — each unblocks one crate port. |
| **C. Host services** | 4 syscall surfaces | ~1–2 months once paideia-os phase 13 lands | External — depends on paideia-os OS-syscall shape more than paideia-as. |

Class A is the hard problem. Every remaining class-A gap has an open issue
lineage stretching from v0.18 (#994/#995 closures) through today; the closure
kernel landed structurally in v0.19 (#1233 encoder emit) but end-to-end
generic monomorphization (#997c) has not.

## 1.3 The five hard blockers

Ranked by "closes the widest surface":

1. **Generic monomorphization end-to-end (#997c, #994, #995).** Blocks every
   collection type (`Vec<T>`, `HashMap<K,V>`, `Option<T>::map`, iterator
   adapters). Without it, self-hosting requires hand-monomorphized types for
   every T the compiler traffics in — impractical for a 90k-LoC compiler.

2. **A `.pdx` runtime evaluator OR a full lowering path for high-level `.pdx`
   to native code.** Today most stdlib types are declared as parser fixtures;
   only `unsafe { block: {…asm…} }` bodies actually execute. Either
   (a) a `.pdx` interpreter reads elaborated IR and evaluates it, or
   (b) every high-level construct lowers all the way to encoded bytes.
   Path (b) is where the language is trending. See §6.

3. **File I/O + argv + stdout / stderr surface.** The current CLI does
   `paideia-as build path/to/file.pdx --out artifact.efi`. A self-hosted
   `paideia-as build` invoked from paideia-os needs the OS's file namespace,
   command-line argv, and a stdout/stderr write path. The stubs in
   `pdx/speculative/file.pdx` are parser fixtures; none of them execute.
   Depends on paideia-os phase 13 syscall surface.

4. **Serialisation framework (serde-equivalent).** Blocks
   `paideia-as-diagnostics` port (SARIF JSON output + TOML config).
   Either port serde-equivalent (large — 5–10k LoC of derive-macro work) or
   accept a diagnostics-format regression (drop SARIF; hand-roll a plain
   ASCII output format). Recommend the latter for v0.21 and revisit later.

5. **BLAKE3 in `.pdx`.** Blocks `paideia-as-elaborator` (module hash) and
   `paideia-as-emitter-pax` (chunk hashes). Constant-time crypto in `.pdx`
   is feasible but ~500 LoC of careful work. No fundamental language gap;
   just labor.

## 1.4 The three biggest structural surprises since Phase 4 m13

Things this v0.20-close audit surfaces that the m13 catalogue did not:

- **Elaborator is now 65k LoC**, not the m13-quoted 19k. Its port is 3× the
  effort budgeted in the m13 plan. Growth is real elaboration work — cap
  inference, branch merging, borrow walker, effect walker, closure dispatch,
  five distinct emit_* modules — not accidental complexity.
- **paideia-as (CLI) is now 38k LoC**, up from 2k in the m13 plan. It grew
  a full command-dispatch surface (build/check/doc/fmt/test) plus target
  triplets, filesystem discovery, error rendering. The CLI is now larger
  than every non-elaborator crate.
- **Encoder is now 35k LoC**, up from 2k. This is the byte-emission table
  — dispatch on ~600 mnemonic forms. The growth is intrinsic; every v0.13
  through v0.20 encoder issue added ~30 opcodes on average.

Net: the **compiler is now ~145k LoC of Rust**, up from ~93k at Phase 4
close. A `.pdx` port at 1:1 scale is a v0.21–v0.24 arc, not a single
milestone.

## 2.1 Elaborator — the 3× growth explanation

The elaborator triples the m13 quote (19k → 66k LoC) because the elaboration
pipeline expanded across every phase-4 through phase-6 milestone. A partial
enumeration by module name shows the shape:

- Structural (borrow_walker, branch_merge, capture, closure_dispatch,
  data_encoder, derive_fn_sig, effect_walker, name_resolution_walker) —
  Phase 3–4 walker infrastructure.
- Checkers (check_bounds, check_coherence, check_expr, check_fn_ptr_sig,
  check_handler, check_lambda, check_linearity, check_match, check_ordered,
  check_pattern, check_pure) — Phase 4 m4–m8 semantic gates.
- Emitters (emit_arith_lambda, emit_block_body, emit_call, emit_control_flow,
  emit_enum_match, at least 20 more) — Phase 4 m9 emit walker + every
  v0.15–v0.19 add-on.
- Inference (cap_infer, effect_infer, effect_unify, elab_builtin) — Phase 4
  m3 + v0.16 cap-row expansions.

**Port shape implication.** A 1:1 port of the elaborator is ~66k LoC of `.pdx`.
At self-hosting velocity (compiler-in-compiler, careful discipline) that is
easily 6–9 months of focused work. It is the single largest port line item
in the arc.

## 2.2 Encoder — the 15× growth explanation

The encoder ballooned from 2.2k LoC to 34.9k LoC because every v0.13–v0.20
issue added mnemonic-form encoding rules. `encode_instruction.rs` alone is
10k LoC — a single monster dispatch table on `Mnemonic` variants.

**Port shape implication.** The encoder is table-heavy — port is mechanical
per row. A `.pdx` port can plausibly emit the encoder table from a schema
file (JSON, TOML, or a `.pdx`-native data-literal), producing the `.pdx`
encoder source directly. That is a v0.22+ project: schema-driven encoder
generation lets both the Rust encoder and the `.pdx` encoder share one
truth. Recommended v0.22 issue: **schema-driven encoder**.

## 2.3 CLI — the 19× growth explanation

The CLI grew from 2.2k LoC to 38.3k LoC because it now hosts:

- Command dispatch (build/check/doc/fmt/test/lsp) with target triplets.
- Filesystem walker (find `.pdx` files, resolve module paths, load
  `paideia.toml`, resolve dependencies).
- All of `cmd_build.rs`'s emit-format branching (PE/ELF/PAX/PVH).
- 140 `std::fs`/`std::io`/`std::path` sites — the compiler's entire disk
  contact surface lives here.
- 120 `std::process::exit` / `std::process::Command` sites — process
  control + subprocess launch for stage-0 assembler shell-outs.

**Port shape implication.** The CLI is the point of maximum OS coupling.
Its port depends on paideia-os phase 13 (userspace CLI framework) more than
on paideia-as language work. A hybrid "CLI stays Rust, everything below
CLI is `.pdx`" ships value faster; full CLI port comes later.

# 3. Language features audit (checklist)

For each language construct the compiler depends on, mark the current
support level. Legend:

- **STABLE**: end-to-end supported (parses + elaborates + emits + executes).
- **PARSER-ONLY**: parser accepts; elaborator/emitter deferred to future issue.
- **HAND-MONO**: only hand-monomorphized instantiation works (e.g., `OptionU64` yes, generic `Option<T>::map` no).
- **UNSAFE-ONLY**: only expressible via `unsafe { block: {…asm…} }` raw
  assembly.
- **GAP**: not yet designed or implemented.

# 4. Ranked gap list (the seed to v0.21+ milestones)

Same list as §1.3, expanded with implementation notes and issue candidates.
Ranked by **surface-close power**: how many crate ports each gap unblocks.

## Rank 1 — Generic monomorphization end-to-end

**Blocks:** every collection type used by the compiler. Elaborator has 53
`HashMap` sites, IR has 26. Without generic mono, every one of these needs
a hand-mono equivalent — 79+ hand-mono types just to seed the port.

**Existing lineage:** #994 closures, #995 closure call, #997c generic-enum
monomorphization. Structural pieces landed in v0.19; end-to-end for
compiler-heavy types (`HashMap<Ty, Ty>`, `HashMap<Symbol, Ty>`) has not.

**Proposed v0.21 issues:**

- **pa-r21-001**: end-to-end generic-enum monomorphization for `Option<T>` /
  `Result<T,E>` where `T` is any struct or primitive. AC: `option_str_extract`
  canary compiles and executes.
- **pa-r21-002**: end-to-end generic-struct monomorphization for `Vec<T>`
  where `T` is any struct or primitive. AC: `vec_str_push_pop` canary.
- **pa-r21-003**: end-to-end generic-struct monomorphization for
  `HashMap<K, V>` (specialised to `K = str | u64 | pointer`, `V = arbitrary`).
  AC: `hashmap_str_str` canary.

**Estimated close:** 3–6 months. #997c is the load-bearing piece; #994/#995
are prerequisites.

## Rank 2 — Runtime execution path for high-level `.pdx`

**Blocks:** every non-`unsafe`-body use of the language. Today, high-level
`.pdx` (structs, generics, methods, closures) *parses and elaborates*, but
the code that actually runs at ELF/PE entry is exclusively the
`unsafe { block: {…asm…} }` bodies. Self-hosting a compiler that is 90%
high-level `.pdx` requires the elaborator emit walker to lower every
construct all the way to encoded bytes, not just to IR.

Two paths:

- **Path (a): interpreter.** Ship a `.pdx` interpreter that walks elaborated
  IR and evaluates it. Fastest to a *demo*, slowest to production. Rejected
  in `design/toolchain/self-hosting-phase5-plan.md §7`; not revisited here.
- **Path (b): full lowering.** Every high-level construct lowers to bytes
  via the emit walker. Aligned with the language's design intent (no VM,
  no interpreter, native from source). **Recommended.**

**Existing lineage:** Every v0.17–v0.20 emit walker issue. `emit_control_flow`,
`emit_enum_match`, `emit_call`, `emit_arith_lambda`, `emit_lambda`,
`emit_closure_cons` are all landed pieces of path (b). What remains:
compound composition (calling a monomorphized method on a heap-allocated
value returned from a closure) end-to-end.

**Proposed v0.21 issues:**

- **pa-r21-004**: emit-walker composition test — three-level construct
  (closure returning `Option<Struct>` matched with payload extraction that
  calls a method). AC: canary produces expected exit code from the
  end-to-end elaborate + emit + link + run cycle.
- **pa-r21-005**: emit-walker performance audit. Lowering high-level `.pdx`
  produces byte-level output today; measure the byte-size and instruction-
  count overhead against hand-written asm. Guide compiler self-host cost.

**Estimated close:** 4–8 months (largely already underway; pa-r21-004 is
the exit criterion, not the starter).

## Rank 3 — File I/O, argv, stdout/stderr syscall surface

**Blocks:** CLI port. paideia-as-diagnostics also needs stderr writing.

**Existing lineage:** `pdx/speculative/file.pdx` (parser-only). No `argv`
grammar exists; no `stdout` primitive exists beyond `pdx/io.pdx`
declarations. paideia-os phase 13 owns the syscall surface these bind to.

**Proposed v0.21 issues:**

- **pa-r21-006**: `.pdx` file I/O primitives — `file_open`, `file_read`,
  `file_write`, `file_close` as effectful `!{IO}` functions. AC: canary
  reads a `.pdx` source from disk and prints its byte length.
- **pa-r21-007**: argv accessor — `fn argv() -> Vec<Str>` or equivalent.

## Rank 4 — Serialisation framework (serde-equivalent)

**Blocks:** paideia-as-diagnostics port (SARIF JSON output).

**Proposed v0.21 issue:**

# 6. Design premise: no interpreter, native from source

This audit's Path (b) recommendation (§4 Rank 2) is a deliberate re-affirmation
of the design intent captured in `design/toolchain/self-hosting-phase5-plan.md
§7`. It is worth restating here because it structures every future v0.21+
choice:

paideia-as **does not ship an interpreter**. Every `.pdx` construct lowers
to native x86_64 bytes. Self-hosting means the elaborator + emit-walker
compose end-to-end for every high-level surface the compiler uses. Where
today they gap out (generic mono, iterator dispatch, closure with captures
that outlive scope), those gaps are the roadmap.

**Consequence:** an all-`.pdx` compiler is functionally equivalent to an
all-Rust compiler in emitted bytes. The Wheeler-CTTTDC test at self-host
close (stage-1 Rust produces stage-2 Rust byte-equal to stage-2 .pdx) is
the exit criterion. This is documented in
`design/toolchain/self-hosting-phase5-plan.md §3` as the "Phase 5 closure."

The v0.20 API-freeze (#1026) is what makes this exit test tractable: the
frozen public surface guarantees that a stage-2-.pdx compiler's emit
output matches stage-2-Rust *because both link to the same runtime crate*.
The runtime crate is the fulcrum of the Wheeler argument at the self-host
layer.

# 7. What this audit does NOT include

Explicit non-goals for #1027, to prevent scope creep in the deliverable:

- **paideia-lsp self-hosting.** Deferred to Phase 6+ per
  `self-hosting-phase5-plan.md §5`. Requires async runtime + threading.
- **paideia-pq-sign self-hosting.** Deferred to Phase 6+. Requires
  constant-time crypto ports or FFI shim. Decision belongs to a Phase 6+
  design gate.
- **Alternate targets (RISC-V, ARM64).** Encoder is x86_64-shaped through
  v0.20; port to alternate ISAs is post-v0.21 territory.
- **Formal verification of encoder.** Mechanical property tests + iced-x86
  round-trip continue; machine-checked encoder spec (Coq/Lean) is a
  post-v0.20 concern.
- **paideia-as CLI on paideia-os with all features.** The runtime library
  ships in v0.20; the CLI remains a Rust binary through v0.21+. Full
  self-hosted CLI is the exit of the v0.21–v0.24 arc.
- **`unsafe` block ergonomics.** The `unsafe { effects, capabilities,
  justification, block }` grammar is verbose but load-bearing for stdlib
  work today. Ergonomic sugar is a language-comfort issue, not a
  self-hosting blocker.
