# 0. TL;DR

Ship **one std-side function** on `paideia-pq-sign`:

```rust
pub fn sign_runtime_buffer(
    sk:  &HybridSecretKey,
    buf: &[u8],
) -> HybridSignature;

pub fn verify_runtime_buffer(
    pk:  &HybridPublicKey,
    buf: &[u8],
    sig: &HybridSignature,
) -> bool;
```

Body: `Hybrid::sign(sk, blake3::hash(buf).as_bytes())` (and the mirror verify).
That is the whole business logic — **≈20 LOC** at the core.

**Why so thin.** The crypto substrate (`Hybrid`, ML-DSA-65, Ed25519, BLAKE3)
already exists in `paideia-pq-sign` and is battle-tested through
`sign_pax_hash` and `sign_release_artifact`. The difference this issue lands
is the *composition shape* for **runtime buffers**: raw JIT/WASM-emitted
bytes with no canonicalization step, no PAX header, no filesystem path — just
`&[u8]` → detached signature. That composition doesn't exist today (§2.2
enumerates why the two existing shapes don't fit).

The remaining ~430 LOC (of the ~450 AC target) is the design doc
(`design/security/pq-runtime-signing.md`, ~200 LOC), the JIT-signing example
in `paideia-as-emit/examples/wasm_add_signed.rs` (~150 LOC — extends #1023's
`wasm_add.rs`), and 6 test canaries (~80 LOC).

**Deliver vs. defer.**

- **Deliver in v0.20 (#1025).** The std-side signing surface + example +
  design doc.
- **Defer to Phase 6+ (post-v0.21).** A **no_std verifier** callable *from
  inside* `paideia-as-runtime` itself. This would need either a no_std port
  of `ml-dsa` (currently std-tied via `alloc + std::error::Error` in the
  RustCrypto stack) or an FFI shim to the OS's verifier. Both are Phase 6+
  work per `design/paideia-as/v0.20-issue-1027-self-hosting-audit.md §7`
  which already classifies `paideia-pq-sign` self-hosting as Phase 6+
  ("huge external crypto surface").

# 1. Scope

## 1.1 In scope

1. **New module `crates/paideia-pq-sign/src/runtime_buffer.rs`** with:
   - `pub fn sign_runtime_buffer(sk: &HybridSecretKey, buf: &[u8]) -> HybridSignature`
   - `pub fn verify_runtime_buffer(pk: &HybridPublicKey, buf: &[u8], sig: &HybridSignature) -> bool`
   - `pub fn runtime_buffer_digest(buf: &[u8]) -> [u8; 32]` — exposed so callers who need to persist the digest separately (loader manifest, PAX header slot, JIT cache key) can compute it once without re-signing.
   - 6 unit tests (§5).

2. **Re-exports on `crates/paideia-pq-sign/src/lib.rs`:**

   ```rust
   pub use runtime_buffer::{
       runtime_buffer_digest, sign_runtime_buffer, verify_runtime_buffer,
   };
   ```

3. **New example `crates/paideia-as-emit/examples/wasm_add_signed.rs`** that:
   - Extends the `wasm_add.rs` example from #1023 by adding a sign + verify
     step over the emitted `CodeBuffer.bytes`.
   - Demonstrates the composition pattern the paideia-os Phase 13 WASM jail
     will use: emit → sign → (later) verify + execute.
   - Runnable via `cargo run -p paideia-as-emit --example wasm_add_signed`.

4. **New design doc `design/security/pq-runtime-signing.md`** covering:
   - What "runtime buffer" means (post-resolve, post-emit, pre-load bytes).
   - Why the runtime *crate* itself doesn't PQ-sign (no_std boundary).
   - The two-crate composition pattern (`paideia-as-emit` produces bytes;
     `paideia-pq-sign` signs bytes).
   - Verifier duties on the loader / paideia-os side.
   - Cross-links to `design/security/pq-trust-root.md`.

## 1.2 Out of scope — owned by later issues or explicitly deferred

- **no_std verifier inside `paideia-as-runtime`.** Deferred (§7.1). Adding a
  PQ dep to the runtime crate breaks its no_std, `x86_64-unknown-none`
  verification bar (`design/paideia-as/v0.20-issue-1019-runtime-crate-v2.md
  §1.3`). Phase 6+.
- **Boot-time verifier / trust root registry.** Owned by paideia-os Phase 13
  (`design/security/pq-trust-root.md`). Not a paideia-as deliverable.
- **HSM-backed signing keys for runtime buffers.** The soft-HSM shim
  (`paideia-pq-sign/src/soft_hsm.rs`) is dev-only per the Phase 2 m7
  deferral and remains so. Hardware HSM is Phase 6+.
- **Signature storage / envelope format.** #1025 returns the detached
  `HybridSignature` (3373 bytes) to the caller. How the caller packages the
  signature (embedded in a JIT manifest, stashed in a `.paideia.sig`-shaped
  side-section, indexed by a loader hash-map) is caller policy. §2.3
  discusses why a runtime-specific envelope belongs in Phase 13, not #1025.
- **Signing over an IR/AST representation rather than encoded bytes.**
  Rejected as §7.2 backtrack: the encoder is the authoritative producer of
  what the CPU will execute, and byte-identical emit (v0.20 exit criterion,
  ongoing) is the property that makes byte-level signing tractable across
  hosts. Signing IR would require a canonicalization pass no one has
  written.
- **ACVP vectors landing.** #525 remains open by design. §3 discusses non-
  blocking status.

# 2. Design premises drawn from the tree at HEAD 0c73777

## 2.1 The crypto substrate already exists

`crates/paideia-pq-sign/src/lib.rs` at 0c73777 re-exports:

- `Hybrid` — Ed25519 ‖ ML-DSA-65 concatenation, AND-verify semantics.
  Hybrid PK 1984 B, SK 64 B, Sig 3373 B (`hybrid.rs:22-31`).
- `sign_pax_hash(sk, &[u8; 32]) -> HybridSignature` — thin wrapper for
  32-byte PAX content hashes (`pax.rs:10-12`).
- `sign_release_artifact(sk, path) -> io::Result<HybridSignature>` — reads
  file, BLAKE3-hashes, hybrid-signs (`release.rs:17-24`).

The FIPS-204 ML-DSA-65 implementation goes through `sign_internal` /
`verify_internal` with a deterministic all-zeros `rnd` per Phase 2 m7-001,
so sign/verify halves agree without carrying an rnd across the wire
(`mldsa.rs:53-70` + `pq-trust-root.md §0` bullet 2). Signature bytes are
therefore deterministic for a given `(sk, message)` pair — a property the
canary in §5 (T-8) asserts explicitly, because reproducible signing is
what makes byte-identical-emit downstream verification tractable.

## 2.2 Neither existing wrapper fits the runtime-buffer shape

| Wrapper                    | Input                                | Why not for JIT buffers                                              |
|----------------------------|--------------------------------------|----------------------------------------------------------------------|
| `sign_pax_hash`            | pre-computed 32-B PAX content hash   | Assumes caller ran `pax_message_to_sign(&header, &table, &contents)` (`paideia-as-emitter-pax/src/sign.rs:16`). JIT buffers have no PAX header, no section table, no canonicalization pass. |
| `sign_release_artifact`    | `&Path` (reads from disk)            | JIT buffers live in-memory, never touch a filesystem, and often exist only for the lifetime of a WASM invocation. `std::fs::read(path)` is the wrong seam. |

`sign_runtime_buffer(&[u8])` fills the gap between them: no PAX
canonicalization, no filesystem detour, just `bytes → BLAKE3 → hybrid sign`.

## 2.3 Why the signature envelope stays outside `paideia-pq-sign`

The AC ("returns a `Signature`") and the design premise agree: `#1025` ships
a detached-signature function. Storage/envelope shape is caller policy:

- **paideia-os Phase 13 WASM jail** will likely maintain a `BTreeMap<CodeId,
  HybridSignature>` in the jail's supervisor state — no on-disk artifact,
  no marshalling.
- **paideia-as self-hosting driver** (v0.21+) will emit a `.paideia.sig`
  section alongside the PAX artifact via the existing PAX two-tier shape,
  reusing the `sign_pax_hash` path — no runtime-buffer envelope needed
  because the driver produces a PAX.
- **Ad-hoc caching JITs** (out-of-tree consumers) may want a manifest file;
  designing one prescriptively locks callers into a shape none of the v0.20
  consumers need.

The right move is to keep #1025 detached-only and let Phase 13 name and own
the envelope. §7.4 tracks this as a named backtrack in case a v0.21 consumer
disagrees.

## 2.4 The runtime crate stays no_std — no PQ dep

`crates/paideia-as-runtime/Cargo.toml` (v0.16.0, 0c73777) at HEAD has no
`paideia-pq-sign` dep and cannot add one:

- `paideia-pq-sign` transitively depends on `reqwest` (RFC 3161 timestamp
  client), `cryptoki`, `yubihsm`, `thiserror`, `serde` — all std.
- The runtime crate's `cargo check --target x86_64-unknown-none -p
  paideia-as-runtime` bar (documented as load-bearing in
  `v0.20-issue-1019-runtime-crate-v2.md §1.3`) fails immediately if any
  such dep is added.
- The v0.20 exit criterion is "runtime crate is linkable into paideia-os
  processes" — that means bare-metal loadable, so std-tie is fatal.

Consequence: the sign function must live in **`paideia-pq-sign`**, not in
`paideia-as-runtime`. Callers compose the two at the std-side driver.
This is the same discipline #1020 established for `emit_instruction`
(runtime owns the type; emit crate owns the std-flavoured error impls).

## 2.5 Post-resolve, post-emit — signing scope is the emitted bytes

The v0.20 pipeline at HEAD:

```
build IR → resolve_symbols(...) → for ins in &insts { emit_instruction(&mut buf, ins.clone()) } → buf.bytes: Vec<u8>
                (#1024, LabelMap/SymbolTable)   (#1020)                                             ↑
                                                                                       this is what we sign
```

Signing must happen **after** `emit_instruction` so no unresolved reloc or
label sneaks through. Signing must happen **before** the buffer is handed to
the loader / trap-handler-registrar so the loader can verify before
execution. Both invariants are natural: emit produces `buf.bytes`, the
loader consumes it, sign fits in between.

The AC's phrase "every artifact produced by `paideia-as-runtime`" reads,
under this seam, as "the `Vec<u8>` produced by emitting instructions
generated by paideia-as-runtime types" — the emit crate is the actual byte
producer; the runtime crate produces the *typed IR* upstream of emit.

## 2.6 BLAKE3 handles arbitrary-size buffers

The `blake3` crate is already a workspace dep (via `paideia-pq-sign` and
`paideia-as-emitter-pax`). BLAKE3's streaming API handles buffers from
zero-length up to gigabytes without measurable overhead per input byte
beyond memcpy speed. A JIT buffer is typically 4KiB–1MiB (a WASM function
body), so hashing is <200µs and dominated by the ML-DSA-65 sign itself
(~800µs on x86_64). No hashing-strategy design work needed.

# 3. #525 (ACVP vectors) is not blocking

Verified against `tests/pq-corpus/ML_DSA_ACVP_STATUS.md` at 0c73777:

- The ML-DSA-65 code path signed via `Hybrid::sign` is the **same** code
  path already validated by `mldsa::tests::mldsa65_kat_deterministic`
  (`crates/paideia-pq-sign/src/mldsa.rs:130-160`). Adding one more wrapper
  (`sign_runtime_buffer`) exercises the identical `sign_internal` /
  `verify_internal` calls with the same deterministic rnd.
- ACVP vectors are gold-standard corpus verification (siggen, sigver,
  keygen — 20+ test groups from NIST). They validate the **algorithm
  implementation**, not the wrapper we're adding.
- If ACVP later reveals a bug in `ml-dsa` upstream, that bug lives *below*
  the `sign_runtime_buffer` wrapper; the wrapper doesn't gain or lose
  correctness from ACVP integration.
- Upstream status: `ml-dsa` v0.1.1 (2026-06-05) still excludes the ACVP
  JSON files from the published crate. `#525` remains OPEN pending upstream
  action; documented as such in
  `tests/pq-corpus/ML_DSA_ACVP_STATUS.md`.

**Prereq gap for #1025: none.** #525 stays open; #1025 does not depend on
its closure.

The one honesty caveat worth surfacing in the design doc: `sign_runtime_buffer`
is *no more validated* than the crypto substrate underneath it. If the
Phase 13 threat model demands ACVP-grade validation before shipping signed
runtime buffers to production, that's a Phase 13 gate on `paideia-pq-sign`
as a whole, not on #1025 specifically. The design doc (§4) names this
explicitly to prevent later readers from thinking #1025 shipped
production-grade PQ signing when it really shipped a wrapper over the same
crypto substrate everyone was already using.

# 4. API signature and shape

## 4.1 The three public functions

Owned by `crates/paideia-pq-sign/src/runtime_buffer.rs`; re-exported via
`crates/paideia-pq-sign/src/lib.rs`.

```rust
/// Sign a runtime-emitted byte buffer with a hybrid keypair.
///
/// This is the signing seam for JIT / dynamic-emit workflows:
///
/// ```text
/// build IR → resolve_symbols → emit_instruction → buf.bytes
///                                                    │
///                                       sign_runtime_buffer(&sk, &buf.bytes)
///                                                    │
///                                       hand HybridSignature to loader
/// ```
///
/// # Deterministic
///
/// Signature is deterministic for a given `(sk, buf)` pair (backed by the
/// deterministic-rnd discipline in `mldsa::sign` — see `pq-trust-root.md §0`).
/// Reproducible signing is what makes byte-identical-emit tractable across
/// hosts.
///
/// # Stability
///
/// Part of the paideia-as v0.20 stable public API on the pq-sign side.
/// Signature changes require a major-version bump.
///
/// # See also
///
/// - [`verify_runtime_buffer`] — mirror on the verify side.
/// - [`runtime_buffer_digest`] — the intermediate BLAKE3 hash, exposed so
///   callers can persist / index by digest without re-signing.
/// - [`sign_pax_hash`] — for **PAX artifacts** where the canonical content
///   hash was already computed by the emitter pipeline. Use that, not
///   `sign_runtime_buffer`, whenever a PAX header exists.
pub fn sign_runtime_buffer(
    sk:  &HybridSecretKey,
    buf: &[u8],
) -> HybridSignature {
    let digest = runtime_buffer_digest(buf);
    Hybrid::sign(sk, &digest)
}

/// Verify a hybrid signature over a runtime-emitted byte buffer.
///
/// AND-verify semantics (both Ed25519 and ML-DSA-65 halves must verify).
/// Returns `false` on any tamper: buffer bytes changed, signature bytes
/// changed, wrong public key, or half-verify failure.
pub fn verify_runtime_buffer(
    pk:  &HybridPublicKey,
    buf: &[u8],
    sig: &HybridSignature,
) -> bool {
    let digest = runtime_buffer_digest(buf);
    Hybrid::verify(pk, &digest, sig)
}

/// Compute the BLAKE3 digest of a runtime buffer without signing it.
///
/// Exposed so callers who need to persist the digest separately (loader
/// manifest, JIT cache key, PAX header slot) can compute it once instead
/// of BLAKE3-hashing the buffer twice.
pub fn runtime_buffer_digest(buf: &[u8]) -> [u8; 32] {
    blake3::hash(buf).into()
}
```

## 4.2 What `#[non_exhaustive]` and stability mean here

Neither of the two signing functions is a match candidate (they return a
struct / bool), so the `#[non_exhaustive]` discipline that
`ResolveError` / `EmitError` need does not apply here.

Adding a *third* signing function later (e.g., `sign_runtime_buffer_with_context(sk, buf, ctx)`) is additive and safe under the existing crate-level stability contract. Not in v0.20 scope.

## 4.3 No `Result` — the failure modes are compile-time or vacuous

- `sign_runtime_buffer` cannot fail at runtime: BLAKE3 hashing over `&[u8]`
  is infallible, and `Hybrid::sign` is infallible given a valid
  `HybridSecretKey` (validity is proven at the type level — the struct can
  only be constructed via `keygen` or `from_bytes`, the latter validates
  lengths).
- `verify_runtime_buffer` returns `bool` per the shape of `Hybrid::verify`;
  the AND-verify convention already establishes that any failure collapses
  to `false`.

Neither function needs `Result`. `sign_pax_hash` sets the precedent.

# 7. Backtrack candidates

## 7.1 no_std verifier in `paideia-as-runtime` itself

**Deferred to Phase 6+.** Would require:

- A no_std port of `ml-dsa` (currently `alloc + std::error::Error`-tied via
  the RustCrypto stack).
- A no_std port of `ed25519-dalek` (already almost no_std; the `std`
  feature is dev-side).
- A no_std BLAKE3 (already available; `blake3 = { workspace = true }` with
  `default-features = false, features = ["no-std"]` — small tweak).
- Threading the verifier through the runtime crate as a **pure verify**
  path (never sign, keys never present at runtime).

Rejected for v0.20 because:

- Consumer of a runtime-local verifier is the paideia-os loader / bootstrap
  — which is Phase 13 (post-v0.20). No v0.20 caller.
- Porting ml-dsa to no_std is speculative work (upstream may not accept
  the port; forking the crate is a 30k-LOC maintenance burden per
  `self-hosting-audit.md §3` classification of paideia-pq-sign as Phase
  6+).
- The current design (std-side sign, std-side verify at the loader) is
  correct for every v0.20 through paideia-os-Phase-13 use case.

Reversal cost if adopted later: additive. Runtime-side verifier and std-
side signer coexist; caller picks the right one.

## 7.2 Sign the pre-emit `Vec<Instruction>` (IR) instead of the post-emit `Vec<u8>` (bytes)

**Rejected.** IR-level signing appeals aesthetically (the IR is
higher-level than the bytes and would let a verifier reconstruct the
emitted bytes on the verify side, making the emission itself a proof
object). Rejected because:

- No canonicalization pass on `Vec<Instruction>` exists — `EncodingHint`,
  `byte_offset_in_text`, `emission_order` all carry cache state that would
  fluctuate across build hosts. Signing them would rot signatures across
  every rebuild.
- The v0.20 byte-identical-emit exit criterion, when it lands, gives us
  the property that IR-level signing would offer *and* keeps the signature
  narrowly bound to what the CPU will actually execute. Bytes are the
  correct signing target.
- IR-level signing would need to be re-designed each time we change the
  IR — a costly stability commitment.

Reversal cost if adopted later: prohibitive after v0.21 self-hosting starts
consuming byte-level signatures. Track as a v0.21-plus decision if it
becomes attractive.

## 7.3 Bake a signature-envelope struct into #1025

Rejected per §2.3. Envelope format is caller policy; prescribing one
would force paideia-os Phase 13 (WASM jail) and paideia-as self-hosting
(v0.21) to convert on/off a shape neither of them wants. Detached
`HybridSignature` is the least-committal shape.

Reversal cost if adopted later: additive — the detached function stays,
the envelope helper composes on top. Track only if a v0.21 consumer files
the request.

## 7.4 Reuse `sign_pax_hash` by having the caller BLAKE3-hash the buffer first

Superficially attractive: "just call `sign_pax_hash(sk,
&blake3::hash(buf).into())`, no new function needed." Rejected because:

- The function name lies. `sign_pax_hash` is documented as signing "a
  32-byte PAX content hash — the m4-007 BLAKE3 over header + section
  table + content." Passing a raw-buffer BLAKE3 there miscommunicates
  intent to readers and tools (search-for-caller, grep-audit).
- The composition step (BLAKE3 the buffer) is exactly what
  `sign_runtime_buffer` bundles. Fanning that step out to every caller
  invites callers to skip it (signing the raw buffer — which would fail
  because `Hybrid::sign` takes an arbitrary message but our discipline
  is "sign a 32-byte digest"), or to use different hash algorithms
  inadvertently.
- The wrapper is 3 lines. Not shipping it is worse ergonomics with no
  substrate savings.

## 7.5 Ship `sign_runtime_buffer` in `paideia-as-emit`, not `paideia-pq-sign`

Rejected. Two reasons:

- `paideia-as-emit`'s only dep in `[dependencies]` is
  `paideia-as-encoder + paideia-as-runtime`. Adding `paideia-pq-sign` to
  `[dependencies]` gives every downstream consumer of `paideia-as-emit`
  (JIT, WASM host) a transitive dep on `reqwest`, `cryptoki`, `yubihsm`
  — an unforced std-fanout that some consumers (embedded JIT) won't
  want.
- The AC ("`paideia-pq-sign` gains a `sign_runtime_buffer(&[u8]) ->
  Signature` entry point") is explicit. Deviating from the AC costs
  future-reader friction.

The example (`wasm_add_signed.rs`) *does* live in `paideia-as-emit`
because that's where the WASM example that composes emit + sign belongs
— the example's `dev-dependencies` (not `[dependencies]`) gain
`paideia-pq-sign`, which is the correct fanout.

## 7.6 Also ship `attest` — bind sig to TPM PCR (per pq-trust-root §8.4 "v0.18 full PQ integration")

# 10. Cross-repo (paideia-os) impact

- **paideia-os Phase 13 (PQ trust root).** The concrete Phase-13 consumer.
  On the sign side: WASM jail composes `emit_instruction` +
  `sign_runtime_buffer` at its supervisor's std boundary before publishing
  a signed code page into the jail's execution namespace. On the verify
  side: the same jail (or the OS's loader) calls `verify_runtime_buffer`
  before granting the code page executable permission. Both call sites
  use the exact public API this issue lands.

- **paideia-os R14B autonomous loop.** No consumer today. R14B builds
  static PAX artifacts through the existing `sign_pax_hash` path;
  runtime-buffer signing has no R14B customer. Zero risk of R14B
  regression.

- **paideia-as v0.21+ self-hosting driver.** When the `.pdx` module
  loader lands (Phase 14 per the tactical roadmap), it may want to
  sign JIT-emitted code before executing it inside a `.pdx`-hosted
  process. That path uses the same API. The loader will run in std at
  first (v0.21+), so std-side signing is directly applicable. Later,
  if a *no_std loader* is required (bare-metal .pdx execution), the
  §7.1 deferred no_std verifier gets picked up then.

- **paideia-os `design/security/pq-trust-root.md`.** The paideia-os-side
  spec doc references paideia-as's crypto substrate. #1025 does not
  invalidate any of pq-trust-root.md; it adds a new wrapper that
  pq-trust-root.md's §2 "PAX two-tier signature storage" section does
  *not* apply to (runtime buffers are detached, not two-tier). The new
  `design/security/pq-runtime-signing.md` fills that gap.
