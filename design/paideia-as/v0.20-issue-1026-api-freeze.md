# 1. Scope

Land the v0.20 **API-freeze test suite** for the two crates whose surface is
promised to downstream consumers of paideia-as:

- `paideia-as-runtime` — the no_std IR-plus-resolver library (`Instruction`,
  `Mnemonic`, `Operand`, `IrNodeId`, `SymbolTable`, `LabelMap`, `ResolvePolicy`,
  `ResolveError`, `resolve_symbols`).
- `paideia-as-emit` — the std-flavoured emit wrapper (`CodeBuffer`, `EmitError`,
  `emit_instruction`) that re-exports the runtime surface for one-import
  consumption.

The freeze is a **checked-in signature snapshot** plus a small compile-fence
canary suite. Any diff to the snapshot files surfaces in PR review; only
additions to `#[non_exhaustive]` enums / structs are compatible under our
stability convention. Any other diff is breaking and requires an explicit,
reviewer-approved snapshot update.

**In scope:**

- One new integration test file per frozen crate:
  - `crates/paideia-as-runtime/tests/api_freeze.rs`
  - `crates/paideia-as-emit/tests/api_freeze.rs`
- Two auto-generated snapshot files (insta output) under each crate's
  `tests/snapshots/` directory.
- One `syn = "2"` dev-dependency added to each of the two crates for the
  extractor. (`insta` is already a workspace dep at `Cargo.toml:108`.)
- Backward-compat compile fences: five witness tests that lock signature
  identities the freeze must guarantee to consumers (§5).

**Out of scope (owned by later issues or ruled out by the AC):**

- Freezing internal crates (`paideia-as-ir`, `paideia-as-elaborator`,
  `paideia-as-encoder`). These have no stable-surface commitment; churn is
  expected through v0.21. §7.3 documents why the ir re-export chain does not
  need its own snapshot.
- `cargo-public-api` / `cargo-semver-checks` external tooling. §7.1 names them
  as the fallback if reviewer objects to a syn-based approach.
- Automatic semver-bump inference. The freeze surfaces a *diff*; a human
  reviewer classifies it as additive-compat vs breaking and pairs it with a
  version bump per `design/toolchain/paideia-as-version-discipline.md`.
- ABI freeze (layout, calling convention). Only the source-level Rust API is
  frozen; ABI is not a stability commitment through v0.20.

# 2. Design premises drawn from the tree at HEAD 3c8df4e

## 2.1 The two crates that own stable API today

At HEAD 3c8df4e the source-level public surface is:

**`paideia-as-runtime/src/lib.rs`** (24 LOC — the manifest):
```rust
#![no_std]
extern crate alloc;

pub mod instruction;
pub mod label_map;
pub mod node_id;
pub mod resolve;
pub mod symbol_table;

pub use instruction::{
    Cond, CpuFeature, EncodingHint, InstrMode, Instruction, IntWidth, Mnemonic,
    Operand, RegId, Scale, SegPrefix, SegReg,
};
pub use label_map::{DuplicateLabel, LabelMap};
pub use node_id::IrNodeId;
pub use resolve::{resolve_symbols, ResolveError, ResolvePolicy};
pub use symbol_table::SymbolTable;
```

**`paideia-as-emit/src/lib.rs`** (181 LOC — the wrapper):
```rust
pub use paideia_as_encoder::CodeBuffer;
pub use paideia_as_runtime::Instruction;
pub use paideia_as_runtime::Mnemonic;
pub use paideia_as_runtime::Operand;
pub use paideia_as_runtime::{
    resolve_symbols, DuplicateLabel, LabelMap, ResolveError, ResolvePolicy, SymbolTable,
};

#[non_exhaustive]
pub enum EmitError { … }              // 5 variants at HEAD

pub fn emit_instruction(
    buf: &mut CodeBuffer,
    ins: Instruction,
) -> Result<(), EmitError>;
```

The emit crate's surface is a strict *superset* of the runtime crate's exported
identifiers minus a few internals (e.g., `IrNodeId`, `Cond`, `CpuFeature`,
`EncodingHint`, `InstrMode`, `IntWidth`, `RegId`, `Scale`, `SegPrefix`, `SegReg`
are runtime-only — not re-exported by emit). Freezing both crates independently
is required because:

- Runtime consumers building for `x86_64-unknown-none` never pull emit. If we
  froze only emit, no_std embedded consumers would have no visible stability
  guarantee.
- Emit re-exports overlap runtime symbols; freezing both catches the day
  someone accidentally deletes a re-export line while the underlying runtime
  item is still there.

## 2.2 What "public API" means for this test

The frozen contract is Rust source-level:

- The name of every `pub` item reachable from `crates/<name>/src/lib.rs`.
- Its kind (fn / struct / enum / trait / type-alias / const / module).
- Its shape (function signature, struct fields with types, enum variants with
  payloads).
- The stability-relevant attributes it carries (`#[non_exhaustive]`,
  `#[derive(...)]`, `#[must_use]`, `#[repr(...)]`, `#[deprecated]`).

The frozen contract explicitly does **not** cover:

- Doc comments (`///`, `//!`, `#[doc = "..."]`) — churn is normal and expected.
- Function bodies — only signatures matter.
- `impl` blocks for trait bounds that live in third-party crates (we snapshot
  the paideia-side declaration, not the resolved bound).
- Private (`pub(crate)`, `pub(super)`) items.
- Item ordering in source (the extractor sorts by kind then name, so
  reordering is a no-op).

## 2.3 `#[non_exhaustive]` — the one stability discipline the freeze must express

Three of our four public error/policy enums are `#[non_exhaustive]` today:

- `paideia_as_emit::EmitError` — 5 variants, `#[non_exhaustive]`.
- `paideia_as_runtime::ResolveError` — 4 variants, `#[non_exhaustive]`.
- `paideia_as_runtime::ResolvePolicy` — 1 variant (`AbsoluteImm64`),
  `#[non_exhaustive]`.

`Mnemonic`, `Operand`, `Cond`, `CpuFeature`, `IntWidth`, `SegReg`, `SegPrefix`
are **not** `#[non_exhaustive]` today. This is deliberate: the roadmap's
additive-only-through-v0.20 discipline (`design/roadmap/paideia-as-tactical-issues.md`
issue-1026 body) rests on `Mnemonic` and `Operand` growing by variant addition,
and downstream `paideia-as-ir`/elaborator match arms are exhaustive against
those. Adding a variant to `Mnemonic` today would break the match arms in the
encoder — those arms live one crate over and are considered internal, so this
is a coordinated cross-crate change, not a public API surprise.

**Consequence for the freeze**: A `Mnemonic`-variant addition is a
snapshot-changing diff (adds a line). Reviewer classifies it as a coordinated
internal change and approves the diff. A `ResolveError`-variant addition is
similarly a snapshot-changing diff, but the reviewer's approval criterion is
weaker — external consumers using wildcard match arms are safe. Both cases are
"the reviewer sees the diff and decides"; the freeze is a diff-forcing
mechanism, not a policy engine.

## 2.4 The consumer-code witness comes from #1023

`crates/paideia-as-emit/examples/wasm_add.rs` (267 LOC, landed by #1023) opens
with:

```rust
use paideia_as_emit::{emit_instruction, CodeBuffer, EmitError};
use paideia_as_runtime::{Instruction, InstrMode, Mnemonic, Operand, RegId, Scale};
```

Every subsequent line in the example resolves to those imports. **If the freeze
holds, the example continues to compile with unchanged sources.** This is the
strongest consumer-canary we have — a real downstream construction, not
synthesized. §5.4 promotes it into a compile-fence witness inside
`crates/paideia-as-emit/tests/api_freeze.rs`.

## 2.5 insta is already a workspace dep; syn is not

`Cargo.toml:108` pins `insta = "1"` at workspace level, and `paideia-as-parser`,
`paideia-as-diagnostics` already use it for AST + diagnostic snapshots. Adopting
insta again for the API freeze is idiomatic — reviewers know the toolchain.

`syn` is not in the workspace at HEAD. Adding it as a `[dev-dependencies]` on
runtime + emit is a moderate ask. Alternatives:

- Hand-rolled parsing via `regex`: fragile, tracked as §7.2 rejected fallback.
- Rustdoc-JSON via `public-api` crate: nightly-only, tracked as §7.1 fallback.
- **syn = "2" with `full`, `parsing`, `extra-traits` features**: the industry
  default for exactly this use case. Compile cost is amortized over test-only
  builds; runtime crate ships nothing to production consumers from this dep.

The chosen mechanism is **syn-parsed items → canonical text render →
insta snapshot**.

# 3. Snapshot mechanism — chosen shape

## 3.1 One extractor, two callers

Each of `paideia-as-runtime/tests/api_freeze.rs` and
`paideia-as-emit/tests/api_freeze.rs` includes a ~50-LOC extractor helper.
Duplication is deliberate — the alternative (a workspace-central helper crate)
adds a boundary that pays back only after we freeze a third crate, which is
not on the r20/r21 roadmap. §7.4 flags "extract into a helper crate" as the
backtrack for v0.21+.

The extractor:

1. Reads `crates/<name>/src/lib.rs` off-disk relative to `CARGO_MANIFEST_DIR`.
2. Parses via `syn::parse_file`.
3. Walks the module tree following `pub mod` declarations to sibling
   `.rs` files (no descent into `pub(crate)` / private modules).
4. Collects every `pub` item into a normalized `Vec<RenderedItem>`.
5. Sorts by `(kind, name)` for stability.
6. Emits one signature line per item plus multi-line renders for enums and
   structs.
7. Joins the whole thing into a single `String` and hands it to
   `insta::assert_snapshot!(...)`.

## 3.2 What the canonical render looks like

An example of one rendered enum in the snapshot file — this is the exact text
that would appear in `tests/snapshots/api_freeze__emit_surface.snap` after
approval:

```text
enum EmitError
  attrs:
    #[non_exhaustive]
    #[derive(Clone, Debug, PartialEq, Eq)]
  variants:
    InvalidOperand
    OperandCount { mnemonic: Mnemonic, expected: usize, got: usize }
    OperandShape { mnemonic: Mnemonic }
    Unsupported
    UnresolvedRelocation
```

An example of one rendered function:

```text
fn emit_instruction
  attrs: (none)
  sig: (buf: &mut CodeBuffer, ins: Instruction) -> Result<(), EmitError>
```

And a re-export:

```text
pub use paideia_as_runtime::Instruction
```

Rules for canonical rendering (all in the extractor):

- **Sort variants alphabetically** within an enum. This prevents "I added
  variant at the bottom vs middle" from causing spurious churn.
- **Sort fields alphabetically** within a struct or struct-variant. Same
  rationale.
- **Sort `#[derive(...)]` list alphabetically.** Rust doesn't care about
  derive order; the snapshot shouldn't churn on cosmetic reordering.
- **Drop `#[doc = "..."]`, `///`, `//!`**. Docs are not part of the freeze.
- **Keep `#[non_exhaustive]`, `#[repr(...)]`, `#[must_use]`, `#[deprecated]`**.
  These are semantically load-bearing.
- **Type spellings preserved verbatim** for function signatures and struct
  fields (via `quote!` or `syn::spanned::Spanned + prettyplease`). This is the
  one place where "canonical" is intentionally weak — if we `smart-format`
  types, we lose the ability to detect a change from `usize` to `u32` in a
  field.

## 3.3 What breaks the snapshot

| Change                                                                      | Diff shape                          | Reviewer verdict                 |
|-----------------------------------------------------------------------------|-------------------------------------|----------------------------------|
| Add a `pub fn`                                                              | +1 fn block                         | Compatible (additive).           |
| Add a variant to a `#[non_exhaustive]` enum                                 | +1 line in variants list            | Compatible (per convention).     |
| Add a field to a `#[non_exhaustive]` struct                                 | +1 line in fields list              | Compatible.                      |
| Remove any pub item                                                         | −1 item block                       | **Breaking**.                    |
| Change a function signature                                                 | line changes in `sig:`              | **Breaking**.                    |
| Rename a variant / field / function                                         | Rename appears as −line + +line     | **Breaking** (silent-rename cover). |
| Drop `#[non_exhaustive]`                                                    | Attribute disappears from `attrs:`  | **Breaking** (widens matching).  |
| Add `#[non_exhaustive]` to an existing enum                                 | Attribute appears in `attrs:`       | Compatible (narrows matching).   |
| Reorder variants without adding/removing                                    | No diff (extractor sorts)           | No action needed.                |
| Edit a doc comment                                                          | No diff (extractor strips)          | No action needed.                |

The reviewer's decision protocol becomes: "Every diff to `api_freeze__*.snap`
is a public-API change. Classify per the table above, pair with a
CHANGELOG entry, approve or reject."

## 3.4 Skipping into `use paideia_as_runtime::...` re-exports

The emit crate re-exports several runtime types. When the extractor hits
`pub use paideia_as_runtime::Instruction;` in `paideia-as-emit/src/lib.rs`, it
does **not** descend into runtime's source — it emits one line:

```
pub use paideia_as_runtime::Instruction
```

That single line is the freeze contract for the emit crate: "we promise that
`paideia_as_emit::Instruction` is exactly `paideia_as_runtime::Instruction`
(a nominal re-export)." The runtime snapshot separately locks the shape of
`paideia_as_runtime::Instruction`, so composed the two snapshots freeze the
whole surface.

This design decision matters: if we descended into re-exports on the emit
side, a change to `Mnemonic`'s variants (which lives in runtime) would produce
diffs in *both* snapshot files. That's noise. Keeping the emit snapshot re-
export-shallow means one canonical location per identity.

# 4. Extractor pseudocode

Duplicated (~50 LOC each) in both test files:

```rust
use std::path::Path;
use syn::{Item, ItemEnum, ItemFn, ItemStruct, ItemUse, Visibility};

/// Read `lib.rs` + every pub-reachable sibling module, return sorted
/// canonical snapshot text.
fn extract_surface(crate_src_dir: &Path) -> String {
    let mut items: Vec<RenderedItem> = Vec::new();
    walk_module(crate_src_dir, "lib.rs", &mut items);
    items.sort_by_key(|r| (r.kind, r.name.clone()));
    items.iter().map(|r| r.render()).collect::<Vec<_>>().join("\n\n")
}

fn walk_module(dir: &Path, file_name: &str, out: &mut Vec<RenderedItem>) {
    let src = std::fs::read_to_string(dir.join(file_name)).unwrap();
    let file: syn::File = syn::parse_str(&src).unwrap();
    for item in &file.items {
        match item {
            Item::Mod(m) if is_pub(&m.vis) && m.content.is_none() => {
                // pub mod foo; → descend into foo.rs
                walk_module(dir, &format!("{}.rs", m.ident), out);
            }
            Item::Enum(e)   if is_pub(&e.vis) => out.push(render_enum(e)),
            Item::Struct(s) if is_pub(&s.vis) => out.push(render_struct(s)),
            Item::Fn(f)     if is_pub(&f.vis) => out.push(render_fn(f)),
            Item::Trait(t)  if is_pub(&t.vis) => out.push(render_trait(t)),
            Item::Type(t)   if is_pub(&t.vis) => out.push(render_type_alias(t)),
            Item::Const(c)  if is_pub(&c.vis) => out.push(render_const(c)),
            Item::Use(u)    if is_pub(&u.vis) => out.push(render_use(u)),
            _ => {}
        }
    }
}

fn render_enum(e: &ItemEnum) -> RenderedItem {
    let attrs = collect_stability_attrs(&e.attrs);   // strips docs
    let mut variants: Vec<String> = e.variants
        .iter()
        .map(|v| format!("{}{}", v.ident, render_variant_fields(&v.fields)))
        .collect();
    variants.sort();
    let body = format!(
        "enum {}\n  attrs:\n{}\n  variants:\n{}",
        e.ident,
        indent(&attrs, "    "),
        indent(&variants.join("\n"), "    ")
    );
    RenderedItem { kind: Kind::Enum, name: e.ident.to_string(), body }
}

// … render_struct, render_fn, render_trait, render_type_alias, render_const,
//   render_use analogous. Each ~5–10 LOC. …

fn is_pub(v: &Visibility) -> bool { matches!(v, Visibility::Public(_)) }
```

The extractor is intentionally simple: no rustdoc, no name resolution, no
type inference. It sees exactly what a human reader sees in the source file,
minus comments and reorderings. That simplicity is the freeze's whole worth
— any drift the extractor cannot detect is a hole; the design surface it
covers is precisely the surface the AC promises to lock.

### 4.1 Handling of macro-expanded items

None of the two frozen crates use macros to define public items today. If a
future issue introduces (say) `impl_error_enum!(EmitError, ...)`, the
extractor would see an `Item::Macro` and skip. That is a **failure mode we
accept**: macro-expanded public items would need the extractor to be extended
(the syn-level solution is to run `cargo expand` before extraction, which is
tracked as §7.5). For v0.20, no macro-emitted public items exist and the
freeze is complete over the actual surface.

## 5.3 Non-exhaustive discipline — compile-fence for the `_ =>` covenant

Adding a variant to a `#[non_exhaustive]` enum is only compatible if every
external match arm is either exhaustive-over-current-variants **plus a
wildcard**, or is a bare wildcard. The compile fence proves that our
own downstream code (the emit example, the runtime resolver internals) obeys
this discipline. If the freeze holds and someone silently deletes
`#[non_exhaustive]`, adding this test's wildcard arm would trigger
`unreachable_patterns` (dead-arm warning), which the compile-fence turns into
a hard error via `#![deny(unreachable_patterns)]`.

```rust
// crates/paideia-as-emit/tests/api_freeze.rs
#[deny(unreachable_patterns)]
#[test]
fn emit_error_is_non_exhaustive_wildcard_arm_reachable() {
    let e = EmitError::InvalidOperand;
    let msg = match e {
        EmitError::OperandCount { .. } => "count",
        EmitError::OperandShape { .. } => "shape",
        EmitError::InvalidOperand      => "invalid",
        EmitError::Unsupported         => "unsupported",
        EmitError::UnresolvedRelocation => "reloc",
        _ => "future",   // ← proves EmitError is #[non_exhaustive]
    };
    assert_eq!(msg, "invalid");
}

#[deny(unreachable_patterns)]
#[test]
fn resolve_error_is_non_exhaustive_wildcard_arm_reachable() {
    let e = ResolveError::UnknownSymbol {
        instr_index: 0, operand_index: 0, name: String::from("_"),
    };
    let _ = match e {
        ResolveError::UnknownSymbol { .. } => 0,
        ResolveError::UnknownLabel { .. } => 1,
        ResolveError::OutOfRange { .. } => 2,
        ResolveError::NotEncodableForMnemonic { .. } => 3,
        _ => 4,  // ← proves ResolveError is #[non_exhaustive]
    };
}
```

Under Rust's rules, `#[non_exhaustive]` on an enum *from another crate* forces
the wildcard arm to be considered reachable. If someone removed
`#[non_exhaustive]`, the compiler would flag the `_ =>` arm as unreachable
(because all variants are named above it), and `#[deny(unreachable_patterns)]`
would fail the build. The .snap diff would surface the removed attribute
independently, but this compile fence is the belt-and-braces.

Two canaries, one per crate boundary. `ResolvePolicy` has just one variant
today, so a wildcard against it is trivially reachable; we skip its own
canary and rely on the snapshot.

## 5.4 The #1023 consumer witness — real downstream code compiles

The strongest possible backward-compat canary is a fragment of real downstream
code that must continue to compile. `examples/wasm_add.rs` is that fragment.
We import the exact identifiers it does, and write one witness function that
mirrors its use of `emit_instruction`:

```rust
// crates/paideia-as-emit/tests/api_freeze.rs
use paideia_as_emit::{emit_instruction, CodeBuffer, EmitError};
use paideia_as_runtime::{Instruction, InstrMode, Mnemonic, Operand, RegId, Scale};

/// Compile-only witness that mirrors the #1023 wasm_add example's import + call
/// surface. If any of these identifiers moves or changes signature, this test
/// fails to compile — which the workspace test discipline surfaces as a build
/// failure, not a test failure. Both are review-visible.
#[test]
fn wasm_example_consumer_import_shape_compiles() {
    fn _witness_emit(buf: &mut CodeBuffer, ins: Instruction) -> Result<(), EmitError> {
        emit_instruction(buf, ins)
    }
    fn _witness_instr_ctor() -> Instruction {
        Instruction {
            mnemonic: Mnemonic::Nop,
            operands: smallvec::SmallVec::new(),
            encoding_hint: None,
            byte_offset_in_text: None,
            mode: InstrMode::Mode64,
            emission_order: 0,
        }
    }
    // Force MemSib/Scale usage so those types are frozen too.
    let _ = Operand::MemSib {
        base: RegId(4),
        index: None,
        scale: Scale::X1,
        disp: 16,
    };
}
```

Rationale for this shape: the actual example lives at
`crates/paideia-as-emit/examples/wasm_add.rs` and is built as an example, not
as a test. `cargo test -p paideia-as-emit` does not compile examples. If the
freeze breaks and the example stops compiling, only `cargo build --examples`
or `cargo test --examples` would surface it — neither is in every developer's
tight loop. By promoting the imports into a witness test, we get the same
compile-fence semantics inside the tight-loop test run.

## 5.5 Runtime type-identity re-affirmation

Existing `crates/paideia-as-runtime/tests/runtime_surface.rs:222-232` already
tests that `paideia_as_runtime::Instruction` and `paideia_as_ir::Instruction`
are the same nominal type. #1026 does not duplicate; the design note flags
this as **the freeze's third canary layer** and links it here for
completeness. If it fails, the ir re-export chain broke — which is not
technically a public-API breakage on the paideia-as-runtime/emit surface
(paideia-as-ir is internal), but it is the invariant that lets the shared
`Instruction` value flow across crate boundaries in the workspace.

## 5.6 no_std reachability — the runtime crate stays linkable bare-metal

Not a new test — the #1019 v2 verification recipe already asserts
`cargo check --target x86_64-unknown-none -p paideia-as-runtime` passes. The
freeze harness inherits that check as a prerequisite via the pre-push gate.
If a future PR both (a) updates the runtime snapshot with a compatible-looking
diff and (b) accidentally introduces a `std::` import into runtime, the
no_std check catches it before the freeze test does. This is orthogonal
defence-in-depth; #1026 needs no new work here.

# 7. Backtrack candidates

Named-and-owned, each with a specific trigger.

## 7.1 Reviewer objects to a syn-based extractor; wants `cargo-public-api`

**Trigger.** Reviewer argues that a hand-rolled extractor duplicates what
`cargo-public-api` (which uses rustdoc JSON) does better, and that vendor
tooling reduces our maintenance surface.

**Fallback.** Add `cargo-public-api` as an external binary requirement to the
pre-push hook. Replace the syn extractor with a `Command::new("cargo")
.args(["public-api", "-p", "paideia-as-runtime"])` call. Feed stdout to
insta.

**Cost.** Loss of `x86_64-unknown-none` self-containment (public-api needs
rustdoc JSON, which requires a nightly-compatible toolchain), addition of an
external binary dependency to the pre-push protocol, and stronger vendor lock
than the syn approach. Rejected as default because:
- Nightly requirement conflicts with the reproducible-build discipline (v0.20
  builds on stable).
- Adding an external binary to the pre-push hook creates a new failure mode
  ("please install cargo-public-api") for contributors.
- The syn approach ships in the paideia-as workspace itself; no new toolchain
  installation is required.

## 7.2 Regex extraction instead of syn

**Trigger.** Reviewer objects to adding a syn dev-dep because it noticeably
increases the test-build cold-compile time.

**Fallback.** Replace the syn walker with a `regex` walker over the source
files, matching lines like `^pub (fn|struct|enum|trait|use|mod|const|type)`.

**Cost.** Fragile. Multi-line declarations (which are the whole point of
freezing signatures) don't parse cleanly with line-based regex. Attribute
tracking requires stateful regex (which is not what regex means). Rejected
except as a last resort.

## 7.3 Freeze `paideia-as-ir` too, not just runtime and emit

**Trigger.** Someone argues that paideia-as-ir is a public interface because
existing call sites in the elaborator/encoder import from it via the
re-export chain.

**Fallback.** Add a third test file `crates/paideia-as-ir/tests/api_freeze.rs`
mirroring the runtime and emit patterns.

**Cost.** Ir is a *workspace-internal* re-export node with no stability
commitment. Freezing it locks the elaborator/encoder's ability to refactor.
Rejected — the design intent is that ir stays fluid through v0.21.
Downstream-of-paideia-as consumers never link ir; they link runtime + emit.

## 7.4 Extract the syn walker into a helper crate

**Trigger.** A third crate (e.g., `paideia-as-linker`) also promises a stable
public surface in v0.21+, and we want to freeze it too.

**Fallback.** Create `tests/api-freeze-tool/` as a new workspace member (a
library crate exporting `pub fn extract_surface(root: &Path) -> String`).
Each frozen crate's test then does `use api_freeze_tool::extract_surface;`.

**Cost.** +40 LOC crate scaffold; deduplicates ~50 LOC × N-1 across the N
frozen crates. Payback threshold: N = 3. Rejected for #1026 (N = 2) but
prepared for.

## 7.5 Macro-expanded public items would silently escape the extractor

**Trigger.** A future r21 issue lands `impl_stable_enum!(EmitError, ...)` or
similar. The extractor's `syn::Item::Macro` arm sees only the invocation,
not the expanded pub items.

**Fallback.** Two options:
- Run `cargo expand -p <crate>` before the extractor and feed the expanded
  source into syn. Adds an external tool + longer test runtime.
- Add a lint at the review boundary: "no macros that emit pub items in
  frozen crates." Simpler, keeps the extractor simple.

Prefer the lint. Track as an issue-when-it-happens hazard, not a v0.20
concern.

## 7.6 Snapshot fixture noise from formatting-preserving type rendering

**Trigger.** `prettyplease` (the type rendering helper) reformats a generic
`Result<(), EmitError>` differently after a rustc-toolchain bump, causing
spurious diffs that aren't semantic changes.

**Fallback.** Pin the rendering path to a hand-written `type_to_string`
function inside the extractor that walks `syn::Type` and emits a canonical
form we own. ~20 more LOC in the extractor, but the format never drifts under
us.

**Cost.** +20 LOC. Rejected initially (prettyplease is stable enough for a
few releases), promoted on first observed churn.

## 7.7 Consumer imports drift into `paideia_as_ir::Instruction` instead of `paideia_as_runtime::Instruction`

**Trigger.** A future consumer imports the ir path (still supported through
v0.20 via the re-export chain, per #1019 v2 §2.2) and downstream test #6
(instruction_type_identity_across_crates) doesn't catch a subsequent
divergence because someone re-forks the type.

**Fallback.** Extend the runtime snapshot to include a "reachability of the
ir re-export" section — a compile-fence assertion that the ir path continues
to alias runtime. Already covered by existing `runtime_surface.rs` test #10
in `tests/runtime_surface.rs:220-232`. No new work in #1026.

# 8. Cross-repo (paideia-os) impact

- **paideia-os phase 10 (WASM jail)** will link `paideia-as-emit` as its
  runtime dep. The freeze test suite is the promise it relies on: after v0.20
  ships, phase 10 code that compiles against paideia-as-emit v0.20.0 will
  continue to compile against v0.20.x patch releases. Additive-only variant
  changes to `EmitError` / `ResolveError` are safe because phase 10 will
  wildcard-match those errors.

- **paideia-os phase 14 (self-hosting)** will consume the frozen surface via
  the paideia-as submodule bump. The freeze protects it during the many
  cross-repo submodule bumps expected between v0.20 → v0.21.

- **No submodule bump required for #1026 alone**. This issue lands the freeze
  infrastructure; it doesn't change the surface it freezes. The bump lands
  with the v0.20 release (pa-r20-010).

# 9. Discipline anchors (per issue template)

- **iced-x86 round-trip must stay green through every commit.** #1026 changes
  no encoder output; iced round-trips at runtime + emit level are untouched.
- **Existing paideia-os canary tests must stay green.** #1026 adds tests, does
  not modify source. `crates/paideia-as/tests/*.rs` untouched.
- **Additive-only changes to `Mnemonic`, `Operand`, attribute vocabulary
  through v0.20.** The freeze test *enforces* this discipline for the two
  frozen crates. Any additive change to `Mnemonic` will appear as a
  snapshot diff and require reviewer sign-off — the freeze is the mechanism
  by which the additive-only discipline becomes machine-checkable.
- **Version discipline.** #1026 does not bump `workspace.version` — the r20
  release close does. See `design/toolchain/paideia-as-version-discipline.md`.
- **softarch → workerbee → debugger triangle.** This document is the softarch
  step for #1026. Workerbee implements the extractor + tests. Debugger
  verifies by (a) running the freeze suite, capturing the two `.snap` files;
  (b) intentionally introducing a variant addition to `EmitError` in a
  scratch branch, observing the .snap diff, reverting; (c) intentionally
  removing `#[non_exhaustive]`, observing both the .snap diff *and* the
  compile fence #3 failure, reverting.

# 10. Tractability

**Single-PR, single-workerbee-pass: yes.**

The mechanical work is small — two test files at ~90 LOC and ~110 LOC, plus
the initial snapshot generation which insta handles automatically on first
`cargo test`. Reviewer effort concentrates on:

- The initial `.snap` file contents: reviewer skims to confirm every intended
  public item is present, no accidental private items leaked.
- The workspace-level `syn` dev-dep addition, if we opt for the shared
  workspace pin.

**Estimated workerbee runtime**: 30–45 minutes for extractor + tests, plus
~10 minutes to run the freeze once to generate the snapshots and inspect
them.

**Risk band: low.** The extractor's failure modes (macro-expanded items,
type rendering churn) are named-and-owned in §7.5, §7.6 and have compat
fallbacks. The freeze itself is inert until the surface changes, at which
point the diff is the mechanism.

**Prereq gaps: none.** HEAD 3c8df4e has all four prereq issues (#1019, #1020,
#1023, #1024) landed. The `paideia-as-runtime` + `paideia-as-emit` surface is
stable and ready to be locked.
