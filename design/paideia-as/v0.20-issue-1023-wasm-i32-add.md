# 1. Scope

Deliver a **consumer-story demonstration**: a runnable Rust program that
takes one WASM opcode (`i32.add`, byte `0x6A`) and, using only the paideia-as
public runtime + emit APIs landed in #1019 / #1020 / #1024, produces the
correct x86_64 byte sequence. This is the first end-to-end proof that the
v0.20 SELF-HOST substrate is usable by the paideia-os Phase-10 WASM jail —
before Phase 10 opens a single issue.

## 1.1 In scope

- One example binary at **`crates/paideia-as-emit/examples/wasm_add.rs`** (see
  §2.1 for why the AC's file path — `paideia-as-runtime/examples/wasm_add.rs`
  — is stale and cannot host this code).
- One integration-test file at
  `crates/paideia-as-emit/tests/wasm_add_lowering.rs` with **6 canary tests**
  (3 byte-fixture, 1 iced-x86 round-trip, 1 stream-lowering, 1 error).
- One design document at
  **`tools/paideia-as/design/toolchain/dynamic-emit.md`** (new — the doc AC).
  It captures the "how to consume paideia-as at runtime" recipe: minimal
  imports, three-line emit, symbol-resolution flow. This is the reference
  paideia-os Phase 10 will link into its own design docs, so it lives on
  the paideia-as side (single source of truth for the emit contract).
- Pre-push gate update: add `cargo check -p paideia-as-emit --examples` to
  the existing recipe so example bit-rot is caught in-loop.

## 1.2 Out of scope

- **JIT execution of the emitted bytes** (mmap → PROT_EXEC → transmute →
  call). Genuinely useful, but pulls in `libc` or `region`, adds three
  `#[cfg(target_os = ...)]` branches, and needs `unsafe`. The example
  demonstrates the pattern; runtime JIT belongs to paideia-os Phase 10 which
  will own the executable-page allocation policy. §7.3 backtracks the JIT
  variant, gated on an explicit signal.
- **Real WASM parsing** (`wasmparser` crate, section walk, function-body
  reader). The point of this example is the *lowering* — the operand
  construction pattern that consumers repeat for every WASM opcode. Wrapping
  it in a full parser adds ~200 LOC of scaffolding that does not test any
  paideia-as surface. §7.2 backtracks a `wasmparser`-based fixture if the
  reviewer wants "real" WASM input; the same lowering code is unchanged.
- **Additional WASM opcodes** (`i32.sub`, `i32.mul`, `i32.and`, `i32.or`,
  `i32.xor`). One opcode is enough to prove the pattern; the mnemonic
  translation table is trivially extended (see §3.4). §7.4 backtracks a
  small-catalog demo if pa-r20-007 or Phase 10 asks.
- **Symbol resolution demonstration.** The `i32.add` opcode lowers to a
  self-contained `ADD r64, r64` — zero symbols, zero labels. #1024's
  `resolve_symbols` API is mentioned in the dynamic-emit doc (§4) with a
  three-line snippet showing where it fits, but the example does not
  exercise it. A companion `i32.call_indirect` example (not proposed here)
  would demonstrate resolve_symbols end-to-end and is the right place for
  that fixture.

# 2. Design premises at HEAD 4001f6c

## 2.1 Why the AC's file path is stale — the correct home is `paideia-as-emit`

The issue body says:

> - Example in `crates/paideia-as-runtime/examples/wasm_add.rs`.

This wording predates the #1020 crate-split decision. The v0.20 sequence
executed:

| Issue | Landed at | What moved where |
|-------|-----------|------------------|
| #1019 | 28c3c2c   | Runtime crate extracted; `#![no_std] extern crate alloc;` |
| #1020 | d2803b3   | `emit_instruction` public API — landed in **`paideia-as-emit`** (new crate), not runtime (§7.1 backtrack of #1020 was chosen because the encoder dep is std-flavored) |
| #1024 | 4001f6c   | `resolve_symbols` + `SymbolTable` + `LabelMap` — landed in runtime (no encoder dep needed), re-exported through `paideia-as-emit` |

Runtime's `Cargo.toml` at 4001f6c has:

```toml
[dependencies]
smallvec = { workspace = true }
static_assertions = { workspace = true }

[dev-dependencies]
paideia-as-encoder = { path = "../paideia-as-encoder" }
paideia-as-ir = { path = "../paideia-as-ir" }
iced-x86 = { workspace = true }
```

No `paideia-as-encoder` under `[dependencies]`. An `examples/wasm_add.rs`
under runtime cannot call `emit_instruction` — the symbol lives in
`paideia-as-emit`, which runtime does not depend on and must not depend on
(would create a cycle: encoder ← runtime ← emit, and encoder already
imports runtime types). Examples that need `emit_instruction` therefore
belong in the crate that owns it.

**Decision.** Land the example at
`crates/paideia-as-emit/examples/wasm_add.rs`. Note this in the issue
comment on close so the file-location deviation from AC is documented and
does not later trip a "spec vs. tree" audit.

`paideia-as-emit`'s Cargo.toml at 4001f6c is:

```toml
[dependencies]
paideia-as-encoder = { path = "../paideia-as-encoder" }
paideia-as-runtime = { path = "../paideia-as-runtime" }

[dev-dependencies]
iced-x86 = { workspace = true }
smallvec = { workspace = true }
```

The example needs only what's already there — no new dependencies. Cargo
examples get both `[dependencies]` and `[dev-dependencies]`, so
`iced-x86` and `smallvec` are available in the example scope for
round-trip disassembly and the `smallvec!` macro respectively.

## 2.2 The three imports the example (and every downstream consumer) uses

```rust
use paideia_as_emit::{emit_instruction, CodeBuffer, EmitError};
use paideia_as_runtime::{Instruction, InstrMode, Mnemonic, Operand, RegId};
use smallvec::smallvec;
```

The `Instruction { mnemonic, operands, encoding_hint, byte_offset_in_text,
mode, emission_order }` shape is verbose to build by hand — the existing
test `crates/paideia-as-emit/tests/emit_api.rs:12-25` shows the pattern
consumers must repeat. The example commits to a **tiny helper**:

```rust
fn instr(mnemonic: Mnemonic, operands: &[Operand]) -> Instruction {
    Instruction {
        mnemonic,
        operands: operands.iter().cloned().collect(),
        encoding_hint: None,
        byte_offset_in_text: None,
        mode: InstrMode::Mode64,
        emission_order: 0,
    }
}
```

Deliberately kept as an example-local helper, **not** promoted to a public
runtime constructor. The moment we ship
`paideia_as_runtime::Instruction::new(mnemonic, &[operand])` we've made a
new stability commitment on the constructor shape — which the v0.20 API
freeze (pa-r20-008) would then have to snapshot. Better to leave the field
struct literal as the exposed surface and let each consumer make its own
helper. §7.6 backtracks promotion if pa-r20-008 review asks for a builder.

## 2.3 The i32.add mapping is a one-line table

WASM 1.0 core spec, section 4.4.1, defines `i32.add` (opcode `0x6A`) as
"pop two i32s, push i32 sum". The natural x86_64 lowering under the
WASM-jail calling convention (evaluation stack in registers, `rax` = top,
`rbx` = next, discard `rbx` after) is:

```asm
add rax, rbx      ; ADD RAX, RBX → 48 01 D8
```

**One byte fixture, three bytes emitted.** Anything more complex belongs
to the WASM engine, not this example. §3.4 spells out the mapping table
that admits `i32.sub`, `i32.mul`, `i32.and`, `i32.or`, `i32.xor` as
one-liner extensions — deliberately not shipped, but the mapping is
sketched so a future PR knows exactly what shape to add.

## 2.4 What `emit_instruction` guarantees the example needs to lean on

From `crates/paideia-as-emit/src/lib.rs:138-181`:

- **Rollback on error.** If lowering produces a shape mismatch, the buffer
  is left untouched. The example does not need to snapshot buf.len()
  before each emit — the contract does it.
- **Reloc-refusal pre-flight.** `Operand::SymbolRef` etc. return
  `EmitError::UnresolvedRelocation` before the encoder runs. The example's
  error canary (§4.2 test #6) uses this to prove that a WASM opcode which
  would produce a symbol reference (e.g., `call $host_func`) fails cleanly
  rather than silently emitting a zero-disp `call rel32`.
- **Byte-identical determinism.** Same input → same bytes on every host.
  The v0.20 exit criterion (milestone one-liner: "Deterministic
  byte-identical emit across host platforms") makes tests #1-#3
  byte-exact assertions and not "assert the bytes contain 0x48 somewhere".

## 3.2 What `examples/wasm_add.rs` does when run

`cargo run -p paideia-as-emit --example wasm_add` prints:

```
paideia-as v0.20 dynamic-emit demo — WASM i32.add lowering
===========================================================

Input WASM byte stream (function body of `(func (param i32 i32) (result i32) local.get 0 local.get 1 i32.add)`):
  20 00 20 01 6a 0b

Decoded WASM ops:
  [0] local.get 0    -> mov rax, [rsp+16]        (arg0)
  [1] local.get 1    -> mov rbx, [rsp+24]        (arg1)
  [2] i32.add        -> add rax, rbx
  [3] end            -> ret

Emitted x86_64 bytes (via emit_instruction):
  48 8b 44 24 10                    ; mov rax, [rsp + 0x10]
  48 8b 5c 24 18                    ; mov rbx, [rsp + 0x18]
  48 01 d8                          ; add rax, rbx
  c3                                ; ret

Total: 14 bytes.

Verification: iced-x86 round-trip
  [00] mov rax, qword ptr [rsp+10h]       (5 bytes)
  [05] mov rbx, qword ptr [rsp+18h]       (5 bytes)
  [10] add rax, rbx                       (3 bytes)
  [13] ret                                (1 byte)
  OK — all instructions decoded successfully.
```

The demo shows two layers of the pattern:

1. **The atom** — a single opcode → single instruction lowering
   (`i32.add` → `Mnemonic::Add`). This is the load-bearing demo.
2. **The composition** — chaining `local.get` / `local.get` / `i32.add` /
   `end` into a sequence, calling `emit_instruction` in a loop.
   Consumers will always want to see this because a real WASM engine
   emits basic-block-sized runs, not single opcodes.

## 3.3 Internal structure of the example (~160 LOC)

```rust
// examples/wasm_add.rs — runnable demonstration of WASM → x86_64 lowering
// via paideia-as-emit::emit_instruction.

use paideia_as_emit::{emit_instruction, CodeBuffer, EmitError};
use paideia_as_runtime::{Instruction, InstrMode, Mnemonic, Operand, RegId, Scale};
use smallvec::smallvec;

// --- WASM opcode surface (tiny subset, one shape per opcode we handle) ---

#[derive(Debug, Clone)]
enum WasmOp {
    LocalGet(u32),   // 0x20
    I32Add,          // 0x6A
    End,             // 0x0B
}

fn decode_body(bytes: &[u8]) -> Result<Vec<WasmOp>, &'static str> {
    let mut out = Vec::new();
    let mut i = 0;
    while i < bytes.len() {
        match bytes[i] {
            0x20 => {
                // local.get <leb128 u32>. Single-byte encoding is sufficient
                // for indices < 128; the example never emits indices ≥ 128.
                let idx = bytes.get(i + 1).copied().ok_or("truncated local.get")? as u32;
                out.push(WasmOp::LocalGet(idx));
                i += 2;
            }
            0x6A => { out.push(WasmOp::I32Add); i += 1; }
            0x0B => { out.push(WasmOp::End); i += 1; }
            b => return Err(match b { _ => "unsupported opcode" }),
        }
    }
    Ok(out)
}

// --- Lowering table (WASM → x86_64 Instruction) ---

// Evaluation-stack model: WASM's operand stack is materialized in RAX (top)
// and RBX (second-from-top). local.get pushes onto this stack; i32.add pops
// both and pushes the sum. Realistic engines spill to memory when depth > 2;
// this example handles only depth-2 to keep the code obvious.
//
// Arg layout: caller places arg0 at [rsp + 16], arg1 at [rsp + 24]
// (SysV convention for stack-passed slots after 6 register-passed).
fn lower(op: &WasmOp, stack_depth: usize) -> Vec<Instruction> {
    match op {
        WasmOp::LocalGet(idx) => {
            let dst = if stack_depth == 0 { RegId(0) /* RAX */ }
                      else                { RegId(3) /* RBX */ };
            let disp = 16 + (*idx as i32) * 8;
            vec![instr(Mnemonic::Mov, &[
                Operand::Reg(dst),
                Operand::MemSib { base: RegId(4) /* RSP */, index: None, scale: Scale::X1, disp },
            ])]
        }
        WasmOp::I32Add => {
            vec![instr(Mnemonic::Add, &[
                Operand::Reg(RegId(0)),  // RAX (dst / top-of-stack)
                Operand::Reg(RegId(3)),  // RBX (src / second-of-stack)
            ])]
        }
        WasmOp::End => vec![instr(Mnemonic::Ret, &[])],
    }
}

fn instr(mnemonic: Mnemonic, operands: &[Operand]) -> Instruction { /* §2.2 */ }

// --- Emit driver ---

fn emit_body(ops: &[WasmOp]) -> Result<Vec<u8>, EmitError> {
    let mut buf = CodeBuffer::new();
    let mut depth = 0usize;
    for op in ops {
        for ins in lower(op, depth) {
            emit_instruction(&mut buf, ins)?;
        }
        depth = match op {
            WasmOp::LocalGet(_) => depth + 1,
            WasmOp::I32Add      => depth.saturating_sub(1),
            WasmOp::End         => depth,
        };
    }
    Ok(buf.bytes)
}

// --- Roundtrip verifier (self-check via iced-x86) ---

fn verify_with_iced(bytes: &[u8]) -> Result<Vec<String>, String> { /* ~30 LOC */ }

// --- Main ---

fn main() {
    let function_body: &[u8] = &[0x20, 0x00, 0x20, 0x01, 0x6A, 0x0B];
    println!("paideia-as v0.20 dynamic-emit demo — WASM i32.add lowering");
    println!("===========================================================\n");
    // ... print structured tables per §3.2 output.
}
```

Total: ~150-160 LOC. Every line lands on the pattern the paideia-os Phase-10
jail will replay.

## 3.4 The extension point — what a future opcode looks like

Adding `i32.sub` (opcode `0x6B`) is a one-line change to `decode_body`:

```rust
0x6B => { out.push(WasmOp::I32Sub); i += 1; }
```

and one arm in `lower`:

```rust
WasmOp::I32Sub => vec![instr(Mnemonic::Sub, &[
    Operand::Reg(RegId(0)), Operand::Reg(RegId(3)),
])],
```

This deliberately-small delta is the whole point of the demo: the runtime
API is stable, the lowering table is data, and every new opcode is
touching only the table.

# 4. Test canaries (6)

New file `crates/paideia-as-emit/tests/wasm_add_lowering.rs`. Each test
uses **only** the public `paideia-as-emit` + `paideia-as-runtime` surfaces
— no `paideia-as-encoder` import, matching the AC-lock discipline from
#1020 tests.

The lowering functions (`instr`, `lower`, `decode_body`, `emit_body`) are
duplicated between the example binary and the test file. This is a
Cargo-imposed cost — `examples/` items are not importable by `tests/`
without promoting them to `src/`. §7.5 backtracks moving them into
`paideia-as-emit::wasm_demo` behind a `wasm-demo` Cargo feature if the
duplication drifts (unlikely at ~50 LOC of shared surface).

## 4.1 Byte-fixture canaries (3)

| # | Test name | Input | Expected bytes |
|---|-----------|-------|----------------|
| 1 | `lower_i32_add_produces_add_rax_rbx` | `WasmOp::I32Add` alone, depth=2 fixture | `[0x48, 0x01, 0xD8]` (3 bytes) |
| 2 | `lower_local_get_at_depth_zero_targets_rax` | `WasmOp::LocalGet(0)`, depth=0 | `[0x48, 0x8B, 0x44, 0x24, 0x10]` (5 bytes; MOV RAX, [RSP+0x10]) |
| 3 | `emit_full_body_produces_expected_stream` | Byte stream `[0x20, 0x00, 0x20, 0x01, 0x6A, 0x0B]` → `emit_body` | `[0x48, 0x8B, 0x44, 0x24, 0x10, 0x48, 0x8B, 0x5C, 0x24, 0x18, 0x48, 0x01, 0xD8, 0xC3]` (14 bytes) |

Tests #1 and #2 are unit-level: one WasmOp in, one Instruction out, bytes
asserted. Test #3 is the end-to-end fixture — the whole demo function
body, byte-exact. Together they pin every byte the example prints in
§3.2. A drift in either the encoder or the lowering table breaks these.

## 4.2 Interop and error canaries (3)

| # | Test name | Fixture | Assertion |
|---|-----------|---------|-----------|
| 4 | `emit_full_body_roundtrips_via_iced` | Emit the full body from #3, feed to iced-x86 decoder | Sequence of `IcedMnemonic::{Mov, Mov, Add, Ret}` recovered |
| 5 | `decode_body_rejects_truncated_local_get` | Byte stream `[0x20]` (missing index byte) | `decode_body` returns `Err("truncated local.get")` |
| 6 | `emit_refuses_unsupported_opcode` | Fabricated WasmOp that would produce `Operand::SymbolRef` (see fixture below) | `emit_instruction` returns `Err(EmitError::UnresolvedRelocation)`; buffer unchanged |

Test #4 is the required "iced-x86 round-trip must stay green" discipline
from the issue body. It closes the loop: bytes → decode → mnemonic
sequence, no assertion on operand form beyond mnemonic. That keeps the
test robust against reg-encoding tweaks in the encoder while still
catching mnemonic-level regressions.

Test #6 fixture: not a real WASM opcode — a hand-built `Instruction`
where `Mnemonic::Call` takes an `Operand::SymbolRef { name: "host_call",
addend: 0 }`, showing that if someone tries to lower a future `call
$host_func` naively (without seeding a `SymbolTable` and running
`resolve_symbols` first), the emit refuses cleanly. This is where the
example points readers at #1024's `resolve_symbols` as the correct next
step, without actually exercising it (out of scope per §1.2).

# 5. The `dynamic-emit.md` companion document

## 5.1 Structure (~200 LOC of markdown)

```
# Dynamic emit — the paideia-as runtime API

1. Purpose
   - What this surface exists to serve (JIT, WASM jail, self-hosting).
   - What it deliberately does NOT do (linker, .o file emission, DWARF).

2. The three imports
   (§2.2 of this doc, verbatim.)

3. The three-step recipe
   3.1 Build an Instruction (struct literal).
   3.2 emit_instruction(&mut buf, ins).
   3.3 buf.as_slice() is the encoded byte stream.

4. Handling symbol references — resolve_symbols
   3-line snippet: build a SymbolTable, call resolve_symbols on the
   Instruction slice, then loop emit_instruction. Reference #1024 for
   the full API and its 4 tests.

5. Error taxonomy
   Table of EmitError variants + what a JIT caller does with each.
   (Mirror of §4 of #1020's design doc, condensed.)

6. Determinism guarantees
   The v0.20 exit criterion: byte-identical emit across hosts. What
   that means for a fuzz-testable consumer (property: emit is a pure
   function of Instruction).

7. Worked example — the WASM i32.add lowering
   Cross-reference to `crates/paideia-as-emit/examples/wasm_add.rs`;
   quote the ~15 load-bearing LOC (the lower() match arms and the
   emit loop).

8. Where this cannot help you (yet)
   - No JIT-page allocation policy — call site owns mmap/mprotect.
   - No cross-block relocation (call inter-function jumps within one
     emit slice must use LabelRef + resolve_symbols).
   - No REX-optimizer / peephole. The bytes you get are the bytes the
     encoder emits; downstream tightening (rel8 tightening etc.) is
     tracked in v0.21 issues.
```

The doc is deliberately terse: it references, not restates, #1019 /
#1020 / #1024 design docs. Consumers who want to dig deeper open those.

## 5.2 Why this doc lives in paideia-as, not paideia-os

Two candidates for the home:

- `tools/paideia-as/design/toolchain/dynamic-emit.md` (paideia-as tree)
- `design/paideia-os/wasm-jail/dynamic-emit.md` (paideia-os tree)

Choose **paideia-as**. The doc describes the *paideia-as public runtime
API contract*. Its stability, its versioning, its extension points all
belong to paideia-as's release train. A paideia-os-side doc would
inevitably duplicate content and drift from the crate's actual API
shape as future issues (rel8 tightening, JIT-page hooks) land. paideia-os
Phase-10 design docs cross-reference this one with a stable pointer:
`https://github.com/paideia-os/paideia-as/blob/v0.20.0/design/toolchain/dynamic-emit.md`.

# 7. Backtrack candidates

Named-and-owned, each with a specific trigger.

## 7.1 Reviewer objects that the example belongs under `paideia-as-runtime`

**Trigger.** Reviewer holds the AC's file path literally and wants
`crates/paideia-as-runtime/examples/wasm_add.rs`, arguing that "the
runtime crate is where consumers should look for demos."

**Fallback.** Create the example under runtime, importing
`paideia-as-emit` as a dev-dependency:

```toml
# runtime Cargo.toml
[dev-dependencies]
paideia-as-emit = { path = "../paideia-as-emit" }
```

This adds a dev-dep from runtime → emit. Since emit already depends on
runtime, this creates a **dev-only cycle**. Cargo permits dev-dep cycles
(they don't participate in the release build DAG), but every workspace
touch has to reason about it, and cargo tooling (`cargo tree`) reports
it as a warning-shaped signal.

**Cost.** ~5 LOC of Cargo config, one small mental-model cost per
workspace read. Recommended only if reviewer explicitly refuses the
`paideia-as-emit/examples/` home.

## 7.2 Reviewer asks for real WASM parsing via `wasmparser`

**Trigger.** Reviewer says the tiny hand-rolled `decode_body` doesn't
demonstrate integration with the actual WASM ecosystem.

**Fallback.** Add `wasmparser = "0.219"` under
`crates/paideia-as-emit/[dev-dependencies]`. Rewrite `decode_body` to
consume `wasmparser::FunctionBody`. The `lower()` function is unchanged
— the WasmOp enum is trivially reconstructible from
`wasmparser::Operator`.

**Cost.** +1 dev-dep, +30 LOC of adapter code (walk `Operator` variants,
translate to WasmOp), −20 LOC of hand-rolled decode. Net +10 LOC and a
new dev-dep. **Not recommended by default** — the point of the demo is
the paideia-as surface, not the WASM parser. But cheap to swap in if
reviewer asks.

## 7.3 The JIT-execute step is required in v0.20

**Trigger.** Reviewer or Phase-10 planning meeting says "the example
must prove the bytes actually run, not just decode."

**Fallback.** Add a Linux-only `#[cfg(target_os = "linux")]` block that
uses `libc` (workspace pin) to `mmap` a PROT_EXEC page, copy the bytes,
transmute to `unsafe extern "C" fn(i64, i64) -> i64`, call with (2, 3),
assert result == 5. Wrap in `unsafe`, document why.

```rust
#[cfg(target_os = "linux")]
fn jit_execute(bytes: &[u8]) -> i64 {
    use core::ffi::c_void;
    unsafe {
        let page = libc::mmap(
            std::ptr::null_mut(), 4096,
            libc::PROT_READ | libc::PROT_WRITE | libc::PROT_EXEC,
            libc::MAP_ANONYMOUS | libc::MAP_PRIVATE, -1, 0,
        );
        std::ptr::copy_nonoverlapping(bytes.as_ptr(), page as *mut u8, bytes.len());
        let f: unsafe extern "C" fn(i64, i64) -> i64 = core::mem::transmute(page);
        let r = f(2, 3);
        libc::munmap(page, 4096);
        r
    }
}
```

**Cost.** +~40 LOC, +1 dev-dep (`libc`), +2 `unsafe` blocks, +1
`#[cfg(target_os = "linux")]`. macOS support adds another cfg (mmap
protection flags differ); Windows support adds a whole `VirtualAlloc`
branch. All doable but each adds review surface.

**Recommendation.** Ship without. paideia-os Phase 10 owns the JIT-page
allocator (it has a kernel-side view of executable-page policy) and is
the right place for the execute test. Include a `TODO(paideia-os
phase-10)` comment in the example pointing to this backtrack.
