## 1. Scope

### 1.1 What #1019 delivers

- New workspace member `crates/paideia-as-runtime/`.
- The runtime crate is declared `#![no_std] extern crate alloc;` at its own
  root and compiles under a bare-metal target (verified in-workspace by
  `cargo check --target x86_64-unknown-none -p paideia-as-runtime`).
- Physical relocation of two IR items into the runtime crate:
  - `paideia-as-ir/src/instruction.rs` (1700 LOC) → `paideia-as-runtime/src/instruction.rs`.
  - The `IrNodeId` newtype from `paideia-as-ir/src/node.rs` (≈40 LOC) →
    `paideia-as-runtime/src/node_id.rs`. Rationale in §3.
- `paideia-as-encoder` becomes `#![no_std]`, drops its `paideia-as-ir`
  dependency, and points at `paideia-as-runtime` for `Instruction`,
  `Mnemonic`, `Operand`, `IntWidth`, `Cond`, `InstrMode`, `RegId`, `Scale`.
- `paideia-as-ir` gains a `paideia-as-runtime` dep and re-exports the moved
  items so every existing importer (elaborator, emitters, tests) keeps
  compiling with zero source-site edits.
- `paideia-as-runtime` re-exports:
  - the moved instruction types (`Instruction`, `Mnemonic`, `Operand`, …);
  - the `IrNodeId` newtype;
  - the encoder public surface (`emit_instruction`, `CodeBuffer`,
    `EncodeError`, `EncodeStats`, `EncodeOutput`, `LabelFixup`, `RelocKind`,
    `RelocSite`, `estimated_bytes`) via `pub use paideia_as_encoder::*`.
- The CLI crate `paideia-as` adds `paideia-as-runtime` to its `[dependencies]`
  ahead of the direct encoder dep. Encoder dep stays (elaborator + emitters
  still import from it directly); the runtime dep signals the seam.
- 8 unit tests in `crates/paideia-as-runtime/tests/` that exercise the
  re-exported surface (§5).

Deliberately out of scope for #1019, held for follow-ups:

| Deferred item | Handed to |
|---|---|
| Ergonomic wrapper `emit_instruction(&mut CodeBuffer, Instruction) -> Result<(), EmitError>` distinct from today's `encode_instruction(&Instruction, &mut CodeBuffer, &mut EncodeStats)` signature | **#1020** (pa-r20-002) — that issue explicitly reshapes the signature and introduces `EmitError` as a stable public enum. #1019 must not preempt that naming. |
| `resolve_symbols(&mut Buf, &SymbolTable)` | **#1025** (pa-r20-006) |
| WASM lowering example | **#1023** (pa-r20-005) |
| PQ signing integration | **#1026** (pa-r20-007) |
| Public-API surface snapshot / freeze test | **#1027** (pa-r20-008) |
| Migrating elaborator/emitter/test call sites off `paideia_as_encoder::…` and `paideia_as_ir::instruction::…` onto `paideia_as_runtime::…` | Post-milestone hygiene issue (see §7). |
| no_std migration of the rest of `paideia-as-ir` (~30 files use `std::collections::HashMap` etc.) | Not in v0.20. Runtime scope is only Instruction + IrNodeId, which are self-sufficient in `core+alloc`. |

### 1.3 What #1019 is not

The runtime crate is **not** a paideia-as language stdlib. It is a **Rust
library** (`libpaideia_as_runtime.rlib`) that host-side programs (the CLI
today; paideia-os phase-10 WASM jail tomorrow) link against to construct and
emit x86_64 instruction bytes. The paideia-as source-language runtime
(`crates/paideia-stdlib`) is orthogonal and untouched.

```rust
//! paideia-as-runtime
//!
//! Linkable library exposing the paideia-as instruction IR and x86_64
//! encoder. Consumed by the paideia-as CLI and by paideia-os host processes
//! that emit code dynamically (WASM jail, JIT paths).
//!
//! `no_std + alloc` at the crate root; std is available only when a
//! downstream (e.g. the CLI) links us in a std context.

#![warn(missing_docs)]
#![forbid(unsafe_code)]

pub mod node_id;

    CodeBuffer, EncodeError, EncodeOutput, EncodeStats, LabelFixup, RelocKind,
    RelocSite, encode_instruction, estimated_bytes,
};
```

## 3. `IrNodeId` sub-decision

`instruction.rs` currently has exactly two imports: `crate::node::IrNodeId`
and `smallvec::SmallVec`. `IrNodeId` is a `NonZeroU32` newtype with a
`Display` impl and a `new(u32) -> Option<Self>` constructor — 40 LOC, zero
std usage, zero coupling to the rest of `node.rs`. It is used as a leaf
identifier by every IR-touching crate.

**Decision**: Move `IrNodeId` into `paideia-as-runtime/src/node_id.rs`. Have
`paideia-as-ir/src/node.rs` do `pub use paideia_as_runtime::IrNodeId;` at
its former definition site.

**Rejected alternatives**:

- *Duplicate IrNodeId in runtime under a different name*. Would fork the type
  identity across crates and break every downstream that mixes IR-produced
  IDs with runtime-consumed IDs. Non-starter.
- *Redefine IrNodeId inline in `instruction.rs` and re-export a runtime-local
  copy*. Same forking problem in a smaller package.
- *Break `Instruction` from IrNodeId — parameterise Instruction over an ID
  type*. Aggressive generics churn for zero benefit; call-sites would sprout
  turbofish everywhere.

Straight move + re-export is minimal, preserves nominal identity, and keeps
the ir crate compile-clean.

## 4. Public API surface

For #1019 the surface is exactly the union of the two things being unified,
re-exported verbatim. No new type introductions, no renames, no signature
reshaping — those are #1020's job and doing them here would collide.

### 4.1 Types (all moved from paideia-as-ir/src/instruction.rs)

`Instruction`, `Mnemonic`, `Operand`, `IntWidth`, `Cond`, `InstrMode`,
`RegId`, `Scale`, `SegPrefix`, `SegReg`, `EncodingHint`, `CpuFeature`,
`IrNodeId`.

### 4.2 Functions (re-exported from paideia-as-encoder)

- `encode_instruction(inst: &Instruction, buf: &mut CodeBuffer, stats: &mut EncodeStats) -> Result<EncodeOutput, EncodeError>`
- `estimated_bytes(inst: &Instruction) -> usize`

### 4.3 Support types (re-exported from paideia-as-encoder)

`CodeBuffer`, `EncodeError`, `EncodeOutput`, `EncodeStats`, `LabelFixup`,
`RelocKind`, `RelocSite`.

### 4.4 Explicit non-goals of the #1019 surface

The runtime crate does not yet ship:

- `emit_instruction` under that name (deferred to #1020 — collision with
  encoder's `encode_instruction` is intentional: the wrapper adds error
  homogenisation and buffer-first ordering).
- `EmitError` (deferred to #1020).
- `resolve_symbols` or any symbol-table primitive (deferred to #1025).
- Any `sign_*` entry point (deferred to #1026).

## 6. Migration plan

Twelve steps, one PR, one commit-per-step preferred for reviewability.

1. `mkdir crates/paideia-as-runtime && cargo init --lib` scaffolding.
2. Add `crates/paideia-as-runtime` to workspace members.
3. Write the runtime `Cargo.toml` (§2.1).
4. Write `src/lib.rs` skeleton (§2.3) with empty `instruction` + `node_id`
   modules.
5. **Move** `paideia-as-ir/src/instruction.rs` → `paideia-as-runtime/src/instruction.rs`.
   Change `use crate::node::IrNodeId;` → `use crate::node_id::IrNodeId;`.
6. **Move-extract** `IrNodeId` (the struct, its inherent impl, its `Display`
   impl, its `const_assert` size checks) from `paideia-as-ir/src/node.rs`
   into `paideia-as-runtime/src/node_id.rs`.
7. `paideia-as-ir/src/node.rs`: insert `pub use paideia_as_runtime::IrNodeId;`
   at the former definition site.
8. `paideia-as-ir/src/lib.rs`: change the `pub use instruction::…;` line to
   `pub use paideia_as_runtime::instruction::…;` (verbatim same identifier
   list). Delete `pub mod instruction;`.
9. `paideia-as-ir/Cargo.toml`: add `paideia-as-runtime = { path = "../paideia-as-runtime" }` under `[dependencies]`.
10. `paideia-as-encoder/src/*.rs`: replace `use paideia_as_ir::{…}` (only
    instruction items — verified in §7.3) with `use paideia_as_runtime::{…}`.
    Add `#![no_std] extern crate alloc;` at `lib.rs`.
11. `paideia-as-encoder/Cargo.toml`: drop `paideia-as-ir` dep; add
    `paideia-as-runtime`. `smallvec` + `thiserror` already listed.
12. `paideia-as/Cargo.toml` (CLI): add `paideia-as-runtime` to
    `[dependencies]` (satisfies the "CLI depends on runtime crate" AC line).

### 6.1 Verification loop after each cluster

- After step 4: `cargo check -p paideia-as-runtime` — empty scaffold builds.
- After step 8: `cargo check -p paideia-as-ir` + `cargo check -p paideia-as-encoder` — the two hinge crates build.
- After step 10: `cargo check --workspace` — every downstream still compiles unchanged.
- End: `cargo test --workspace` (5045 tests), then the runtime crate's 8 new tests.

### 6.2 Workspace impact summary

| Crate | Change |
|---|---|
| `paideia-as-runtime` | NEW. ~50 LOC scaffold + ~1740 LOC moved-in + ~400 LOC new tests. |
| `paideia-as-ir` | Delete `pub mod instruction;`, delete `IrNodeId` definition, add runtime dep + two `pub use` re-exports. ~10 LOC diff. |
| `paideia-as-encoder` | Swap dep from ir → runtime, add `#![no_std] extern crate alloc;`, retarget imports. ~20 LOC diff across ≤7 files. |
| `paideia-as` (CLI) | +1 dep line. |
| Every other crate (elaborator, emitters, tests) | **Zero source edits** because ir re-exports preserve the public paths. |

**LOC totals**: net-new ~490 (scaffold + tests), moved ~1740, edited ~30 across the workspace. Well inside the issue's ~600 LOC net budget.

## 7. Backtrack candidates

Named risks the author of this issue should re-scope from if they hit them.
Listed in order of estimated likelihood.

### 7.1 `paideia-as-encoder` won't go `#![no_std]` cleanly

Encoder has zero direct `use std::` (verified) but derives `thiserror::Error`
(`encode_instruction.rs:61`). thiserror 2.x supports no_std since 2.0 with
its `derive` feature; workspace declares `thiserror = "2"`, so it should
work. If a specific error variant references a std-only trait (unlikely —
they're all simple `#[error("…")]` strings), the fallback is:

- Keep encoder as `#![cfg_attr(not(feature = "std"), no_std)]` gated on a
  default-off `std` feature, and have the runtime crate turn it off.
- Or, drop `thiserror` from encoder and hand-implement `core::error::Error`
  (Rust 1.81+; workspace is on 1.94).

**Backtrack**: If either fix balloons past 100 LOC, split the "no_std
encoder" out into its own issue **pa-r20-001b** and let #1019 land with
runtime as `#![no_std]` at its lib.rs but transitively pulling in a std
encoder. The AC's letter is met (the runtime crate itself compiles
no_std+alloc); the true no_std goal slips one issue. This slip should be
called out in the PR and a follow-up filed immediately.

### 7.2 `IrNodeId` extraction breaks a hidden invariant

`IrNodeId` has `const_assert!(size_of::<Option<IrNodeId>>() == 4);` at
`node.rs:311`. If the assert lives in `node.rs` and the type moves, port
the assert to the new site verbatim. If the runtime crate's `#![no_std]`
setting somehow perturbs layout (it shouldn't — `NonZeroU32` is a lang
item), the fallback is to keep the assert in a `#[cfg(test)]` block.

### 7.3 An unnoticed encoder-side ir import that isn't instruction-family

The encoder's imports were surveyed (only from
`paideia_as_ir::{Cond, InstrMode, Instruction, IntWidth, Mnemonic, Operand,
RegId, Scale}` in `encode_instruction.rs:12-14`, plus
`paideia_as_ir::instruction::IntWidth` in `encode.rs:9`). If a stray import
exists in a file not surveyed, the fallback is: keep the `paideia-as-ir` dep
in encoder alongside the runtime dep — the runtime crate is still
extraction-complete, encoder is just the fatter of the two consumers.

### 7.4 iced-x86 round-trip regresses

Extremely unlikely — this is a pure move — but the discipline line in the
issue is explicit: "iced-x86 round-trip must stay green through every
commit". If it regresses, the issue is almost certainly step-6 (IrNodeId
extraction) messing with enum reprs. Bisect at commit granularity.

### 7.5 Elaborator's dev-deps stop resolving `paideia_as_encoder::encode_instruction`

`crates/paideia-as-elaborator/src/emit_walker_tests/scratch_and_ops.rs`
calls `paideia_as_encoder::encode_instruction` 15+ times. Encoder still
exposes it (nothing moved out of encoder), so those calls keep working. If
they don't, the fix is to update the imports to
`paideia_as_runtime::encode_instruction` — mechanical.

## 9. Tractability assessment

**Single-PR / single-workerbee-pass: YES.**

Rationale:
- Twelve mechanical steps, each ≤50 LOC diff.
- No new algorithms, no signature reshaping (that's #1020).
- Zero source changes to elaborator/emitters/tests thanks to the ir-side
  re-export layer — the workspace stays functionally identical.
- The 8 unit tests are small (~50 LOC each on average).
- The "compiles no_std+alloc" AC is bounded by encoder-side work (§7.1);
  even the fallback (feature-gate + slip to #1019b) is a labeled backtrack,
  not a scope explosion.

Risk band: **low**. The one honest hazard is the encoder-goes-no_std check
(§7.1); everything else is mechanical file moves plus re-export lines.

Estimated workerbee runtime: 30-45 minutes for the moves, verification, and
test authoring. Debugger triangle (softarch → workerbee → debugger, per
issue Discipline line) closes the loop.

**Prereq gaps: none.** The issue lists `Deps. None` and that survives
inspection — nothing in v0.18 or v0.19 is a blocker. HEAD 1df49e7 has
everything the extraction needs.

## 10. Cross-repo (paideia-os) impact

Per the issue body's "Cross-repo (paideia-os) impact" line: this extraction
does not by itself retire any paideia-os workaround (the WASM jail wiring
happens at #1020/#1023). But it establishes the **crate identity**
paideia-os will link against, so the phase-10 WASM path can start
prototyping its `Cargo.toml` line as soon as #1019 lands.

No paideia-os submodule bump is required from #1019 alone — that comes with
#1020 when the stable `emit_instruction` façade appears.
