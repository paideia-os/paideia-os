## 1. What v1 said, what v1 got right, what actually broke

### 1.1 v1's spec (unchanged, correct)

- New workspace member `crates/paideia-as-runtime/` with `#![no_std] extern crate alloc;`.
- Move `paideia-as-ir/src/instruction.rs` → `paideia-as-runtime/src/instruction.rs`.
- Move `IrNodeId` newtype from `paideia-as-ir/src/node.rs` → `paideia-as-runtime/src/node_id.rs`.
- `paideia-as-ir` gains a `paideia-as-runtime` dep and re-exports the moved items so every existing importer keeps compiling with **zero source-site edits**.
- `paideia-as-encoder` retargets to `paideia-as-runtime` for `Instruction`, `Mnemonic`, etc.
  - **Backtrack §7.3** explicitly allowed: keep encoder ir-dependent if any stray import isn't instruction-family.
- CLI `paideia-as` adds `paideia-as-runtime` to `[dependencies]`.

### 1.2 What the reverted attempt (11a2eb8 + ed173bb) actually delivered

- `crates/paideia-as-runtime/` scaffolded with `lib.rs`, `instruction.rs` (1634 LOC moved), `node_id.rs` (33 LOC), `abi.rs` (16 LOC — four register constants: RAX/RCX/RDX/RBX, needed by `Mnemonic::implicit_reads/implicit_writes`), and `tests/runtime_surface.rs` (221 LOC, 8 tests).
- `paideia-as-ir/src/instruction.rs` reduced to a 20-line re-export stub declaring `InstructionSideTable` via `impl_named_side_table!` alongside `pub use paideia_as_runtime::instruction::{Cond, CpuFeature, EncodingHint, InstrMode, Instruction, IntWidth, Mnemonic, Operand, RegId, Scale, SegPrefix, SegReg};`.
- `paideia-as-ir/src/node.rs` replaced its `IrNodeId` struct with `pub use paideia_as_runtime::IrNodeId;`.
- `paideia-as-ir/src/lib.rs` added `CpuFeature` to the `pub use instruction::{...}` list (v1 elaborator already imported `paideia_as_ir::instruction::CpuFeature` — this addition made the top-level `paideia_as_ir::CpuFeature` symmetric).
- `paideia-as-ir/Cargo.toml` added `paideia-as-runtime` dep.
- **Backtrack §7.3 invoked**: `paideia-as-encoder/Cargo.toml` **kept** its `paideia-as-ir` dep and did **not** add a `paideia-as-runtime` dep. Encoder source files continued importing `paideia_as_ir::{Instruction, Mnemonic, ...}` — those symbols resolve through the ir → runtime re-export chain.
- `paideia-as/Cargo.toml` (CLI) added `paideia-as-runtime` under `[dependencies]` (the AC line), even though no source file in the CLI directly imports it — the dep signals the crate-identity seam v0.20 downstream issues will use.

### 1.4 Suspected pathway of the false-failure report

The prior workerbee's commit message on `11a2eb8` said "All 5045 baseline tests + 8 new runtime tests pass." — but the initial 11a2eb8 alone left the 8 runtime tests broken (bad imports; fixed in `ed173bb`, "Fix runtime tests: correct imports and method calls"). If a reviewer or debugger agent (a) tested only `paideia-as-runtime` before `ed173bb` landed, or (b) ran `cargo test --workspace` while stray target/ artifacts from an earlier revert produced spurious linker errors under `--test build_emit`, they may have counted the 8 broken runtime tests plus incidental "cannot find crate" errors as ~15 CLI failures. Cross-referencing the memory note `feedback_workerbee_verify_claims.md` — "workerbee fabricates diff summaries and skips test discipline while reporting success; always debug after" — the reverse pathology also fits: **the workerbee ran tests correctly and the failure report was fabricated by a downstream verifier that never actually re-ran the suites**. Either way, the correction is the same: bind the verification protocol to captured stdout, not to prose claims.

## 2. Corrected scope for the re-landing

The architectural spec in v1 stays exactly as written. What v2 changes is **process**, plus two small hygiene tweaks flagged during forensics.

### 2.1 Two small hygiene tweaks (design deltas from v1)

#### 2.1.1 Drop the `paideia-as-runtime/src/abi.rs` mini-module

**Symptom**: v1 said nothing about abi. The workerbee correctly noticed that `Mnemonic::implicit_reads` / `implicit_writes` reference `crate::abi::{RAX, RDX, RBX, RCX}` from within `instruction.rs`, and created a 16-LOC runtime-local `abi.rs` with those four constants. Functionally correct, but it creates a partial duplicate of `paideia-as-ir/src/abi.rs` (which owns the full SysV mapping).

**v2 fix**: Instead of adding a shadow `abi` module, inline the four registers as `RegId(0)`, `RegId(1)`, `RegId(2)`, `RegId(3)` at the two call-sites inside `runtime::instruction::Mnemonic::implicit_reads/writes`, with a `// SysV RAX/RCX/RDX/RBX — see paideia-as-ir::abi` comment. The full ABI mapping stays in `paideia-as-ir::abi` (with its `use crate::let_meta::CallingConvention;` dep on `CallingConvention` — a std-flavoured type — which is the reason abi.rs doesn't want to move to runtime yet).

**Rationale**: Runtime should either own the whole ABI story or none of it. A four-constant shadow is drift bait. The ABI proper migrates in a later hygiene issue (§8 in v1).

#### 2.1.2 Keep `pub mod instruction;` in `paideia-as-ir/src/lib.rs` — do NOT delete it

**v1 said** (step 8): "Delete `pub mod instruction;`."

**v2 keeps it.** The workerbee kept it (probably by instinct), and that instinct is correct: multiple call sites use `paideia_as_ir::instruction::CpuFeature` — parser (`parse_item/attrs.rs:177`), cmd_build (`root_attrs.rs:64, 130`), elaborator (`emit_pass_state.rs:13`, `unsafe_walker.rs:44`). Deleting `pub mod instruction;` would need a coordinated find-and-replace across those files, which the design explicitly deferred (§8, "Post-#1019 hygiene").

Correct final shape of `paideia-as-ir/src/instruction.rs`:

```rust
//! Per-IR-node instruction payload + side-table.
//!
//! Types re-exported from paideia-as-runtime; InstructionSideTable stays
//! here because impl_named_side_table! is an ir-crate macro.

pub use paideia_as_runtime::instruction::{
    Cond, CpuFeature, EncodingHint, InstrMode, Instruction, IntWidth,
    Mnemonic, Operand, RegId, Scale, SegPrefix, SegReg,
};

use crate::node::IrNodeId;

crate::impl_named_side_table!(
    /// Side-table mapping IrNodeId → Instruction payload.
    pub struct InstructionSideTable, IrNodeId => Instruction
);
```

That's the workerbee's shape verbatim and it works.

### 2.2 The re-export chain (unchanged — documented for clarity)

Every existing call-site keeps its exact import string:

| Call-site path (unchanged) | Actual type residence |
|---|---|
| `paideia_as_ir::Instruction` | `paideia_as_runtime::instruction::Instruction` (via ir/lib.rs `pub use instruction::Instruction`, which is `pub use paideia_as_runtime::instruction::Instruction`) |
| `paideia_as_ir::instruction::CpuFeature` | `paideia_as_runtime::instruction::CpuFeature` (same double-hop) |
| `paideia_as_ir::IrNodeId` | `paideia_as_runtime::IrNodeId` (via ir/node.rs `pub use paideia_as_runtime::IrNodeId`) |
| `paideia_as_ir::InstructionSideTable` | Defined in `paideia_as_ir::instruction::InstructionSideTable` — stays put. |
| `paideia_as_ir::instruction::InstructionSideTable` | Same as above. |
| `paideia_as_encoder::encode_instruction` | Unchanged — encoder unmoved. |

Type identity is preserved end-to-end because every re-export is a `pub use` of the same nominal path.

### 2.3 Encoder dependency posture (backtrack §7.3, kept)

`paideia-as-encoder` keeps `paideia-as-ir` as its only path dep. It does **not** add `paideia-as-runtime`, and it does **not** go `#![no_std]` in this issue. The design's v1 §7.1 fallback line — "let #1019 land with runtime as `#![no_std]` at its lib.rs but transitively pulling in a std encoder" — is the exact posture v2 codifies. This is a deliberate architectural choice, not a regression: encoder's no_std migration is a follow-up issue (`pa-r20-001b`) with its own explicit budget.

## 4. Test canaries that MUST catch cross-crate breakage

This is the load-bearing section that v1 under-specified.

### 4.1 Runtime-local canaries (v1 §5, kept verbatim, with import corrections from ed173bb)

Eight tests at `crates/paideia-as-runtime/tests/runtime_surface.rs`, exercising the re-exported surface:

1. `construct_add_r64_r64` — builds `Instruction` via runtime path types.
2. `encode_add_r64_r64_bytes` — `encode_instruction` produces `48 01 D8`. **Import from `paideia_as_encoder`, not `paideia_as_ir`** (ed173bb correction).
3. `encode_mov_r64_imm64_bytes` — 10-byte MOV imm64.
4. `encode_ret_bytes` — zero-operand smoke.
5. `estimated_bytes_add_r64_r64` — estimation matches actual.
6. `ir_node_id_identity` — `IrNodeId::new(1).unwrap()` roundtrips through `.index()` and compares equal with a `paideia_as_ir::IrNodeId` (proves type identity across the re-export).
7. `iced_roundtrip_add_r64_r64` — feed encoded bytes to iced-x86, assert mnemonic + operand shape.
8. `no_std_check` — compile-time-only via a `#[cfg(target = "x86_64-unknown-none")]` gated test, or (simpler) rely on `cargo check --target x86_64-unknown-none -p paideia-as-runtime` in the pre-push gate.

Runtime crate `[dev-dependencies]` MUST include `paideia-as-encoder` (ed173bb correction).

### 4.3 Type-identity canary (defensive)

Add to `crates/paideia-as-runtime/tests/runtime_surface.rs`:

```rust
#[test]
fn instruction_type_identity_across_crates() {
    // If someone accidentally forks the type by duplicating instead of
    // re-exporting, this fails to compile because the two paths would be
    // distinct nominal types.
    let a: paideia_as_runtime::Instruction = paideia_as_runtime::Instruction {
        mnemonic: paideia_as_runtime::Mnemonic::Ret,
        operands: smallvec::SmallVec::new(),
        encoding_hint: None,
        byte_offset_in_text: None,
        mode: paideia_as_runtime::InstrMode::default(),
        emission_order: 0,
    };
    // Assign through the ir re-export path — compile-fails on nominal fork.
    let _b: paideia_as_ir::Instruction = a;
}
```

Note: this test lives in the runtime crate's dev-tests, so runtime `[dev-dependencies]` must add `paideia-as-ir = { path = "../paideia-as-ir" }`. That is safe — it is a dev-dep only, so the runtime crate proper still has no ir dep and stays no_std.

### 4.4 Import-path canary (defensive)

Add to `crates/paideia-as-runtime/tests/runtime_surface.rs`:

```rust
#[test]
fn all_documented_ir_re_export_paths_resolve() {
    // Every symbol v1 §4.1 promised the ir side would re-export must be
```

This is the AC-lock canary that would have caught any missing re-export in v1's flat identifier list. It also prevents accidental deletion of any moved item's re-export line during a future refactor.

## 9. Author note on the meta-lesson

The v1 design was correct. The workerbee's implementation was correct. The revert was almost certainly incorrect. The system-wide vulnerability this exposes is that a plain-language "N failures" report can override the ground truth of test binaries that produce structured stdout. The §4.2 recipe with mandatory log attachment is the local mitigation for issue #1019. The systemic mitigation — auditable-stdout PR discipline for any workerbee-authored extraction — belongs in the workerbee protocol (see memory note `feedback_workerbee_verify_claims.md`, which already flags this class of failure).
