## 1. Scope

**Delivers**:
- Real bodies for `crates/paideia-stdlib/pdx/option.pdx` and `result.pdx`
  (today they are pseudo-code comments — see §3).
- Generic enum declarations `enum Option<T> { Some(T), None }` and
  `enum Result<T, E> { Ok(T), Err(E) }`. Parser accepts this
  (`parse_item/struct_enum.rs:23–24, 129–130`); elaborator accepts unused
  generic-param declarations (`m9_generic_enum_option_t.expect: exit_code=0`).
- One hand-monomorphized concrete instantiation per type used by the runtime
  canaries — `enum OptionU64 { Some(u64), None }`,
  `enum ResultU64U64 { Ok(u64), Err(u64) }` — placed in
  `pdx/option_u64.pdx` and `pdx/result_u64_u64.pdx`. This is what
  the kernel and canaries actually **use** today; the generic forms in
  `option.pdx` / `result.pdx` are the aspirational surface for the API
  once monomorphization lands.
- `unwrap_or` as a **free function** in each monomorphized file (no closure
  required — takes value `default: T`, pattern-matches internally).
- 20 fixture files under `crates/paideia-stdlib/pdx/` (12 already exist as
  empty stubs; 8 are new). Parse-clean tests in `parse_pdx.rs`.
- 4 runtime canaries under `crates/paideia-as/tests/runtime/` exercising
  constructor + match-based extraction round-trips.

**Defers** (with follow-up issue proposals):
- `map`, `and_then`, `map_err`, `ok_or` — all take a closure/fn-pointer
  argument. **Blocked** on #1233 (encoder fat-pointer primitives) → #994
  (closure type/lower) → #995 (closure call). File follow-up
  **#997b** "combinators map/and_then/map_err/ok_or on Option/Result"
  contingent on #995.
- True `impl<T> Option<T> { … }` blocks. The `impl<T>` grammar parses
  (`parse_item/trait_impl.rs:32`), but the elaborator drops
  `generic_params: _` (see `derive_fn_sig.rs:54`,
  `struct_registry.rs:378`) and the `MonomorphisationTable`
  (`paideia-as-ir/src/monomorphisation.rs`) is not yet populated by the
  elaborator. File follow-up **#997c** "generic-enum monomorphization
  end-to-end" — this is the strategic gate that turns the `_u64` files
  from hand-written aliases into auto-generated variants.
- Auto-prelude / `use stdlib::option;`. No module-import mechanism exists
  today; kernel files re-declare enums inline
  (`tests/build-emit/match_enum_pattern.pdx` declares `enum Result`
  in-file). File follow-up **#997d** "stdlib auto-prelude".

**Recommendation vs prompt guidance**: the task prompt suggested delivering
methods once impl-blocks work, and hand-monomorphizing to `Option<u64>` /
`Result<u64,u64>`. Both retained. **Additional** deviation: we ship the
generic declarations in `option.pdx` / `result.pdx` (not just monomorphized
aliases), because the parser already accepts them and the `m9_generic_*`
fixtures already prove they elaborate clean — punting the surface entirely
to `OptionU64` would regress the documented API. The generic declaration
is the canonical API surface; the `_u64` aliases are testing scaffolding
retired once #997c lands.

## 3. Concrete monomorphizations to seed

- `OptionU64` = `Option<u64>`. Only variant with payload uses `u64`;
  covers all current kernel needs (Vec::get returns `Option<u64>`,
  HashMap::get, etc.).
- `ResultU64U64` = `Result<u64, u64>`. Numeric error codes; matches the
  existing `match_enum_pattern.pdx` shape (`enum Result { Ok(u64), Err(u64) }`).

Explicitly **not** seeding `OptionPtr`, `OptionU32`, `ResultPtrU64`, etc.:
these accrete as the kernel needs them and are cheap to add. The MVP is
`{u64, u64}` because that is what all currently-passing enum runtime
fixtures already exercise.

## 7. Backtrack candidates

1. **Generic declaration in `option.pdx` fails to elaborate under the
   parse-clean gate**. Mitigation: fall back to non-generic
   `enum Option { … }` in the stdlib file (matches all working
   fixtures). The `_u64` aliases still deliver testable substance.
   Confidence: **HIGH** — `m9_generic_enum_option_t.expect` shows the
   declaration compiles clean today.
2. **Path resolution `Some(42u64)` (bare) vs `OptionU64::Some(42u64)`
   (qualified) ambiguity when multiple enums share `Some`.** With both
   `Option<T>` (generic) and `OptionU64` in the same crate, an unqualified
   `Some(42u64)` could resolve either way. Mitigation: **always use
   qualified `OptionU64::Some(…)`** in fixtures (matches all existing
   fixtures — e.g., `enum_cons_result_ok.pdx:6` writes
   `Result::Ok(42u64)`). Confidence: **HIGH**.
3. **`impl OptionU64 { fn unwrap_or(...) }` for the concrete alias may
   not elaborate.** Mitigation: ship `unwrap_or` as a free function
   (as shown in §4). This is a known-good code path — several existing
   fixtures declare `pub let name : (T) -> U !{} @{} = fn(...) -> …`.
   Confidence: **HIGH**.
4. **Fixture registration in `build_emit.rs` / `runtime.rs` module
   lists.** Additive change; no known blockers. Existing `mod`
   declarations at `crates/paideia-as/tests/build_emit.rs` and
   `runtime.rs` show the pattern.
5. **`parse_pdx.rs` `#[ignore]` posture.** Every existing parse test is
   `#[ignore]`d and only runs with `cargo test --ignored` after
   `cargo build --release`. Continue that posture for the new tests to
   avoid slowing the default CI loop. Un-ignore is a follow-up.
6. **`build.rs` plumbing**. `paideia-stdlib/build.rs` only conditionally
   regenerates the UEFI header template; no changes needed for
   `option.pdx` / `result.pdx`. No new build-time codegen.
7. **Module resolver / paths like `stdlib::option`**. Not needed —
   fixtures are stand-alone files consumed by the paideia-as driver,
   not imported from stdlib. `use` / `import` is a separate deferred
   issue (#997d).
