# v0.18 issue #995 — closure-call lowering (pa-r18-002)

**Repo**: paideia-os/paideia-as | **Milestone**: v0.18 (#105) | **HEAD**: bf5cd33 (4923 tests green)
**Depends on**: #994 (closure grammar + lowering) LANDED, #1233 (encoder-side closure primitives) LANDED. **Blocks**: #996 (hashmap w/ closure hash-fn).

## 1. Scope

**Delivers**:
- `f(args)` where `f` is a closure-typed function-local binding lowers to the
  invocation sequence `mov r11, f_reg; mov r14, [r11+0]; call [r11+8]`, with
  caller-side arg marshalling into SysV RDI-R9 unchanged.
- Call-site detection via a new `LocalBindingTable::BindingHome::Closure(RegId)`
  variant that classifies closure lets distinctly from scalar/pair/env-slot
  homes.
- 0-6 argument closures; zero-capture and multi-capture bodies; statement and
  final-expression call positions.
- Runtime-verified invocation: 6 canaries that build → link → execute → check
  the returned integer.
- Calling-convention doc amendment: closure convention exempts R14 from the
  standard SysV callee-save discipline for closure body callees.

**Defers**:
- **Nested closure invocation** (closure body invoking another closure).
  Requires R14 spill/reload around the inner call — flagged with a
  best-effort diagnostic (T0575) for phase 1.
- Closure passed as a call argument to another function (16-byte value
  passing — needs pass-by-pointer or two-register form).
- Closure returned from a function (fat-pair sinking to caller frame).
- Closure stored in a record field / array element / global.
- Cross-ABI closure invocation (MS x64 callee).
- Higher-arity closures (>6 args, SysV limit reused).
- Escape-analysis; heap-allocated env (v0.20).

## 2. Call-site detection strategy

The closure binding `let f = <closure literal>` is lowered by
`closure_dispatch::convert_closure_lets` into `Let(ClosureCons(Lambda))`.
`emit_block_body.rs::490` (the `IrKind::ClosureCons` Let-RHS arm) emits
`emit_closure_cons`, which leaves the fat-pointer *address* in RAX
(`lea rax, [rsp + fat_off]`), then binds `f` to a scratch register.

Today (`emit_block_body.rs:497`), that binding uses the plain scalar path:

    self.state.local_bindings.insert(binding_name.clone(), scratch_reg);

This is the root of the trap. `LocalBindingTable::insert` stores
`BindingHome::Reg(reg)` — indistinguishable from a scalar u64 binding.
When the `IrKind::App` arm of `visit_lambda` (`emit_visit_lambda.rs:578`)
sees `f(x)`, it calls `self.state.local_bindings.get("f")`, receives
`Some(reg)`, and dispatches to `emit_indirect_call_via_reg`, which treats
`reg` as a code pointer. That register holds the address of a 16-byte fat
pair, not code — the call jumps into the env_ptr slot's contents,
segfaulting on any real target.

**Detection**: extend `BindingHome` with a `Closure(RegId)` variant
distinct from `Reg`. The invariant is that a `Closure(reg)` binding's
register holds a pointer to a 16-byte `[env_ptr:8 | code_ptr:8]` pair.

- `local_binding_table.rs`
  - Add `BindingHome::Closure(RegId)`; add `insert_closure(name, reg)`.
  - `get(name) -> Option<RegId>` returns `None` for `Closure` (same as
    `EnvSlot`) — existing callers already handle `None` conservatively.
  - `get_home(name)` returns `Some(BindingHome::Closure(reg))`.
  - `live_regs()` and `iter()` include the `Closure(reg)` register (it is
    still a scalar-register-held binding for scratch-save purposes).
- `emit_block_body.rs:497` — replace `insert` with `insert_closure` in the
  ClosureCons Let-RHS arm.
- `resolve_var_operands.rs:99` — add a `Closure(reg)` arm that rewrites
  `Var → Reg` identically to `Reg(reg)`. Non-call uses of `f` (e.g. a
  hypothetical `let g = f`) treat it as a plain pointer.

**Dispatch chain** (`emit_visit_lambda.rs::visit_lambda`, `IrKind::App` arm,
around line 566-584): add a *new first check* before the existing
local-binding path:

    if let Some(BindingHome::Closure(reg)) = self.state.local_bindings.get_home(name) {
        self.emit_closure_call(lambda_node_id, reg, &app_children[1..], arena);
        self.emit_ret_after_call(lambda_node_id, CallingConvention::Sysv, arena);
        return;
    }

This dispatch must appear **before** the existing `get(name)` path at
line 579 so the closure sequence wins over the fn-ptr sequence for
closure-typed bindings. It must appear **after** the FieldAccess-based
call-through-record-field detections (line 462-564) — those key on the
callee being a FieldAccess, not a Var, so no overlap.

Statement-position calls (`emit_call_stmt` / expr-stmt arm in the block
walker) get the same detection. `emit_call_expr` is called from the pos-0
pair-arg splicer (`emit_call.rs:696`) and does not currently handle
closures — that path already errors on non-App callee shapes, so no
new dispatch is required for #995.

## 3. Emit sequence

New method `emit_closure_call(lambda_id, callee_reg, args, arena)` in
`emit_lambda.rs`. Structural sibling of `emit_indirect_call_via_reg`
(line 209) with three deltas.

```asm
; -------- 1. Snapshot fat-pair address to R11 --------
mov r11, callee_reg                 ; R11 caller-save, non-arg — safe stash

; -------- 2. Scratch-save live caller-save bindings --------
; Same discipline as emit_call.rs::emit_call_args_and_call (#1163/#1220):
; push any live binding in {RCX, RDX, R8, R9} that would be clobbered
; by the callee.
push <live-scratch-1>
push <live-scratch-2>
...

; -------- 3. Marshal args into SysV registers --------
mov rdi, <arg0>                     ; standard SysV order; may alias callee_reg
mov rsi, <arg1>                     ; (R11 already snapshotted, so safe)
mov rdx, <arg2>
mov rcx, <arg3>
mov r8,  <arg4>
mov r9,  <arg5>

; -------- 4. Load env_ptr into R14 --------
mov r14, [r11 + 0]                  ; env_ptr slot at offset 0

; -------- 5. Indirect call through code_ptr slot --------
call [r11 + 8]                      ; code_ptr slot at offset 8

; -------- 6. Restore scratch (LIFO) --------
pop <live-scratch-N>
...
pop <live-scratch-1>

; RAX now holds closure's return value (SysV integer-return convention).
```

**Register discipline**:
- **R11**: caller-save, not in SysV arg pool, not reserved — the same
  intermediate `emit_indirect_call_via_reg` uses for fn-ptr snapshot.
  Reuse without contention.
- **R14**: paideia-reserved for env-ptr. Loaded from `[r11+0]` before
  the call. Not in SysV arg pool (RDI, RSI, RDX, RCX, R8, R9), so does
  not conflict with arg marshalling regardless of arg count.
- **R14 preservation**: SysV nominally treats R14 as callee-save. The
  closure convention **exempts** the closure body from preserving R14 —
  the caller wrote R14 immediately before the call and does not expect
  its prior value back. Update `calling-convention.md` §11.5 to document
  this exemption alongside the existing R14/R15 reservation.
- **Ordering**: R14 load MUST follow the mov-to-R11 snapshot and the
  `arg-mov` sequence is order-agnostic w.r.t. the R14 load (R14 is not
  an arg reg). Placing R14-load before the CALL is required; placing it
  after arg marshalling reads best (adjacent to the CALL that consumes
  it).

**Zero-capture case**: `[r11+0]` holds 0 (per `emit_closure_cons`
zero-capture path). R14 receives 0. The closure body never emits
`[R14+off]` reads (its `closure_meta.captures` is empty, so
`visit_lambda`'s capture-registration loop registers no `EnvSlot`
bindings). The body executes correctly with R14 = 0; no null-safety
guard needed at the call site.

**Encoder support** (verified in `encode_instruction.rs`):
- `call [base + disp]` — line 3137, arm `[Operand::MemSib { base, ..., disp }]`.
- `mov r64, [base + disp]` — standard MOV MemSib arm.
- No new encoder work required.

## 4. Interaction with `emit_call`

`emit_call.rs::emit_call_args_and_call` (the shared direct-call path) is
not touched. Reasons to fork rather than parameterise:

- **ABI crossing** (bridge_saves, MS shadow-space, SysV align pad) is
  paideia→paideia for closures — always inert. Reusing
  `emit_call_args_and_call` would require passing sentinel values for
  five parameters that are always the same. A parameterised path would
  be harder to audit than a small parallel emitter.
- **stdlib recipe splice** (line 384) is inapplicable — closures are
  never `Trait::method` targets.
- **pos-0 pair enum handling** (`classify_pos_zero_pair`, line 78) is
  inapplicable — closure calls do not receive register-pair enum args
  in phase 1.

What #995 **does** copy verbatim into `emit_closure_call`:
- The scratch-save discipline (#1163, #1220): live bindings in
  `{RCX, RDX, R8, R9}` get push/pop-bracketed across the call.
- The `first_emission` shim discipline for `record_lambda_entry` /
  `arm_pending_first_instr_unless_claimed` — the first-emitted
  instruction (the `mov r11, callee_reg` snapshot) must claim
  `lambda_first_instr[L]` when the closure call is the caller's first
  emitted instruction (e.g. bare-tail call body).

**Consolidation for a later phase**: after #995 lands, extract the
scratch-save + first-emission-shim block into a shared helper
`emit_scratch_bracket(&mut self, live: &HashSet<RegId>, body: FnOnce(...))`
callable from both direct and closure paths. Not part of #995's scope.

## 5. Test canaries

Six `.pdx` fixtures under `tests/build-emit/closure_type/` + matching
`.rs` runtime drivers under `crates/paideia-as/tests/runtime/closure_type/`.
All drivers use the existing `run_and_verify` harness (`RetTy::I64`,
integer exit-code assertion) — same shape as
`closure_no_capture.rs:closure_no_capture_returns_42`.

The AC's "10 integration tests" splits as 6 runtime canaries + 4
build-emit assertion tests (encoder byte-pattern checks against the
emitted sequence).

**Runtime canaries** (must invoke the closure and derive result from the
invocation's return value — this is where things get REAL):

1. **`closure_call_zero_capture.pdx`** — bare invocation, no captures.
   `let f : |u64| -> u64 = fn (x: u64) -> x + 1; f(41)` returns 42.
   Verifies the invocation sequence executes end-to-end; verifies R14 = 0
   is harmless for zero-capture bodies.

2. **`closure_call_single_capture.pdx`** — invoke closure that reads one
   capture.
   `let c : u64 = 37; let f : |u64| -> u64 = fn (x: u64) -> x + c; f(5)`
   returns 42.
   Verifies caller writes capture to `[rsp+env_off]`, `emit_closure_cons`
   stores `env_ptr = &captures[0]`, R14 gets that pointer at call, body
   reads `[r14+0]` to recover `c`.

3. **`closure_call_two_captures.pdx`** — two captures at offsets 0 and 8.
   `let a: u64 = 10; let b: u64 = 32; let f: |u64| -> u64 = fn (x: u64) -> a + b; f(0)`
   returns 42.
   Verifies symbol-sorted env layout (per `capture.rs::analyze_captures`)
   places `a` at `[r14+0]`, `b` at `[r14+8]`.

4. **`closure_call_two_args_zero_capture.pdx`** — multi-arg, no captures.
   `let f: |u64, u64| -> u64 = fn (x: u64, y: u64) -> x + y; f(20, 22)`
   returns 42.
   Verifies R14 load doesn't clobber RDI/RSI arg regs.

5. **`closure_call_multiarg_with_capture.pdx`** — args + capture combined.
   `let base: u64 = 40; let f: |u64, u64| -> u64 = fn (x: u64, y: u64) -> base + x - y; f(3, 1)`
   returns 42.
   Verifies the caller's env write and R14 load coexist with 2-arg
   marshalling; verifies binding-in-arg-reg no-op path
   (`emit_call.rs:525`).

6. **`closure_call_after_intervening_direct_call.pdx`** — construct
   closure, invoke direct-call function that clobbers scratch, invoke
   closure. Something like:
   `let f: |u64| -> u64 = fn (x: u64) -> x + 1; let sink: u64 = other(7); f(41)`
   returns 42, where `other` is a plain SysV function.
   Verifies the closure binding survives an intervening call — the
   scratch-save discipline must include `Closure(reg)` in `live_regs()`
   when the closure's home register is in the caller-save scratch pool.

**Build-emit assertion tests** (byte-pattern checks against `.text`
emission — the closure-invocation sequence has a signature footprint
that lets us assert it lands in the right shape without running):

7. **`closure_call_emits_r14_before_call.pdx`** — assert the emitted
   instructions contain `mov r14, [r11+0]` (opcode `4C 8B 33`) followed
   by `call [r11+8]` (opcode `41 FF 53 08`) in that order within the
   caller function's byte range.
8. **`closure_call_scratch_save_present.pdx`** — assert push/pop
   bookends when a caller has a live RCX/RDX/R8 binding at closure
   invocation.
9. **`closure_call_snapshot_r11_before_arg_marshal.pdx`** — assert
   `mov r11, <f_reg>` precedes the first arg-marshal MOV when `f`'s
   home aliases an arg reg (deliberately place `f` in RDI's slot).
10. **`closure_call_nested_invocation_diagnostic.pdx`** — verify T0575
    fires when a closure body itself contains a closure-typed call.
    (Compile-only; no runtime execution.)

## 6. Backtrack candidates

Where the workerbee is most likely to stumble. Watch here first.

- **`local_bindings.get(name)` at the App-arm dispatch fires today**.
  Any change to `emit_block_body`'s ClosureCons Let arm that swaps
  `insert` for `insert_closure` MUST be paired with the new call-site
  detection arm — otherwise the closure binding becomes invisible to
  every existing consumer (all of which use `get()`, which returns None
  for `Closure`), silently regressing existing #994 build-emit canaries
  which assert the binding is registered.
- **`get(name)` returning None for Closure**: existing call sites
  outside the App arm assume `get()` yields the register holding a
  usable scalar value. For a `Closure(reg)`, the scalar value IS the
  fat-pair address (a valid pointer). Decide per-site whether to route
  through `get()` (treats as scalar pointer) or `get_home()` (dispatches
  on Closure). Preferring `get_home()` is safer; verify no in-tree call
  site takes a raw `RegId` out of `get()` and misinterprets it as code.
- **`live_regs()` inclusion**: must include Closure(reg). If omitted,
  the scratch-save loop in `emit_call.rs` won't push the closure binding
  across an intervening direct call, silently corrupting it (canary #6
  will fail).
- **`resolve_var_operands` treats Closure as Reg**: adding a Closure
  arm is trivial but essential — omitting it leaves `Operand::Var { name }`
  unresolved, firing spurious T0528 for legitimate closure references
  in non-call contexts. The arm rewrites Closure → Reg identically to
  the Reg arm.
- **R14 preservation across intervening direct calls**: SysV callees
  preserve R14 as callee-save, so R14 written by our previous closure
  call and read later (in a different closure body) would in theory
  survive an intervening direct SysV call. However, the paideia
  `PAIDEIA_BRIDGE_SAVE = [R15, R14]` bracket at MS/SysV bridges saves
  and RESTORES R14, and closures are paideia-native (no bridge saves).
  Within a single caller, R14 is fresh-loaded before every closure
  call — no stale-R14 hazard.
- **Nested closure invocation clobber**: if closure body A itself
  invokes closure B via `emit_closure_call`, A's R14 (its own env-ptr)
  is overwritten by B's env-ptr. On return from B, A's captures
  become unreachable. Guard with a `check_lambda` diagnostic
  **T0575**: "closure body may not invoke another closure (nested
  closure invocation not yet supported)". Detection: scan closure body
  post-lowering for `IrKind::App` whose callee resolves to a
  `BindingHome::Closure` OR a nested `IrKind::ClosureCons`. Defer the
  real fix to a follow-up issue (R14 stack spill/reload in the closure
  prologue).
- **Scratch-save when `f_reg` is in the save set**: if `f`'s home is
  `Closure(RCX)`, RCX is both the fat-pair holder AND in the scratch
  pool. The scratch-save will push RCX (harmless — we've already
  snapshotted to R11); the scratch-restore will pop RCX (harmless —
  RCX was going to be clobbered by arg marshalling anyway). Net:
  redundant but correct. Do not attempt to skip it — the save runs
  before the R11 snapshot; skipping requires reordering.
- **`arm_pending_first_instr_unless_claimed` and bare-tail closure**:
  a lambda body `fn () -> f(x)` where the entire body is a closure
  call. The first-emitted instruction (the `mov r11, callee_reg`
  snapshot) must claim `lambda_first_instr[L]`. Reuse the App-arm's
  existing `arm_pending_first_instr_unless_claimed(lambda_node_id)`
  call at line 417 — it already fires for every App body. `first_id`
  discipline inside `emit_closure_call` mirrors `emit_call_args_and_call`.
- **Symbol name mismatch on the code_ptr LEA**: the closure body's
  ELF symbol is inserted by #1233's `register_closure_body_symbols`
  pre-pass; the `mangled_name` in `ClosureMeta` is used by both the
  LEA at construction site and the CALL at invocation site (which
  reads through the fat pointer, not by symbol — so #995 is symbol-
  name-agnostic here). But if #1233's symbol were somehow missing,
  `emit_closure_cons`'s `lea r11, [rip + mangled]` would leave a
  dangling PcRel32 reloc; the CALL would jump to garbage. Verify
  canary #1 builds cleanly end-to-end.
- **Encoder gap on `call [reg + disp]` non-zero disp**: verified
  present at `encode_instruction.rs:3137`. Ensure the emitted
  Instruction uses `Operand::MemSib { base: R11, index: None,
  scale: Scale::X1, disp: 8 }` (not `MemRipRel` or `MemRipRelSym`).

## 10. Deeper prerequisite gap flagged

None discovered. Every primitive #995 needs — fat-pair layout,
`ClosureMeta`, `ClosureFrameLayout`, capture-registration in the closure
body, closure body symbol registration in `.symtab`, the `emit_ret`
prologue/epilogue helper, `BindingHome::EnvSlot` — is present and
exercised by the three currently-passing #994 runtime canaries.

The single new machinery #995 introduces (`BindingHome::Closure` +
`emit_closure_call`) is additive and mirrors patterns already proven in
tree (`EnvSlot`, `emit_indirect_call_via_reg`). No refactor of shared
infrastructure is required; the scratch-save discipline in `emit_call.rs`
can be copied in-place rather than extracted (see §4 note).

The one non-trivial forward-looking gap — nested closure invocation
requiring R14 spill/reload — is deliberately deferred behind T0575 to
keep #995's blast radius within the AC's ~450 LOC estimate.
