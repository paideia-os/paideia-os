# v0.18 issue #1233 — encoder-level closure emit primitives

**Repo**: paideia-os/paideia-as | **Milestone**: v0.18 (#105) | **HEAD**: 695521f (4881 tests green)
**Blocks**: #994 (closure grammar + lowering), #995 (closure call), #996 (hashmap w/ closure hash-fn).
**Depends on**: none in-tree, but flags a nested prerequisite (§8) that #1233 must own.

## 1. Scope

**#1233 delivers pure emit-side infrastructure**: three encoder primitives that
two prior workerbee attempts on #994 lacked, plus the frame-layout stage that
those attempts assumed existed but does not.

1. **Symbol registration for closure body functions** — mangled name
   `closure_<parent>_<node_id>` reaches ELF `.symtab` via a new pre-pass in
   `EmitWalker::walk_inner`, mirroring the module-Let symbol path.
2. **Fat-pointer stack allocation + population** — 16 B `{env_ptr, code_ptr}`
   pair materialised on the caller's frame, plus the `env_rec` for captures.
3. **R14 env-register discipline** — closure body reads captures from
   `[R14 + off]`; caller's `#995` future call sequence `mov r14, [f+0]; call
   [f+8]` is enabled by the byte-layout committed here.

**#1233 does NOT deliver**: closure grammar, closure IR variant, capture
analysis wiring — those are #994's scope. #1233 designs the emit-side hooks
that #994's IR nodes plug into.

## 2. Anchoring: what the two failed attempts hit

Both workerbee attempts on #994 landed AST + parser + type-lowering + IR
side-tables. Both failed identically at runtime canary link:

```
undefined reference to `closure_<N>'
```

Root cause: `lea rax, [rip + closure_<N>]` emitted correctly, encoder
generates `PcRel32` relocation, but no `Symbol` for `closure_<N>` was ever
inserted into `arena.symbols_mut()`. The linker sees a dangling reference
because the closure body Lambda has no top-level `Let` parent to trigger
the symbol-insert branch at `emit_walker.rs:521`.

Adjacent gap: even if symbol registration were fixed, current paideia-as
functions emit `mov rax, ...; ret` with **no per-function stack frame**.
Reserving 16 B for the fat pair plus N B for the env-rec requires prologue
`sub rsp, N+16` and epilogue `add rsp, N+16` before every `ret` — neither
exists in the emit pipeline.

## 4. Fat-pointer allocation strategy

### 4.1 Byte layout (frozen by #1233; #995 depends on it)

At the ClosureCons site inside the caller's stack frame:

```
Caller frame slot for closure literal C:
  [rbp - fat_off        ]  env_ptr    (8 B)
  [rbp - fat_off + 8    ]  code_ptr   (8 B)
  [rbp - env_off        ]  capture[0] (8 B)
  [rbp - env_off + 8    ]  capture[1] (8 B)
  ...                          ... symbol-sorted per capture.rs
```

The value of `let f = <closure literal>` is a pointer to the 16 B pair, i.e.
`f == &[env_ptr, code_ptr] == rbp - fat_off`. #995 will emit
`mov r14, [f+0]; call [f+8]` against that pointer, so **the pair layout is
frozen at `[env:8 | code:8]`, offsets 0 and 8**.

**Zero-capture case**: `env_off` is absent; `env_ptr` slot stores 0. #995 still
emits `mov r14, [f+0]` unconditionally, loading 0 into r14 — harmless because
a zero-capture closure body never reads from `[r14+off]`.

### 4.2 Ownership of the slot

**Owner: the caller Lambda's frame layout**, computed by a new pre-pass at
`visit_lambda` entry. Not the ClosureCons emit itself.

- Per-Lambda side-table `closure_frame_meta: HashMap<lambda_id, FrameLayout>`.
- `FrameLayout { total_size: u64, cons_slots: HashMap<cc_id, (fat_off, env_off)> }`.
- Populated by `precompute_caller_frame(caller_lambda_id, arena)` — recursive
  scan of caller body for `IrKind::ClosureCons` descendants, sums their
  `16 + env_size` requirements, assigns non-overlapping `[fat_off, env_off]`
  pairs per ClosureCons.

`emit_closure_cons` reads its own `(fat_off, env_off)` from the side-table
and emits the mov sequence; it does NOT emit `sub rsp` — that is the
prologue's job (§4.3).

### 4.3 Prologue / epilogue emit — the nested prerequisite

**Current state (audited)**: paideia-as function bodies emit no prologue.
`emit_identity_lambda` emits 5 bytes total: `48 89 f8 c3` (`mov rax, rdi;
ret`). Even `emit_call.rs` `sub rsp, N` sequences are *around calls*, not
per-function frame reservation.

**Required for #1233**: a prologue/epilogue stage keyed on `frame_layout
[caller_id].total_size > 0`.

- **Prologue**, emitted before body dispatch inside `visit_lambda` when
  `total_size > 0`:
  ```
  sub rsp, total_size          ; reserve fat_pair + env_rec slots
  ```
  (No `push rbp; mov rbp, rsp` — paideia-as does not use rbp as frame base;
  all references are rsp-relative. Keep parity with the current no-rbp
  convention. Callers that need alignment beyond `sub rsp` handle it with
  extra padding in `total_size`.)

- **Epilogue**, emitted before every `ret` in the caller body:
  ```
  add rsp, total_size
  ret
  ```

**Refactoring implication**: every existing `ret` emit site needs a hook
that consults `frame_layout[current_function].total_size` before writing
the `ret`. There are ~20 such sites (grep `Mnemonic::Ret` in elaborator).
Design choice: introduce `emit_ret(&mut self, ret_id: IrNodeId)` helper on
`EmitWalker` that (a) emits `add rsp, N` if `N > 0`, then (b) emits `ret`.
Replace the ~20 raw-Ret literals with `self.emit_ret(id)`.

**Alignment**: SysV requires `rsp ≡ 0 (mod 16)` at CALL sites. If
`total_size` is not a multiple of 16, round it up (add 8 B pad). This is
free — the extra slot is just unused stack.

### 4.4 emit_closure_cons byte sequence

Given `(fat_off, env_off)` from `closure_frame_meta`:

```asm
; captures are symbol-sorted (per capture.rs::analyze_captures)
; capture c_i is at [rsp + env_off + i*8]

; For each capture (i, kind, src):
;   Consume/Value:  mov <tmp>, <src>
;                   mov [rsp + env_off + i*8], <tmp>
;   Reference:      lea <tmp>, <src>           ; take address
;                   mov [rsp + env_off + i*8], <tmp>
; (tmp scratched from RCX/RDX/R8/R9 per emit_block_body scratch pool)

; Materialise fat pair:
lea  r11, [rsp + env_off]        ; env_ptr = &captures[0]
mov  [rsp + fat_off + 0], r11    ; store env_ptr slot
lea  r11, [rip + closure_<mangled>]  ; code_ptr via PcRel32 reloc
mov  [rsp + fat_off + 8], r11    ; store code_ptr slot
lea  rax, [rsp + fat_off]        ; return pointer-to-fat-pair
```

**Zero-capture short-circuit**: skip env-populate loop, skip the `lea r11,
[rsp + env_off]`, emit `mov qword [rsp + fat_off + 0], 0` for env_ptr slot.
Encoder must support `mov [mem], imm32/imm64` — verified present at
`encode_instruction.rs`.

**Operand primitive reused**: `Operand::MemRipRelSym { name: "closure_...",
addend: 0 }` — already wired to `PcRel32` relocation via encoder line 3311
(`Lea` + `MemRipRelSym` form). No encoder changes required.

## 5. R14 discipline

### 5.1 Closure body entry sequence

R14 arrives holding `env_ptr` (set by caller's future `#995` sequence,
`mov r14, [f+0]`). The closure body:

```asm
; No prologue push/pop of R14 required.
; Captures resolve via [r14 + off] operand rewrites (§5.3).

; Standard SysV callee-save discipline for the OTHER regs: rbx, rbp, r12,
; r13, r15. R14 is the exception — closure body may freely clobber R14
; because the caller has just written to it and does not expect its prior
; value preserved across this call.
```

### 5.2 Interaction with SysV callee-save

- **SysV standard**: r12–r15 are callee-preserved.
- **Closure calling convention amendment**: closure bodies exempt R14 from
  callee-save. Justified because the caller wrote R14 immediately before
  the call — no useful prior value to preserve.
- **PAIDEIA_BRIDGE_SAVE = [R15, R14]** (verified at `abi.rs:142`): already
  saves R14 at paideia/MS ABI boundaries. Bridging into/out of a closure
  body via a non-paideia call path is safe.
- **Caller-side**: if caller uses R14 for its own bindings (should not —
  R14 reserved per calling-convention.md §11.5), it must push/pop around
  each closure invocation. Enforce by convention, not by emit-side
  bookkeeping: `emit_walker` never allocates R14 as a scratch or binding
  home.

### 5.3 Capture binding: [R14 + off] operand rewrites

Captures inside the closure body are `Operand::Var { name }` at emit time,
same shape as ordinary locals. `resolve_var_operands` rewrites them to
`Operand::Reg` today. For captures we need `Operand::MemSib { base: R14,
index: None, scale: X1, disp: off }`.

**Design**: extend `LocalBindingTable::BindingEntry` with a home variant:

```rust
// crates/paideia-as-elaborator/src/local_binding_table.rs
pub enum BindingHome {
    Reg(RegId),
    RegPair(RegId, RegId),      // #1154 existing pair form
    EnvSlot(i32),               // NEW: R14-relative offset (disp only)
}

pub struct BindingEntry {
    pub home: BindingHome,       // was `reg: RegId, payload: Option<RegId>`
}

// Companion API preserved for existing call sites:
impl LocalBindingTable {
    pub fn insert(&mut self, name: String, reg: RegId) { /* home = Reg(reg) */ }
    pub fn insert_pair(&mut self, name: String, reg: RegId, payload: RegId) { /* RegPair */ }
    pub fn insert_env(&mut self, name: String, off: i32) { /* EnvSlot */ }  // NEW

    pub fn get(&self, name: &str) -> Option<RegId>            { /* Reg only, backcompat */ }
    pub fn get_home(&self, name: &str) -> Option<BindingHome> { /* NEW */ }
}
```

`resolve_var_operands` (crates/paideia-as/src/resolve_var_operands.rs)
learns a new arm:

```rust
match effective_bindings.get_home(name) {
    Some(BindingHome::Reg(r))            => Operand::Reg(r),
    Some(BindingHome::RegPair(r, _))     => Operand::Reg(r),       // primary only
    Some(BindingHome::EnvSlot(off))      => Operand::MemSib {
        base: abi::R14, index: None, scale: Scale::X1, disp: off,
    },
    None => ... existing T0528 / T0531 path
}
```

### 5.4 Registration order inside visit_lambda for closure bodies

At closure body entry, `visit_lambda` recognizes closure-bodyness via
`arena.closure_meta().get(lambda_id).is_some()`:

```rust
if let Some(meta) = arena.closure_meta().get(lambda_node_id) {
    // 1. Register captures FIRST (lowest priority in scope walk).
    for cap in &meta.captures {
        self.state.local_bindings.insert_env(cap.name.clone(), cap.offset);
    }
    // 2. Then register params — inner param names shadow outer captures.
    self.register_nested_lambda_params(lambda_node_id, arena, 0);
} else {
    self.register_nested_lambda_params(lambda_node_id, arena, 0);
}
```

Param inserts overwrite in `LocalBindingTable::flat` — this is the desired
shadowing semantics.

## 7. Backtrack candidates

Each entry names a plausible stumble a workerbee will hit. Watch here first.

- **Symbol pre-pass placement**. If the closure-body symbol pre-pass runs
  after the `#1131` `mark_data_let_handled` pre-pass but before the main
  Let iteration at line 460, the flat walker's main Lambda arm (line 574)
  will pick up the closure body Lambda normally. If placed *after* the
  main iteration, `visit_lambda` on the closure body will have already
  fired without a symbol registered — the symbol arrives too late, but
  the emit succeeds because symbol insertion is idempotent by name (line
  137 in `symbol.rs`). Ordering is not correctness-critical for the
  Function symbol itself, but IS critical for any encoder path that
  consults `arena.symbols().lookup_by_name(name)` during instruction
  emission — grep confirms only `resolve_var_operands` does this (for
  T0531), and that path runs after `walk_inner` returns. **Recommendation**:
  put the pre-pass before line 460 for clarity, not correctness. Flag: a
  future emit-time symbol lookup could regress silently — document that
  the pre-pass runs before line 460.

- **`ret` sites not routed through `emit_ret`**. There are ~20 `Mnemonic::
  Ret` literals scattered across `emit_lambda.rs`, `emit_visit_lambda.rs`,
  `emit_arith_lambda.rs`, `emit_control_flow.rs`. Every one MUST become
  `self.emit_ret(id)`. Miss one → stack imbalance → segfault in runtime
  canary #5. Add a `#[deny(clippy::disallowed_methods)]` gate on raw
  `Mnemonic::Ret` construction inside `impl EmitWalker`; require going
  through `emit_ret`. Alternatively, a grep-based pre-push canary similar
  to the existing opcode-canary (see `.plans/refactor-*`), keyed on
  `Mnemonic::Ret` count vs `emit_ret` call count.

- **Frame size computed on stale arena state**. `precompute_caller_frame`
  runs at `visit_lambda` entry and scans the caller body. If ClosureCons
  nodes are added during elaboration *after* this pre-pass, they'll not
  be counted. Constrain: ClosureCons IR insertion must complete before
  the `EmitWalker::walk` call. This is naturally true because #994's
  lowering runs before `walk`, but make it a documented invariant.

- **`Operand::MemSib { base: R14, ... }` from `resolve_var_operands`**.
  If the encoder path for MemSib with base=R14 has any assumption that
  R14 is not used as a base (unlikely — R14 is a normal r64 with REX.B),
  it will silently mis-encode. Verify by hand-encoding one instance and
  comparing against `objdump -d`. `encode_instruction.rs` handles
  arbitrary r64 base for MemSib via `reg64_from` — no R14-specific
  gotchas expected, but audit before landing.

- **Closure body emit order vs caller symbol offset**. The flat walker
  visits nodes by ascending id. Child ids < parent ids in the arena, so
  closure bodies emit BEFORE their enclosing caller Lambda. That places
  closure-body bytes ahead of caller bytes in `.text`. `function_offsets`
  is derived per-Lambda so this is correct — but symbol sizes computed
  via `sorted_offsets.partition_point` in `cmd_build/elf.rs:265` assume
  contiguous per-symbol ranges. Verify no closure-body-to-caller ordering
  produces a `size = 0` regression on canary #1.

- **Zero-capture env_ptr = 0 vs null-safety of `[r14+off]`**. Zero-capture
  closure bodies must never emit `[r14+off]` reads because R14 = 0 will
  segfault. Guard: if `closure_meta.captures.is_empty()`, do NOT register
  any `insert_env` entries — all `Var` operands resolve to param regs.
  Test canary #2 verifies the zero-capture body reads only from RDI.

- **Caller and closure body share the arena, share `state.per_lambda_
  bindings`**. Between visits, `local_bindings` must be cleared or scoped
  correctly so the caller's captures do not leak into the closure body's
  binding table. Line 576 in `emit_walker.rs` already calls
  `self.state.clear_scratch()` at each Lambda entry — extend to also
  clear `local_bindings` at closure-body entry, OR use fresh
  `LocalBindingTable::new()` per Lambda visit and snapshot into
  `per_lambda_bindings` at exit.

## 8. Deeper prerequisite gap flagged

The prologue/epilogue infrastructure in §4.3 is a genuine emit-side gap
that is bigger than #1233's original one-paragraph description. Two ways
to sequence:

- **(A) Fold into #1233 (RECOMMENDED)**. Ship prologue/epilogue as part of
  the closure-emit primitives. Adds ~150 LOC but keeps the design
  self-contained. `emit_ret` becomes the single required refactor point.
- **(B) File a new prerequisite issue #1234 "Emit-side per-function frame
  layout"**. #1233 depends on #1234. Cleaner separation, but adds a hop.
  Prefer only if #1234 has independent value (it does, faintly — future
  spill/reload will want it).

**Verdict for #1233**: fold in. The prologue/epilogue is small enough
(three touch points: `visit_lambda` entry, new `emit_ret` helper, ~20
`Mnemonic::Ret` call-site rewrites) that a separate issue adds more
overhead than it removes. Document this decision in the #1233 PR body.

#994's prior IR shape differs from what §9 assumes, adapt the pre-pass
`register_closure_body_symbols` and `precompute_caller_frame` to match —
the semantics (mangled name, fat-pointer layout) do not change.

### Delivered scope preserves #995 wiring

The `[env_ptr:8 | code_ptr:8]` byte layout is frozen. #995 will emit:

```asm
mov r14, [<f>+0]   ; env_ptr into R14
call [<f>+8]       ; through code_ptr
```

against any `f` produced by `emit_closure_cons`. No further #1233 changes
required to enable #995.
