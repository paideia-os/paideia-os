Issue #1003 is the **integration** ticket that #999 explicitly deferred: the 30-command terminal scaffold that combines the three landed substrates (hashmap + Str + fn-ptr call) into one compile-and-run witness.

## 1. Scope

**Deliver**: an executable demonstration that a u64-keyed hash map can route by hashed command name to a handler function, via raw-asm indirect call, at a command-count representative of a semantic-shell (30 commands is the AC target; 8-command floor is acceptable if scaffold cost forces a trim).

**Defer** (with explicit notes in the CHANGELOG bullet):
- **Str-keyed dispatch.** #998b (Str::eq) and #998c (Str::hash) are P0273 — until they land, the command name is not the runtime key, only the *identity* of the entry. The runtime key is a client-pre-computed FNV-1a u64 baked into the fixture as a literal.
- **Closure-valued dispatch.** HashMap value slot is 8 bytes; a closure fat-pointer is 16 bytes. The u64 value slot cannot hold a closure. Deferred to a `HashMap<u64, Closure>` follow-up once the hashmap V is generic (#996 successor, not in v0.18).
- **Safe-mode `match` on `get` result at 30 arms.** Hits the elaborator's match-scrutinee-must-be-Var gap (#1243), the 4-in-flight-let budget (T0527), and #1244 (StmtExpr-vs-raw-asm ordering). The scaffold shall dispatch in one raw-asm entry block, not via safe-mode `match`.
- **Str-keyed put/get API.** Deferred to a follow-up once #998b/c land; the shape is a mechanical rewrite of the u64 fixture (swap key type; delegate to `Str::hash`).

## 2. Key hashing strategy — client-side FNV-1a

The compiler does not yet emit hash calls for Str values (#998c). The fixture therefore uses **client-side FNV-1a 64-bit hashes computed in the Rust test harness** and baked as `u64` literals into the fixture:

```rust
// tests/common/fnv1a.rs — Rust host helper (LOC ~15)
pub const FNV_OFFSET: u64 = 0xcbf2_9ce4_8422_2325;
pub const FNV_PRIME:  u64 = 0x0000_0100_0000_01b3;
pub fn fnv1a_u64(name: &str) -> u64 {
    let mut h = FNV_OFFSET;
    for b in name.as_bytes() { h ^= *b as u64; h = h.wrapping_mul(FNV_PRIME); }
    h
}
```

The test harness computes `fnv1a_u64("ls")`, `fnv1a_u64("pwd")`, ..., and either (a) prints them for the fixture author to paste in, or (b) uses `include_str!` + regex substitution at test build time. Option (a) is preferred for one-shot fixtures (mechanical, auditable in `git diff`). Option (b) is preferred for the 30-command scaffold where hand-transcription is error-prone.

**Runtime lookup key.** The Rust runtime canary passes the input command's hash as the "user typed X" input by encoding it in the fixture's entry point (each fixture pre-selects one command to dispatch; the runtime canary's assertion is on the resulting exit code).

## 3. Value type + dispatch mechanism — fn-ptr address stored as u64

**Recommended MVP**: value slot holds the **address of the handler function** as a u64 (obtained via `lea` on the RIP-relative symbol inside a raw-asm block). Dispatch retrieves the u64, treats it as a call target, and issues `call rdx` (or equivalent register-indirect call).

This is the archetypal hash-dispatch shape and combines the three substrates named in the AC (hashmap × Str-hash-as-u64 × fn-ptr call) into one program. Rejected alternatives:

| Alternative | Why rejected |
|---|---|
| Value = enum discriminant + safe-mode `match @jump_table` on get | Doubles up on §3.2 of the dispatch-pattern doc without new semantic gain; also blows the elaborator's 30-arm safe-mode `match` limits (see §1 deferrals). |
| Value = enum discriminant + raw-asm `cmp/je` cascade | Equivalent latency but doesn't exercise the fn-ptr call substrate the AC calls out. Retained as **backtrack #1** if `lea [rip + handle_X]` on a plain-fn symbol proves not to work from an unsafe-asm block. |
| Value = closure fat-pointer | 16-byte value doesn't fit u64 slot. Deferred (see §1). |

**Entry-block skeleton** (illustrative):

```
entry unsafe { block:
  // Populate: hash(name) → &handler, once per command.
  mov rdi, 0xHASH_LS ; lea rsi, [rip + handle_ls] ; call hashmap_u64_put
  mov rdi, 0xHASH_PWD; lea rsi, [rip + handle_pwd]; call hashmap_u64_put
  ...
  // Look up one pre-chosen input hash.
  mov rdi, 0xHASH_ECHO
  call hashmap_u64_get
  // rax = discriminant (0=Some,1=None); rdx = payload (handler address or 0).
  cmp rax, 0
  jne dispatch_miss
  call rdx              // <-- the hash-dispatched indirect call.
  ret                   // return handler's rax as fixture exit code.
  dispatch_miss:
  mov rax, 999
  ret
}
```

Every fixture in §4 follows this skeleton with the (name → handler) set and the pre-chosen input hash varied.

## 4. Test canaries

Four fixtures under `tools/paideia-as/tests/build-emit/` plus matching Rust runtime canaries under `tools/paideia-as/crates/paideia-as/tests/runtime/`.

1. **`pa_r18_010_hash_dispatch_smoke.pdx`** — 4 commands (`ls`,`pwd`,`echo`,`exit`), dispatch on `echo`. Expected exit `3`. This is the shape-proof; nothing exercises collision or miss paths.
2. **`pa_r18_010_hash_dispatch_miss.pdx`** — same 4-command table, dispatch on a hash absent from the table. Expected exit `999`. Proves the None branch is reached and returned.
3. **`pa_r18_010_hash_dispatch_collision.pdx`** — 2 command names whose FNV-1a values fold to the *same* 5-bit slot under `fibhash`. Dispatch on the second-inserted name (which must have linear-probed to slot+1). Expected exit = the second handler's return value. Proves linear-probing works under fn-ptr values, not just u64-value payloads.
4. **`pa_r18_010_hash_dispatch_30cmd.pdx`** — the AC scaffold: 30 command handlers, 30 puts, dispatch on one pre-chosen command. Expected exit = that command's handler return. Proves the pattern scales past the density-contract threshold that governs `@jump_table` fallback.

Canaries #1–#3 are hand-authored; #4 uses a Rust build-time codegen step (option 2b in §2) to avoid hand-transcribing 30 FNV hashes.

## 5. Backtrack candidates

1. **`lea rsi, [rip + handle_ls]` on plain-fn symbols inside an unsafe-asm block may not resolve.** The existing witness `pa_r17_004c_call_rip_fnptr.pdx` proves `pub let f = &handle_ls; (f)(x)` works — that is a safe-mode AddrOf on a top-level `pub let`. Whether raw asm inside a *different* fn body can `lea` the RIP-relative symbol of another `pub let` fn is unverified. If it doesn't work, fall back to **safe-mode `pub let hp_ls : (u32)->u32 !{}@{} = &handle_ls`** at module scope, then in the entry raw asm do `lea rsi, [rip + hp_ls]; mov rsi, [rsi]; ...` — one indirection more but proven.
2. **If option (1)'s fallback also fails**, drop to raw-asm cmp/je cascade on the retrieved discriminant (value = u64 tag). This still uses hashmap + Str-hash-as-u64 (2 of 3 AC substrates), and cross-refs pa_r17_004c for the third. Documented as a scoped downgrade, not a punt.
3. **30 handlers may blow the 32-slot table's 50% load factor** at FNV-1a distribution. Expand storage arrays to `[u64; 64]` (mask `0x3f`) or `[u64; 128]` (mask `0x7f`). Storage-array size is a fixture-local parameter, not a stdlib change.
4. **30 sequential `call hashmap_u64_put` sites in one raw-asm block may hit an encoder budget** (unverified). If so, split entry into two functions: `bootstrap` populates the table; `entry` calls `bootstrap` then dispatches. This keeps everything raw-asm.
5. **`fibhash` fold to 5 bits is too narrow for 30 entries** — the collision fixture #3 must not accidentally over-fill. Either use a 6-bit fold (`shr rax, 58`, mask `0x3f`) with `[u64; 64]` storage, or hand-pick keys.
6. **Cross-repo footprint**. Additive-only: new fixtures, new runtime canaries, one CHANGELOG bullet. No change to `crates/paideia-as/src/`, no submodule bump required at #1003 close (bump happens at v0.18 phase close per version-discipline memory).
