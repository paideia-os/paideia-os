## 1. Scope

**Deliver the encoder substrate, not the vectorised probe.** The issue title reads "AVX2 baseline for hash-table probe" — read the AC carefully: touching files are `Encoder (major addition), docs, tests`. **Not** `paideia-stdlib/`, **not** `hashmap*.pdx`. The word "baseline" here means "the first AVX2 encoder cut, sufficient in shape to write a vector probe as a follow-up" — not "single-witness canary". The AC enumerates all four instructions and 20 byte-exact tests; that's the whole deliverable and it is substantive.

The vectorised hash-table probe itself (load 16 metadata bytes → `vpcmpeqb` → `vpmovmskb` → `bsf`) is **not** a deliverable of #1004. It is a follow-up integration ticket (unfiled today), scoped against v0.19 once real workloads justify the cost of designing a raw-asm YMM-clobber discipline in the ABI.

**In-scope for #1004.**
1. VEX prefix builder (2-byte `C5` and 3-byte `C4` forms) — Intel SDM Vol 2A §2.3.
2. `Ymm` register type and register-name mapping (`ymm0..ymm15`) plumbed through lexer-ident → elaborator → IR `RegId`.
3. Four IR `Mnemonic` variants: `Vpxor`, `Vpcmpeqb`, `Vpmovmskb`, `Vmovdqu` (loads and stores).
4. Encoder functions and dispatch entries for each.
5. 20 byte-exact + 4 iced-x86 round-trip tests (per AC).
6. `design/encoding/vex-prefix.md` (new).
7. CHANGELOG bullet under `## [Unreleased] v0.18`. Version-discipline (workspace.version + tag) moves at v0.18 close, per `feedback_paideia_as_version_discipline.md`.

**Explicitly deferred.**
- Actual vectorised probe rewrite of `crates/paideia-stdlib/src/hashmap.pdx` (no such stdlib file exists yet — hashmap is fixture-inlined per #1003 §1). Deferral is trivial: nothing to rewrite.
- Xmm (128-bit) siblings of the same instructions. AC calls for `ymm` only.
- Cross-encoder discipline for VEX prefix collisions with LOCK/REP (Group 1). Not required for the four in-scope mnemonics; add a paragraph in `vex-prefix.md` §5 documenting the collision matrix for future work.
- ABI extension for callee-saved YMM. Sysv-x64 says XMM/YMM are caller-saved; no ABI plumbing needed for encoder-only tests.
- Parser/elaborator width-checking of YMM operands. AC says "encoder support", not "safe-mode YMM types". Raw-asm `unsafe {}` blocks are the only surface that reaches these mnemonics in v0.18.

## 3. Prerequisite decisions

### 3.1 RegId encoding for YMM

RegId today is `pub struct RegId(pub u8)` and used compactly:
- 0..15: GPR (rax..r15)
- 16..24: cr0..cr8
- 25..32: dr0..dr7
- 33..36: spl/bpl/sil/dil (extended low-byte)
- 0xFF: rip sentinel

**Recommended**: reserve **RegId 37..52 for ymm0..ymm15**. Rationale: dense, contiguous, keeps `pub struct RegId(pub u8)` unchanged (no enum breaking change), unambiguous under existing sub-range decoder in `m2-005 bridge` (per docstring at `register.rs:116`). All existing GPR encoder helpers that mask `reg_id & 7` continue to work — the encoder path for VEX instructions will pull `ymm_id = raw - 37` and pass a `u8` in the low 4 bits of the VEX prefix.

Rejected alternatives:
- Widen `RegId` to `u16` (breaks storage of `Operand::Reg`, forces recompilation of every consumer).
- Add `Operand::Ymm(YmmId)` variant (doubles operand-check surface in every consumer of `Operand`; ~50 sites).
- Reuse `Reg64` low 4 bits (semantic overload; `Reg64::Rax` and `Ymm0` would compare equal by ID, breaks the type-tag invariant most callers rely on).

### 3.2 Register-name mapping

Extend `unsafe_walker/register.rs:register_name_to_regid` with `ymm0..ymm15 => Some(RegId(37..52))`. Extend `register_name_width` with an `Ymm256` width variant, **or** return `None` and let raw-asm skip width classification (consistent with cr0/dr0 today, which return `None`). Recommend the latter — YMM sites are raw-asm only in v0.18, no elaborator width contract to enforce.

### 3.3 Mnemonic variants

Four additive `Mnemonic` variants (no arg struct; single-form encodings):

```rust
Vpxor,        // 3 operands: dst ymm, src1 ymm, src2 ymm  — C5 F5 EF /r (opt. C5 or C4)
Vpcmpeqb,     // 3 operands: dst ymm, src1 ymm, src2 ymm  — C5 F5 74 /r
Vpmovmskb,    // 2 operands: dst r32, src ymm             — C5 FD D7 /r
Vmovdqu {     // 2 operands: dst {ymm|mem}, src {mem|ymm} — C5 FE 6F/7F /r
    is_store: bool,
},
```

Additive per Cross-repo discipline (memory: `paideia-as version-discipline` allows additive Mnemonic through v0.20). No `IntWidth` field — YMM implies 256-bit width unambiguously; encoding is single-form.

### 3.4 Operand encoding for VEX

VEX packs three register indices: `VEX.R` (dst reg), `VEX.vvvv` (src1, 1's-complement), `VEX.B` (src2 rm-field high bit). Existing `Operand::Reg(RegId)` covers all three positions; encoder pulls `ymm_id = raw - 37`, low 3 bits go to ModR/M `reg`/`rm`, high bit inverts into `VEX.R` or `VEX.B`. For `Vpmovmskb` the destination `r32` reuses `Reg32` from GPR territory unchanged.

Memory operand for `Vmovdqu`: reuse `Operand::MemSib` / `MemDisp` / `MemRipRel` untouched. VEX.B carries the base-register high bit, VEX.X carries the SIB.index high bit — computed in the VEX builder from the existing memory-operand structure.

## 4. Prereq gaps I hit while surveying

1. **Opcode-canary pre-push gate.** Commit c961ce2 landed a gate on operator-fragile kernels. VEX byte sequences begin `C5` or `C4`, which are **not** currently kernel opcodes; verify the gate's whitelist does not misclassify `C4`/`C5` at the head of a probe function. If it does, extend the gate's allow-list; if not, no change needed. **Action for workerbee**: touch the gate script with a comment noting VEX-prefix awareness even if no code change is required, so future canary triggers know why C4/C5 at position 0 is expected.
2. **`RegId` docstring at `register.rs:104–117`** enumerates the compact ranges. Update it to include the 37..52 YMM band as part of §3.1. Missed docstrings are a common cause of downstream drift; do it in the same PR.
3. **`register_name_width` returns `None` for cr/dr today** and callers treat that as "no width retarget". Confirm no site up-reads `register_name_width` and *asserts* `Some(...)` — grep before landing YMM.
4. **iced-x86 corpus.** iced already decodes `Vpxor`, `Vpcmpeqb`, `Vpmovmskb`, `Vmovdqu` — verified via `iced_x86::Mnemonic::Vpxor` etc. exists in the 1.x line already used by the workspace. No dep bump.
5. **No AST/lexer/parser change required.** YMM registers arrive as identifiers (like every other GPR name); the elaborator's `parse_register_from_ident` is the entry point, and §3.2 extends only its lookup table. Grep confirms no lexer keyword for register names — they are all `Ident`.

## 5. Test canaries

AC specifies 20 byte-exact + 4 iced round-trip. Suggested distribution across the four mnemonics; put each test file under `crates/paideia-as-encoder/tests/simd/` (new subdir, sibling to `bitwise/`).

**`vex_prefix.rs`** — 4 tests, unit-tests for the VEX builder in isolation (not on the AC's 20-count, but essential for triage). Tests: two-byte-VEX-when-no-B/X/W, three-byte-VEX-with-VEX.B, three-byte-VEX-with-VEX.X, VEX-with-both-B-and-X-forcing-3-byte-form.

**`vpxor.rs`** — 5 byte-exact + 1 iced.
- `vpxor ymm0, ymm0, ymm0` → `C5 FD EF C0` (2-byte VEX, canonical zero-register).
- `vpxor ymm0, ymm0, ymm7` → `C5 FD EF C7`.
- `vpxor ymm0, ymm0, ymm8` → `C4 C1 7D EF C0` (3-byte VEX for B=1).
- `vpxor ymm8, ymm0, ymm0` → `C4 61 7D EF C0` (3-byte VEX for R=1).
- `vpxor ymm15, ymm15, ymm15` → `C4 41 05 EF FF` (all high, W=0, vvvv=~15=0).
- iced: `vpxor ymm3, ymm4, ymm5` round-trip.

**`vpcmpeqb.rs`** — 5 byte-exact + 1 iced.
- `vpcmpeqb ymm0, ymm0, ymm0` → `C5 FD 74 C0`.
- `vpcmpeqb ymm0, ymm1, ymm2` → `C5 F5 74 C2`.
- High-reg variants (dst, src1, src2 each hitting `ymm8..ymm15`).
- iced: `vpcmpeqb ymm3, ymm4, ymm5` round-trip.

**`vpmovmskb.rs`** — 4 byte-exact + 1 iced.
- `vpmovmskb eax, ymm0` → `C5 FD D7 C0`.
- `vpmovmskb eax, ymm7` → `C5 FD D7 C7`.
- `vpmovmskb r8d, ymm0` → `C5 7D D7 C0` (dst-high forces VEX.R).
- `vpmovmskb eax, ymm8` → `C4 C1 7D D7 C0` (src-high, 3-byte VEX).
- iced: `vpmovmskb eax, ymm3` round-trip.

**`vmovdqu.rs`** — 6 byte-exact + 1 iced.
- `vmovdqu ymm0, [rax]` (load, `C5 FE 6F 00`).
- `vmovdqu ymm0, [rax + 0x40]` (load, disp8).
- `vmovdqu ymm0, [rsp]` (load, SIB-escape).
- `vmovdqu ymm0, [r8]` (load, VEX.B for base-high).
- `vmovdqu [rax], ymm0` (store, `C5 FE 7F 00`).
- `vmovdqu [rax + rcx*4], ymm0` (store, SIB with index).
- iced: `vmovdqu ymm3, [rbx + 0x20]` round-trip.

**Total: 20 byte-exact + 4 iced round-trip.** Matches AC exactly.

## 6. Design doc — `design/encoding/vex-prefix.md`

1. **Motivation** — why AVX2 is on the roadmap, cross-ref to §8.3 of tactical plan.
2. **VEX-prefix layout** — 2-byte (`C5 xx`) vs 3-byte (`C4 xx yy`) forms; when 3-byte is mandatory (VEX.B, VEX.X, W=1, or map-select ≠ 0F).
3. **Bit-field diagram** — R, vvvv, L, pp, m-mmmm, B, X, W. Cite Intel SDM Vol 2A §2.3.5 by section number, not by page (page numbering drifts across SDM revisions).
4. **RegId encoding for YMM** — 37..52 band; why not enum widening. Reference this doc.
5. **Interaction with LOCK / REP / segment prefixes** — future-work note. LOCK-VEX collision is architecturally forbidden but the encoder should assert. REP-VEX and segment-VEX collisions are not defined for the mnemonics in this issue; document that we reject them at encode time.
6. **iced-x86 correspondence table** — 4 rows, one per new mnemonic, showing iced enum + hex byte sequence.
7. **Follow-up work** — XMM 128-bit siblings, safe-mode YMM operand types, ABI callee-save contract.

## 7. Backtrack candidates

1. **If VEX builder complexity blows the LOC budget past 1500**, drop `Vmovdqu`'s memory-store form (keep load only). The vectorised probe reads metadata but does not write the metadata block from YMM in the common path — stores are a nice-to-have. Reduces test count to 18 byte-exact + 3 iced, note the trim in CHANGELOG.
2. **If the RegId 37..52 range collides with an undocumented consumer** (grep-and-verify before landing), fall back to `Operand::Ymm(u8)` as a new operand variant. Costs an extra `Operand` match arm at every consumer (~50 sites); painful but mechanical.
3. **If iced-x86's `Decoder` disagrees on 3-byte-VEX canonicalisation** for cases where 2-byte would suffice (some assemblers emit the shorter form, some don't), pin the encoder to **always emit the shortest legal form** and verify iced accepts it. iced's decoder is lenient on this — it decodes both forms to the same `Instruction` — so this is unlikely to bite, but call it out in the doc.
4. **If the opcode-canary gate misclassifies `C4`/`C5` at position 0**, extend the gate's allow-list in the same PR. If the fix is non-trivial, gate the VEX encoder behind a feature flag until the gate can be updated separately (splits the issue into two PRs; permitted but ugly).
5. **If AC's "20 byte-exact + 4 iced" count is missed by ±1** at review time due to test consolidation, adjust the distribution in §5 rather than skipping a mnemonic. All four mnemonics must be exercised for the AC to hold.
6. **v0.18 close pressure.** This is the last v0.18 issue. If the softarch → workerbee → debugger triangle reveals a substantive bug in the VEX builder late, drop `Vmovdqu` store form (per #1) *and* reduce the byte-exact count to 15 (still exercises all 4 mnemonics), and land the trim explicitly in CHANGELOG. Preserving v0.18 close on schedule outranks completeness of the AVX2 substrate — v0.19 has slack.
